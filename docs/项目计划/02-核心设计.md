## 核心设计原则

Core 层遵循以下设计原则：

1. **领域驱动设计 (DDD)**：每个核心服务都是一个独立的限界上下文 (Bounded Context)
2. **接口先行**：所有服务对外暴露接口 (Port)，不依赖具体实现
3. **零框架依赖**：核心业务逻辑不引入任何外部框架
4. **事件广播**：所有状态变更必须通过 Event Bus 广播
5. **最小权限**：服务间调用遵循最小权限原则

## Identity - 身份服务

### 职责概述

身份服务是整个系统认证体系的核心，负责用户身份的创建、管理、验证与多端统一登录。

### 核心概念模型

```
┌─────────────────────────────────────────────────────────┐
│                    Identity Domain                        │
│                                                          │
│  ┌──────────┐         ┌──────────┐        ┌──────────┐  │
│  │   User   │ 1────N  │ Account  │ 1────N │ AuthProv │  │
│  │          │         │          │        │  (OAuth) │  │
│  │ id       │         │ id       │        │          │  │
│  │ username │         │ user_id  │        │ provider │  │
│  │ nickname │         │ type     │        │ uid      │  │
│  │ avatar   │         │ status   │        │ token    │  │
│  │ bio      │         │          │        │          │  │
│  └──────────┘         └──────────┘        └──────────┘  │
│                                                          │
│  ┌──────────┐         ┌──────────┐                      │
│  │ Session  │ N───1   │ Device   │                      │
│  │          │         │          │                      │
│  │ token    │         │ id       │                      │
│  │ user_id  │         │ user_id  │                      │
│  │ device_id│         │ type     │                      │
│  │ expire_at│         │ name     │                      │
│  └──────────┘         └──────────┘                      │
└─────────────────────────────────────────────────────────┘
```

### User 与 Account 解耦设计

这是身份服务最关键的设计决策：

- **User**：代表系统中的"人"，拥有唯一标识、个人信息、社交关系
- **Account**：代表用户的一种登录方式，如邮箱账号、手机号、OAuth 账号

一个 User 可以绑定多个 Account，实现：
- 邮箱 + 手机号 + 微信 + QQ 多种方式登录同一身份
- 用户可以在不丢失身份的前提下切换或解绑登录方式
- 校园网 SSO 账号与平台账号的统一映射

```go
// User 领域实体
type User struct {
    ID        snowflake.ID
    Username  string      // 全局唯一用户名
    Nickname  string      // 昵称（可重复）
    Avatar    string      // 头像 URL
    Bio       string      // 个人简介
    Status    UserStatus  // active / suspended / deactivated
    CreatedAt time.Time
    UpdatedAt time.Time
    DeletedAt *time.Time
}

// Account 领域实体
type Account struct {
    ID        snowflake.ID
    UserID    snowflake.ID
    Type      AccountType  // email / phone / oauth
    Identifier string      // 邮箱地址 / 手机号 / OAuth provider+uid
    Credential string      // 密码哈希 / OAuth token（加密存储）
    Verified  bool         // 是否已验证
    CreatedAt time.Time
    UpdatedAt time.Time
    DeletedAt *time.Time
}
```

### 认证流程

#### 邮箱/密码登录

```
用户输入邮箱+密码
      │
      ▼
查找 Account（type=email, identifier=email）
      │
      ▼
bcrypt 验证密码哈希
      │
      ├── 失败 → 返回错误（密码错误）
      │
      ▼
生成 Session:
  ├── Access Token  (JWT, 有效期 2h)
  └── Refresh Token (随机字符串, 有效期 30d, 存 Redis)
      │
      ▼
返回 Token 对
```

#### OAuth2 / SSO 登录

```
用户点击 "使用校园网登录"
      │
      ▼
302 重定向到 SSO 授权页
      │
      ▼
SSO 回调 + Authorization Code
      │
      ▼
用 Code 换取 SSO Token
      │
      ▼
用 SSO Token 获取用户信息
      │
      ▼
查找 Account（type=sso, provider=campus, uid=sso_uid）
      │
      ├── 已存在 → 更新 Token，返回登录态
      │
      └── 不存在 → 创建 User + Account，返回登录态
```

### Token 设计

```
Access Token (JWT):
{
  "sub": "1234567890",          // User ID
  "iss": "campusos",            // 签发者
  "iat": 1700000000,            // 签发时间
  "exp": 1700007200,            // 过期时间 (2h)
  "jti": "unique-token-id",     // Token ID (用于吊销)
  "roles": ["user", "moderator"], // 角色列表快照
  "permissions": ["thread:read", "thread:write"] // 权限快照
}

Refresh Token:
- 256-bit 随机字符串
- 存储在 Redis 中，绑定 User ID + Device ID
- 支持 Token Family 检测（防止 Refresh Token 泄露后被滥用）
```

### 多端管理

系统支持同一用户在多个设备上同时登录，每个设备拥有独立的 Session：

- **Web 浏览器**：浏览器指纹 + IP 作为设备标识
- **移动端**：设备 ID 作为标识
- **API Key**：第三方应用专用，支持独立权限范围与过期策略

### 事件广播

| 事件 | 触发时机 | 携带数据 |
|:---|:---|:---|
| `user.registered` | 用户注册成功 | user_id, account_type, ip |
| `user.login` | 用户登录成功 | user_id, device_id, ip, method |
| `user.logout` | 用户登出 | user_id, device_id |
| `user.password_changed` | 密码修改 | user_id, ip |
| `user.account_bound` | 绑定新账号 | user_id, account_type |
| `user.account_unbound` | 解绑账号 | user_id, account_type |
| `user.suspended` | 账号被封禁 | user_id, reason, operator_id |

## Permission - 权限中心

### 职责概述

权限中心负责系统中所有资源的访问控制，采用 RBAC（基于角色的访问控制）结合 ABAC（基于属性的访问控制）的混合模型。

### 权限模型设计

```
┌─────────────────────────────────────────────────────────────────┐
│                    Permission Domain                              │
│                                                                   │
│   ┌──────────┐      ┌──────────┐      ┌──────────────────┐      │
│   │   User   │──N:N─│   Role   │──N:N─│   Permission     │      │
│   │          │      │          │      │                  │      │
│   │          │      │ name     │      │ resource         │      │
│   │          │      │ scope    │      │ action           │      │
│   │          │      │          │      │ conditions (ABAC)│      │
│   └──────────┘      └──────────┘      └──────────────────┘      │
│                                                                   │
│   ┌──────────────────────────────────────────────────────────┐   │
│   │                    Policy Rules (ABAC)                     │   │
│   │                                                           │   │
│   │  规则示例:                                                  │   │
│   │  - 允许 [角色:版主] 在 [资源:thread] 上执行 [action:delete]   │   │
│   │    条件: thread.category_id IN user.moderated_categories   │   │
│   │                                                           │   │
│   │  - 允许 [角色:用户] 在 [资源:thread] 上执行 [action:edit]     │   │
│   │    条件: thread.author_id == user.id                       │   │
│   └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### RBAC 核心设计

#### 内置角色体系

```
super_admin  ──→  系统超级管理员（拥有所有权限，不可删除）
    │
    ├── admin  ──→  社区管理员（管理用户、版块、全局设置）
    │
    ├── moderator  ──→  版主（管理特定版块的内容和用户）
    │
    ├── vip  ──→  VIP 用户（特殊功能权限）
    │
    └── user  ──→  普通用户（基础读写权限）
```

#### 权限表达式

权限采用 `resource:action:scope` 三段式表达：

```
格式: {resource}:{action}:{scope}

示例:
- thread:read:public          // 读取公开帖子
- thread:write:own            // 编辑自己的帖子
- thread:delete:moderated     // 删除管辖版块的帖子
- user:ban:global             // 全局封禁用户
- category:manage:assigned    // 管理分配的版块
- plugin:install:global       // 安装插件

通配符:
- thread:*:own                // 对自己帖子的所有操作
- *:*:*                       // 超级管理员的全部权限
```

### ABAC 扩展设计

在 RBAC 基础上，权限中心支持基于属性的条件判断：

```go
// Policy 定义
type Policy struct {
    ID         snowflake.ID
    Name       string
    Effect     PolicyEffect    // allow / deny
    Resources  []string        // 匹配的资源模式
    Actions    []string        // 匹配的操作模式
    Conditions []Condition     // ABAC 条件
    Priority   int             // 优先级（数字越大越优先）
}

// Condition 定义
type Condition struct {
    Attribute string      // 属性路径，如 "resource.author_id"
    Operator  string      // 操作符：eq / neq / in / contains / gt / lt
    Value     interface{} // 比较值或表达式
}

// 示例策略
// "版主只能编辑自己管辖版块的帖子"
Policy{
    Effect:    Allow,
    Resources: []string{"thread"},
    Actions:   []string{"edit", "delete", "pin"},
    Conditions: []Condition{
        {
            Attribute: "resource.category_id",
            Operator:  "in",
            Value:     "user.moderated_categories",  // 引用用户属性
        },
    },
}
```

### 权限检查流程

```
请求到达 → 提取 (user, resource, action)
            │
            ▼
    ┌───────────────┐
    │  查询用户角色   │
    └───────┬───────┘
            │
            ▼
    ┌───────────────┐
    │  展开角色权限   │  将角色映射为权限列表
    └───────┬───────┘
            │
            ▼
    ┌───────────────┐
    │  匹配权限规则   │  resource:action 模式匹配
    └───────┬───────┘
            │
            ├── 匹配到 deny 规则 → 拒绝访问
            │
            ├── 匹配到 allow 规则（无条件） → 允许访问
            │
            ├── 匹配到 allow 规则（有条件） → 评估 ABAC 条件
            │   │
            │   ├── 条件满足 → 允许访问
            │   └── 条件不满足 → 拒绝访问
            │
            └── 无匹配规则 → 拒绝访问（默认拒绝）
```

### 资源级与字段级授权

#### 资源级授权

控制用户对某一类资源的增删改查操作：

```
用户 A 对 "技术分享" 版块拥有 thread:read, thread:write 权限
用户 A 对 "公告通知" 版块仅拥有 thread:read 权限
```

#### 字段级授权

控制用户在同一资源上可以访问的字段：

```json
// API 返回用户信息时，根据权限过滤字段
// 普通用户看到的：
{
    "id": "123",
    "nickname": "张三",
    "avatar": "https://...",
    "bio": "大家好"
}

// 管理员看到的：
{
    "id": "123",
    "nickname": "张三",
    "avatar": "https://...",
    "bio": "大家好",
    "email": "zhangsan@example.com",
    "phone": "138****1234",
    "status": "active",
    "last_login_at": "2024-01-01T00:00:00Z",
    "created_at": "2023-06-01T00:00:00Z"
}
```

### 事件广播

| 事件 | 触发时机 | 携带数据 |
|:---|:---|:---|
| `role.created` | 创建角色 | role_id, name |
| `role.updated` | 更新角色权限 | role_id, changes |
| `role.deleted` | 删除角色 | role_id |
| `role.assigned` | 给用户分配角色 | user_id, role_id, scope |
| `role.revoked` | 撤销用户角色 | user_id, role_id |
| `permission.granted` | 授予特殊权限 | user_id, permission |
| `permission.revoked` | 撤销特殊权限 | user_id, permission |

## Storage - 存储引擎

### 职责概述

存储引擎提供统一的文件存储抽象层（VFS - Virtual File System），屏蔽底层存储实现差异。

### 架构设计

```
┌─────────────────────────────────────────────┐
│              Storage Service                 │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │         VFS (Virtual File System)      │  │
│  │                                        │  │
│  │  接口:                                  │  │
│  │  - Upload(ctx, file) → URL             │  │
│  │  - Download(ctx, key) → File           │  │
│  │  - Delete(ctx, key) → error            │  │
│  │  - GetPresignedURL(ctx, key, ttl) → URL│  │
│  │  - List(ctx, prefix) → []File          │  │
│  └────────────────┬───────────────────────┘  │
│                   │                          │
│       ┌───────────┼───────────┐              │
│       │           │           │              │
│  ┌────▼────┐ ┌────▼────┐ ┌───▼────┐         │
│  │ Local   │ │  S3 /   │ │  OSS   │         │
│  │ Storage │ │  MinIO  │ │ (阿里云)│         │
│  └─────────┘ └─────────┘ └────────┘         │
└─────────────────────────────────────────────┘
```

### 文件分类与策略

| 文件类型 | 存储策略 | 示例 |
|:---|:---|:---|
| **用户头像** | 生成缩略图，CDN 缓存 | avatar_123.webp |
| **帖子附件** | 原始文件 + 预览图 | thread_attach_456.pdf |
| **系统资源** | 高可用存储 | plugin_assets/* |
| **临时文件** | 自动过期清理 | tmp/* (TTL: 24h) |

### 文件元数据管理

```go
// File 元数据实体
type FileMetadata struct {
    ID          snowflake.ID
    Key         string        // 存储路径/Key
    OriginalName string       // 原始文件名
    ContentType string        // MIME 类型
    Size        int64         // 文件大小 (bytes)
    Hash        string        // SHA-256 哈希（用于去重和校验）
    StorageType StorageType   // local / s3 / oss
    UploaderID  snowflake.ID  // 上传者 ID
    Bucket      string        // 存储桶名称
    Tags        []string      // 标签（用于分类管理）
    CreatedAt   time.Time
    DeletedAt   *time.Time
}
```

### 上传流程

```
客户端上传文件
      │
      ▼
参数验证（文件大小、类型白名单）
      │
      ▼
计算 SHA-256 哈希 → 查询是否已存在（秒传检测）
      │
      ├── 已存在 → 直接返回已有文件 URL
      │
      ▼
上传到后端存储（Local / S3 / MinIO）
      │
      ▼
保存文件元数据到 PostgreSQL
      │
      ▼
发布 file.uploaded 事件
      │
      ▼
返回文件访问 URL
```

### 安全策略

- **文件类型白名单**：仅允许安全的文件类型上传
- **文件大小限制**：单文件最大 50MB（可配置）
- **文件名消毒**：生成随机 Key，防止路径穿越攻击
- **访问控制**：私有文件通过 Pre-signed URL 控制访问

## Audit - 审计中心

### 职责概述

审计中心负责记录系统中所有核心数据变更与高权限操作，采用防篡改设计，满足合规与安全需求。

### 审计日志结构

```go
// AuditLog 审计日志实体
type AuditLog struct {
    ID         snowflake.ID
    TraceID    string          // 链路追踪 ID
    ActorID    *snowflake.ID   // 操作者 ID（系统操作时为 nil）
    ActorType  ActorType       // user / system / plugin / ai_agent
    Action     string          // 操作类型: create / update / delete / login / ban
    Resource   string          // 资源类型: user / thread / post / category
    ResourceID string          // 资源 ID
    Before     json.RawMessage // 变更前的快照（仅 update 操作）
    After      json.RawMessage // 变更后的快照
    Metadata   json.RawMessage // 附加信息（如 IP、User-Agent、操作原因）
    IPAddress  string          // 操作者 IP
    CreatedAt  time.Time
}
```

### 审计范围

| 操作类型 | 资源 | 记录级别 |
|:---|:---|:---|
| 用户注册/登录/登出 | user | 详细记录 |
| 密码修改/重置 | user | 详细记录 + 安全告警 |
| 角色分配/撤销 | role | 详细记录 |
| 帖子创建/编辑/删除 | thread/post | 详细记录 |
| 版块管理操作 | category | 详细记录 |
| 用户封禁/解封 | user | 详细记录 + 通知 |
| 插件安装/卸载/配置 | plugin | 详细记录 |
| 系统配置变更 | configuration | 详细记录 + 告警 |
| API Key 创建/吊销 | api_key | 详细记录 |

### 防篡改设计

```
┌──────────────────────────────────────────────────┐
│              防篡改机制                             │
│                                                   │
│  1. 审计日志只允许 INSERT，禁止 UPDATE / DELETE     │
│                                                   │
│  2. 每条日志计算 HMAC 签名:                         │
│     signature = HMAC-SHA256(                       │
│       key: system_audit_secret,                    │
│       message: id + actor + action + resource      │
│                 + before + after + timestamp        │
│     )                                              │
│                                                   │
│  3. 链式哈希：每条记录包含前一条记录的哈希            │
│     current_hash = SHA256(                         │
│       prev_hash + current_data + signature         │
│     )                                              │
│                                                   │
│  4. 定期校验：后台任务定期校验哈希链完整性            │
└──────────────────────────────────────────────────┘
```

### 查询能力

- 支持按操作者、资源类型、时间范围、操作类型进行组合查询
- 支持导出审计报告（CSV / PDF）
- 敏感操作实时告警（如管理员批量删除）

## Notification - 通知服务

通知服务负责将系统消息通过多种渠道触达用户。

```
┌──────────────────────────────────────────────────────────────┐
│                   Notification Service                         │
│                                                                │
│  ┌──────────────┐    ┌──────────────────────────────────────┐  │

│  │ 路由到对应渠道│    │  │ Inbox  │ │ Email │ │           │  │  │
│  └──────────────┘    │  └────────┘ └───────┘ └───────────┘  │  │
```

### 通知类型

| 类型 | 触发场景 | 渠道 | 示例 |
|:---|:---|:---|:---|
| **系统通知** | 系统维护、公告 | 站内信 + 邮件 | "系统将于今晚 22:00 进行维护" |
| **互动通知** | 回复、@提及、点赞 | 站内信 + WebSocket | "张三 回复了你的帖子" |
| **审核通知** | 帖子审核结果 | 站内信 | "你的帖子已通过审核" |
| **安全通知** | 异常登录、密码变更 | 站内信 + 邮件 | "检测到新设备登录" |
| **管理通知** | 封禁、警告 | 站内信 + 邮件 | "你的账号已被封禁 7 天" |

## Configuration - 配置中心

### 职责概述

配置中心负责管理系统级动态配置，支持热更新，无需重启服务即可生效。

### 设计方案

```go
// Configuration 配置实体
type Configuration struct {
    Key         string      // 配置键（层级命名，如 "community.thread.max_length"）
    Value       interface{} // 配置值
    Type        ConfigType  // string / int / bool / json
    Description string      // 配置说明
    Category    string      // 分类
    IsSecret    bool        // 是否为敏感配置（加密存储）
    UpdatedBy   snowflake.ID
    UpdatedAt   time.Time
}
```

### 配置分类

| 分类 | 示例配置项 | 说明 |
|:---|:---|:---|
| **system** | `system.site_name`, `system.maintenance_mode` | 系统级配置 |
| **community** | `community.thread.max_length`, `community.post.rate_limit` | 社区功能配置 |
| **security** | `security.login.max_attempts`, `security.captcha.enabled` | 安全相关配置 |
| **plugin** | `plugin.napcat.enabled`, `plugin.ai.model` | 插件配置 |
| **notification** | `notification.email.smtp_host` | 通知渠道配置 |

### 热更新机制

```
配置变更请求
      │
      ▼
写入 PostgreSQL
      │
      ▼
发布 config.updated 事件 ──→ Event Bus
      │
      ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  实例 A     │     │  实例 B     │     │  实例 C     │
│  监听事件    │     │  监听事件    │     │  监听事件    │
│  更新本地缓存│     │  更新本地缓存│     │  更新本地缓存│
└─────────────┘     └─────────────┘     └─────────────┘
```

- 本地缓存 + Redis 缓存双层架构
- 配置变更时通过 Event Bus 通知所有实例刷新
- 支持配置回滚（保留历史版本）

### 服务间调用关系

| 调用方 | 被调用方 | 场景 |
|:---|:---|:---|
| Gateway | Identity | JWT Token 验证 |
| Gateway | Permission | 请求权限检查 |
| Community App | Permission | 发帖/回帖权限检查 |
| Community App | Storage | 上传/获取附件 |
| Community App | Notification | 触发通知 |
| 所有服务 | Audit | 记录操作日志 |
| 所有服务 | Configuration | 读取配置 |
| 所有服务 | Event Bus | 发布/订阅事件 |

> 各服务间通过 Event Bus 进行异步解耦通信，同步调用仅用于必须立即获取结果的场景（如权限检查）。
