# 插件保存位置与当前插件作用汇总

本文只说明 CampusOS 当前插件保存位置、Admin 管理入口，以及现有插件的作用。具体安装、打包、配置和运行方法，以各插件目录中的 README 或项目已有插件教程为准。

## 1. 当前结论

- 插件文件默认保存在 `examples/plugins/<插件名>/`。
- 后端启动时默认扫描 `examples/plugins`，也可以通过环境变量 `PLUGINS_DIR` 改成其他插件目录。
- Admin 后台已有 `插件管理` 页面，可以查看已安装插件、启用状态、运行时、版本、描述，并执行导入、导出、启用、禁用、卸载和查看日志。
- 目前没有独立命名为 `插件仓库` 或 `插件市场` 的完整界面；现有 `插件管理` 更接近“已安装插件管理页”，不是远程下载市场。
- PostgreSQL 中保存的是插件元数据、状态、配置、权限和日志；插件代码或 Wasm 文件本身仍保存在插件目录中。

## 2. 插件保存位置

| 内容 | 默认位置或表 | 说明 |
| --- | --- | --- |
| 插件目录 | `examples/plugins/<插件名>/` | 每个插件一个目录，目录内需要有 `plugin.yaml`。 |
| 插件 manifest | `examples/plugins/<插件名>/plugin.yaml` | 记录插件名称、版本、运行时、订阅事件、权限、配置等信息。 |
| Wasm 插件文件 | `examples/plugins/<插件名>/*.wasm` | Wasm Runtime 会根据 `plugin.yaml` 中的配置加载模块，例如 `plugin.wasm`。 |
| CLI 安装目标 | `examples/plugins` | `campusosctl plugin install` 默认把插件包解压到该目录下的插件名子目录。 |
| 可覆盖插件目录 | `PLUGINS_DIR` | 后端启动时如果设置了该环境变量，会改为扫描指定目录。 |
| 插件 KV 数据 | `.campusos/plugin-data/<插件名>/plugin.db` | SQLite-backed 插件 KV 存储，默认根目录可由 `PLUGIN_DATA_DIR` 调整。 |
| 插件元数据 | PostgreSQL `plugins` 表 | 保存插件名称、显示名称、版本、运行时、状态、配置、错误信息等。 |
| 插件权限声明 | PostgreSQL `plugin_permissions` 表 | 保存插件声明或同步后的 Host API 权限。 |
| 插件运行日志 | PostgreSQL `plugin_logs` 表 | 保存插件运行、事件处理、Host API 日志等记录。 |
| API Key | PostgreSQL `api_keys` 表 | 保存插件或用户相关 API Key 信息。 |

补充说明：

- `internal/plugin/` 是插件框架源码，包括 Manager、Runtime、Host API、仓储接口等，不是插件保存目录。
- `admin/src/views/PluginManageView.vue` 是 Admin 插件管理页面源码，不是插件保存目录。
- 如果项目未启用 PostgreSQL 仓储，插件 Manager 仍会在内存中维护本次进程加载到的插件状态。

## 3. 后端如何发现插件

后端启动时会创建 Plugin Manager，并注册当前支持的运行时：

- `grpc`
- `wasm`

随后后端扫描插件目录：

- 默认目录：`examples/plugins`
- 覆盖目录：`PLUGINS_DIR`

扫描规则比较简单：

- 只扫描插件目录下的一级子目录。
- 子目录中存在 `plugin.yaml` 才会被视为插件。
- manifest 解析成功后，插件会被安装到 Plugin Manager 的内存列表中。
- 如果配置了 PostgreSQL 插件仓储，插件元数据会同步到 `plugins` 表。

因此，一个插件被“安装”至少包含两层含义：

- 文件层：插件目录存在，并且有合法的 `plugin.yaml` 和必要运行文件。
- 运行层：后端启动或 CLI 安装后，Plugin Manager 能识别并加载该插件。

## 4. Admin 是否有插件仓库界面

当前 Admin 端已有 `插件管理` 页面：

- 前端路由：`/plugins`
- 页面源码：`admin/src/views/PluginManageView.vue`
- 后端接口前缀：`/api/v1/plugins`
- 权限要求：需要通过 JWT 登录，并具备 `role:manage` 权限。

页面当前能力：

| 能力 | 当前状态 |
| --- | --- |
| 查看已安装插件列表 | 已支持 |
| 查看插件名称、显示名称、版本、描述 | 已支持 |
| 查看运行时类型 | 已支持 |
| 查看启用/禁用状态 | 已支持 |
| 启用插件 | 已支持 |
| 禁用插件 | 已支持 |
| 卸载插件 | 已支持 |
| 查看插件日志 | 已支持 |
| 导入标准插件包 | 已支持 |
| 导出标准插件包 | 已支持 |
| 在线下载插件 | 暂未支持 |
| 插件市场搜索 | 暂未支持 |
| 插件版本回滚 | 暂未支持 |

所以，当前可以理解为：

- 有 Admin 插件管理界面，可管理已安装插件并导入/导出标准插件包。
- 还没有完整的“插件仓库/插件市场”界面。
- 如果后续要做“插件仓库”，建议在现有 `插件管理` 基础上扩展“本地已安装”和“远程可安装”两个视图。

## 5. 当前插件作用汇总

### 5.1 `examples/plugins/hello-plugin`

作用：

- 作为 gRPC Runtime 插件的示例 manifest。
- 说明 gRPC 插件如何声明基础元数据、事件订阅、API 权限和 SQLite 存储。
- 当前订阅了用户、帖子、回复、版块等核心事件，设计目标是“接收并记录核心事件”。
- 适合作为后续开发 gRPC 插件时的 manifest 参考。

注意：

- 当前该目录只有 `plugin.yaml`，没有独立 README。
- 如果后续补充实际 gRPC 服务端实现，应以该插件自己的 README 为准。

### 5.2 `examples/plugins/hello-wasm`

作用：

- 作为 Wasm Runtime 的最小可运行示例插件。
- 用于验证 `runtime: wasm` 插件可以被安装、启动并接收事件。
- 当前订阅 `thread.created` 和 `post.created` 事件。
- 当前 Wasm 入口函数固定返回允许事件继续，用于验证事件派发链路。
- 可用于测试 Manager、Wasm Runtime、事件处理结果和 `plugin_logs` 写入链路。

使用说明：

- 具体 ABI、文件说明、重新生成 `plugin.wasm` 和测试命令，请参考 `examples/plugins/hello-wasm/README.md`。

## 6. 与 v0.4 个人主页能力的关系

v0.4 中的个人主页、风格包导入导出、内容同步等能力，目前主要作为内置后端模块实现，相关代码在 `internal/space/` 一类项目模块中。

这部分能力目前不是 `examples/plugins/<插件名>/` 下的独立插件包，因此不会以“已安装插件”的形式出现在 Admin 插件管理列表中。

后续如果要把“博客园式个人主页”或“风格切换”做成真正插件，需要补齐：

- 独立插件目录和 `plugin.yaml`。
- 插件运行时实现。
- 与个人主页数据模型的 Host API 或后端接口边界。
- Admin 或用户侧的插件配置入口。
- 插件安装、启用、禁用、卸载后的数据保留策略。

## 7. 建议后续演进方向

如果要把现有 `插件管理` 升级为真正的 `插件仓库`，建议按以下顺序推进：

1. 保留现有 `插件管理`，作为“已安装插件”视图。
2. 新增“本地插件包安装”能力，支持上传或指定插件包。
3. 新增“插件仓库”视图，展示远程可安装插件。
4. 增加插件签名、版本校验、兼容版本校验。
5. 增加插件安装前权限预览。
6. 增加插件升级、回滚和数据迁移策略。

当前阶段更建议先稳定：

- 插件目录规范。
- manifest schema。
- Admin 已安装插件管理。
- Host API 权限控制。
- 插件日志和故障定位能力。
