# CampusOS 数据库管理指南

> 本文档涵盖数据库的完整管理知识：表结构设计、账号初始化、角色权限、备份策略、回滚方案、Web UI 管理、日常运维操作等。

---

## 一、数据库基本信息

| 项目 | 值 |
|:---|:---|
| **数据库引擎** | PostgreSQL 16+ |
| **Go 驱动** | pgx 5（原生协议，非 GORM） |
| **数据库名** | `campusos` |
| **用户名** | `campusos` |
| **密码** | `campusos_dev` |
| **端口** | `5432`（容器内）；宿主机默认映射为 `${POSTGRES_PORT:-5432}` |
| **连接池** | pgxpool，默认最大连接数由 pgx 自动管理 |
| **字符集** | UTF-8（PostgreSQL 默认） |
| **数据卷** | `campusos-pgdata`（Docker 命名卷，持久化存储） |

### 连接方式

```bash
# 命令行连接
psql -h localhost -U campusos -d campusos
# 输入密码: campusos_dev

# 或通过 Docker 进入
docker exec -it campusos-postgres psql -U campusos -d campusos

# 连接字符串（供代码/工具使用）
postgres://campusos:campusos_dev@localhost:${POSTGRES_PORT:-5432}/campusos?sslmode=disable
```

## 二、表结构设计总览

共 **16 张表**，分 4 个域：

```
身份域（Identity）          社区域（Community）         系统域（System）           插件域（Plugin）
├── users                  ├── categories             ├── audit_logs            ├── plugins
├── posts                  ├── threads                ├── notifications         └── api_keys
├── accounts               ├── posts（回复）           └── configurations
├── sessions               ├── tags
└── roles / user_roles     └── likes
    / role_permissions
```

> 注：`posts` 表在初始迁移中用于帖子主体，后续迁移中帖子功能迁移至 `threads` 表，`posts` 表改为回复/评论。

## 四、Admin 账号与自动初始化

### 4.1 启动时自动创建 Admin 账号

**是的，系统启动时会自动创建 admin 账号。** 这是通过两种机制实现的：

#### 机制一：SQL 种子迁移（000004_seed_admin.up.sql）

执行 `make migrate-up` 时，SQL 迁移脚本直接插入 admin 数据：

```sql
-- 插入 admin 用户（密码: Admin@123456，bcrypt 哈希）
INSERT INTO users (id, username, nickname, email, password_hash, status)
VALUES (1, 'admin', '系统管理员', 'admin@campusos.local', '<bcrypt哈希>', 'active')
ON CONFLICT (username) DO NOTHING;

-- 分配 admin 角色
INSERT INTO user_roles (user_id, role_id) VALUES (1, 1)
ON CONFLICT (user_id, role_id) DO NOTHING;
```

#### 机制二：Go 代码种子（internal/server/seed.go）

后端启动时，`SeedAdmin()` 函数会：

1. 检查 `users` 表中是否存在 `username = 'admin'` 的记录
2. 如果不存在，则：
   - 生成 bcrypt 哈希（密码：`Admin@123456`）
   - 使用固定雪花 ID 插入 users 表
   - 创建 accounts 表关联
   - 在 user_roles 表中分配 admin 角色
3. 如果已存在，跳过（幂等设计）

### 4.2 Admin 账号信息

| 项目 | 值 |
|:---|:---|
| **用户名** | `admin` |
| **密码** | `Admin@123456` |
| **邮箱** | `admin@campusos.local` |
| **昵称** | 系统管理员 |
| **角色** | admin（系统管理员） |
| **状态** | active |

> ⚠️ **安全提示**：首次部署后应立即修改 admin 密码。

### 4.3 默认版块

种子数据还会创建一个默认版块：

```sql

VALUES ('默认版块', '系统默认版块', 0)
ON CONFLICT DO NOTHING;
```

### 5.3 权限检查流程

```
请求 → JWT 中间件解析 Token → 提取 user_id
    → 查询 user_roles 表获取角色
    → 查询 role_permissions 表获取权限列表
    → 匹配 resource + action
    → 允许/拒绝
```

### 5.4 给新用户分配角色

```sql
-- 查看所有角色
SELECT * FROM roles;

-- 查看用户当前角色
SELECT u.username, r.name AS role
FROM users u
JOIN user_roles ur ON u.id = ur.user_id
JOIN roles r ON ur.role_id = r.id
WHERE u.username = '某用户';

-- 给用户分配版主角色
INSERT INTO user_roles (user_id, role_id) VALUES (用户ID, 2)
ON CONFLICT (user_id, role_id) DO NOTHING;

-- 撤销用户角色
DELETE FROM user_roles WHERE user_id = 用户ID AND role_id = 2;
```

## 六、数据库迁移管理

### 6.1 迁移文件说明

CampusOS 使用**手动 SQL 迁移文件**管理数据库 Schema，所有迁移文件存放在 `migrations/` 目录下，成对出现：

| 迁移编号 | UP 文件（建表/插入） | DOWN 文件（回滚/删除） | 说明 |
|:---|:---|:---|:---|
| **000001** | `000001_init_schema.up.sql` | `000001_init_schema.down.sql` | 创建 11 张核心表 + 索引 |
| **000002** | `000002_add_roles.up.sql` | `000002_add_roles.down.sql` | 创建 roles/user_roles/role_permissions + 种子角色 |
| **000003** | `000003_add_plugins.up.sql` | `000003_add_plugins.down.sql` | 创建 plugins/api_keys 表 |
| **000004** | `000004_seed_admin.up.sql` | `000004_seed_admin.down.sql` | 插入 admin 用户 + 默认版块 |

**每个迁移文件的作用**：

| 文件 | 具体操作 |
|:---|:---|
| 000001 UP | 创建 `users`、`posts`、`accounts`、`sessions`、`categories`、`threads`、`replies`、`tags`、`tag_relations`、`likes`、`event_history` 共 11 张表及索引 |
| 000001 DOWN | `DROP TABLE IF EXISTS` 删除上述所有 11 张表（按外键依赖逆序） |
| 000002 UP | 创建 `roles`、`user_roles`、`role_permissions` 3 张表，并插入 4 个内置角色（admin/moderator/member/guest）和 20 条权限规则 |
| 000002 DOWN | 删除 `role_permissions`、`user_roles`、`roles` 3 张表 |
| 000003 UP | 创建 `plugins`、`api_keys` 2 张表 |
| 000003 DOWN | 删除 `api_keys`、`plugins` 2 张表 |
| 000004 UP | 插入 admin 用户（用户名 admin，密码 Admin@123456）、admin 账号关联、admin 角色分配、默认版块 |
| 000004 DOWN | 删除种子数据（admin 用户、默认版块等） |

### 6.2 迁移命令与迁移文件的关系

#### Makefile 中的命令（当前状态）

当前 Makefile 中的迁移命令**只执行第一个迁移文件**：

```makefile
# ⚠️ 当前实现：只执行 000001_init_schema.up.sql
migrate-up:
	PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos \
	  -f migrations/000001_init_schema.up.sql

# ⚠️ 当前实现：只执行 000001_init_schema.down.sql
migrate-down:
	PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos \
	  -f migrations/000001_init_schema.down.sql
```

> ⚠️ **重要**：`make migrate-up` 只创建了 11 张核心表，**不会**创建角色表、插件表，也**不会**插入 admin 种子数据！

#### 完整迁移（手动执行全部 4 个文件）

要完成全部数据库初始化，需要**按顺序手动执行所有 4 个 UP 迁移**：

```bash
# ===== 完整数据库初始化（首次部署必须执行全部 4 个） =====

# 第 1 步：创建 11 张核心表
PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos \
  -f migrations/000001_init_schema.up.sql

# 第 2 步：创建角色/权限表 + 插入 4 个内置角色
PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos \
  -f migrations/000002_add_roles.up.sql

# 第 3 步：创建插件/API Key 表
PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos \
  -f migrations/000003_add_plugins.up.sql

# 第 4 步：插入 admin 用户 + 默认版块
PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos \
  -f migrations/000004_seed_admin.up.sql
```

或者用一条命令串联执行：

```bash
for f in migrations/00000{1,2,3,4}_*.up.sql; do
  echo "执行迁移: $f"
  PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos -f "$f"
done
```

#### 建议：更新 Makefile 支持全部迁移

建议将 Makefile 更新为执行全部迁移文件：

```makefile
# 数据库迁移 - 执行所有 UP 迁移
migrate-up:
	@echo "=== 执行数据库迁移 ==="
	@for f in migrations/$$(ls migrations/*.up.sql | sort); do \
		echo "▶ 执行: $$f"; \
		PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos -f "$$f" || exit 1; \
	done
	@echo "=== 迁移完成 ==="

# 数据库回滚 - 执行所有 DOWN 迁移（逆序）
migrate-down:
	@echo "=== 执行数据库回滚 ==="
	@for f in $$(ls migrations/*.down.sql | sort -r); do \
		echo "▶ 回滚: $$f"; \
		PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos -f "$$f" || exit 1; \
	done
	@echo "=== 回滚完成 ==="
```

```
        │
        ▼
  等待 PostgreSQL 就绪（约 5s）
        │
        ▼
  执行 000001_init_schema.up    # 2. 创建 11 张核心表
        │
        ▼
  执行 000002_add_roles.up      # 3. 创建角色表 + 4 个内置角色
        │
        ▼
  执行 000003_add_plugins.up    # 4. 创建插件表
        │
        ▼
  执行 000004_seed_admin.up     # 5. 插入 admin 用户 + 默认版块
        │
        ▼
  go run cmd/server/main.go     # 6. 启动后端（seed.go 会检测 admin 已存在，跳过）
        │
        ▼
  ✅ 系统就绪，可用 admin/Admin@123456 登录
```

**两种机制的关系**：

| 机制 | 触发时机 | 作用 | 幂等性 |
|:---|:---|:---|:---|
| `000004_seed_admin.up.sql` | 执行 `migrate-up` 时 | 通过 SQL 直接插入 admin 数据 | ✅ `ON CONFLICT DO NOTHING` |
| `seed.go` → `SeedAdmin()` | 后端启动时 | Go 代码检测并补插 admin 数据 | ✅ 先查询再插入 |

> 两者是**互补关系**：如果只执行了 000001 而没有执行 000004，后端启动时 seed.go 会自动补建 admin 账号。但如果连 000002（角色表）都没执行，seed.go 也会失败，因为 `user_roles` 表不存在。

### 6.4 迁移顺序

**必须按编号顺序执行，不能跳过**，因为存在外键依赖：

```
000001 (users/threads/posts...)
    ↓ 外键依赖
000002 (roles → user_roles → role_permissions，引用 users.id)
    ↓ 无外键依赖但逻辑顺序
000003 (plugins, api_keys)
    ↓ 引用 users.id 和 roles.id
000004 (种子数据：admin 用户 + 角色分配)
```

如果跳过 000001 直接执行 000002，会报错：`relation "users" does not exist`。

### 6.5 回滚顺序

回滚必须**逆序执行** DOWN 文件：

```
000004 DOWN (删除种子数据)
    ↓
000003 DOWN (删除 plugins/api_keys 表)
    ↓
000002 DOWN (删除 roles/user_roles/role_permissions 表)
    ↓
000001 DOWN (删除所有核心表)
```

```bash
# 完整回滚（逆序）
PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos \
  -f migrations/000004_seed_admin.down.sql
PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos \
  -f migrations/000003_add_plugins.down.sql
PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos \
  -f migrations/000002_add_roles.down.sql
PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos \
  -f migrations/000001_init_schema.down.sql
```

### 6.6 添加新迁移

在 `migrations/` 目录下创建新的迁移文件对，编号递增：

```bash
# 命名规范：000005_描述.up.sql / 000005_描述.down.sql
touch migrations/000005_add_audit_enhancements.up.sql
touch migrations/000005_add_audit_enhancements.down.sql
```

**编写规范**：

```sql
-- 000005_add_audit_enhancements.up.sql
-- 添加新表或修改现有表
CREATE TABLE IF NOT EXISTS new_feature (...);
ALTER TABLE users ADD COLUMN IF NOT EXISTS new_field VARCHAR(100);

-- 000005_add_audit_enhancements.down.sql
-- 回滚上述操作
DROP TABLE IF EXISTS new_feature;
ALTER TABLE users DROP COLUMN IF EXISTS new_field;
```

> 建议所有 DDL 语句都使用 `IF NOT EXISTS` / `IF EXISTS` 以保证幂等性。

### 9.1 推荐方案：pgAdmin（Docker 部署）

pgAdmin 是 PostgreSQL 最流行的 Web 管理工具，支持增删查改、SQL 查询、表结构管理等。

#### 一键启动 pgAdmin

项目 `docker-compose.yml` 已内置 pgAdmin 服务：

```yaml
    volumes:
      - pgadmin-data:/var/lib/pgadmin
    extra_hosts:
      - "host.docker.internal:host-gateway"
```

启动命令：

```bash
./scripts/docker-up.sh
# 或手动启动
docker compose up -d postgres pgadmin
```

启动后访问：**http://localhost:5050**

| 项目 | 值 |
|:---|:---|
| 登录邮箱 | admin@campusos.dev |
| 登录密码 | pgadmin123 |

#### 添加数据库连接

1. 登录 pgAdmin
2. 右键 "Servers" → "Register" → "Server"
3. General 标签：Name = `CampusOS`
4. Connection 标签：
   - Host: `postgres`
   - Port: `5432`
   - Database: `campusos`
   - Username: `campusos`
   - Password: `campusos_dev`

注意：

| 场景 | Host | Port | 说明 |
|:---|:---|:---|:---|
| pgAdmin 容器连接 Compose 内的 PostgreSQL | `postgres` | `5432` | 推荐方式。pgAdmin 和 PostgreSQL 在同一个 Docker Compose 网络内 |
| 宿主机命令行连接 PostgreSQL 容器 | `127.0.0.1` | `${POSTGRES_PORT:-5432}` | 例如 `psql -h 127.0.0.1 -p 5433 ...` |
| pgAdmin 容器连接宿主机本地 PostgreSQL | `host.docker.internal` | 宿主机 PostgreSQL 端口 | 只用于你确实要连宿主机数据库的情况 |

不要在 pgAdmin 容器里用 `127.0.0.1` 连接 Compose PostgreSQL。对 pgAdmin 容器来说，`127.0.0.1` 指的是 pgAdmin 自己，不是 `campusos-postgres`。

#### 5432 端口冲突处理

如果启动 PostgreSQL 时出现：

```text
failed to bind host port 0.0.0.0:5432/tcp: address already in use
```

说明宿主机 `5432` 已被本机 PostgreSQL 或其他容器占用。推荐在 `.env` 中改宿主机映射端口：

```env
POSTGRES_PORT=5433
```

然后重建基础服务：

```bash
docker compose down
docker compose up -d postgres pgadmin
```

此时：

| 访问位置 | 连接信息 |
|:---|:---|
| pgAdmin 页面内注册 CampusOS 数据库 | Host=`postgres`，Port=`5432` |
| 宿主机命令行连接 CampusOS 数据库 | `psql -h 127.0.0.1 -p 5433 -U campusos -d campusos` |

不建议通过 `docker network connect` 修复该问题。端口冲突属于容器启动/端口映射问题，手动连网络不能解决宿主机端口已被占用的问题。

## 十二、常见问题 FAQ

### Q1: 启动后没有 admin 账号？

**A**: 检查是否执行了迁移：
```bash
make migrate-up
```
如果已执行，检查 seed.go 的日志输出，确认 `✅ 管理员账号已存在，跳过种子数据`。

### Q2: 忘记 admin 密码？

**A**: 直接在数据库重置：
```bash
# 生成新的 bcrypt 哈希（需要 Go 环境）
# 或直接用已知哈希替换
PGPASSWORD=campusos_dev psql -h localhost -U campusos -d campusos -c "
UPDATE users SET password_hash = '\$2a\$10\$新哈希值' WHERE username = 'admin';"
```

### Q3: 数据库连接失败？

**A**: 检查清单：
```bash
# 1. 容器是否运行
docker ps | grep postgres

# 2. 端口是否监听
ss -tlnp | grep "${POSTGRES_PORT:-5432}"

# 3. 能否直接连接
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p "${POSTGRES_PORT:-5432}" -U campusos -d campusos -c "SELECT 1;"

# 4. 查看容器日志
docker logs campusos-postgres --tail 50

# 5. 如果 pgAdmin 里 Host=postgres 失败，检查两个容器是否都在同一个 Compose 网络
docker compose ps
docker exec campusos-pgadmin getent hosts postgres
```

### Q4: 如何查看某个用户的完整权限？

**A**:
```sql
SELECT DISTINCT r.name AS role, rp.resource, rp.action
FROM users u
JOIN user_roles ur ON u.id = ur.user_id
JOIN roles r ON ur.role_id = r.id
JOIN role_permissions rp ON r.id = rp.role_id
WHERE u.username = '某用户名'
ORDER BY rp.resource, rp.action;
```

### Q5: 如何添加自定义角色？

**A**:
```sql
-- 1. 创建角色
INSERT INTO roles (name, description, is_system) VALUES ('vip', 'VIP 会员', FALSE);

-- 2. 分配权限（假设角色 ID 为 5）
INSERT INTO role_permissions (role_id, resource, action) VALUES
  (5, 'thread', 'create'),
  (5, 'thread', 'read'),
  (5, 'thread', 'update'),
  (5, 'post', 'create'),
  (5, 'post', 'read');

-- 3. 分配给用户
INSERT INTO user_roles (user_id, role_id) VALUES (用户ID, 5);
```

### Q6: 如何查看数据库迁移执行状态？

**A**: 当前没有版本跟踪表（如 golang-migrate 的 schema_migrations），迁移通过手动执行 SQL 文件。建议：
- 每次迁移后记录执行日志
- 或引入 migrate 工具自动管理版本
