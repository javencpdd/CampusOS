# CampusOS 数据库管理指南

> 更新日期：2026-07-02
> 适用范围：当前 CampusOS 开发环境、Docker Compose PostgreSQL、pgAdmin、`migrations/000001` 到 `000010`
> 说明：本文以仓库中的 migration 和当前 Docker Compose 配置为准，不再沿用早期旧表结构说明。

## 1. 快速结论

当前 CampusOS 使用 PostgreSQL 16，数据库服务主要由 Docker Compose 提供。本机只需要 Go、Node/npm/pnpm 等开发工具，PostgreSQL、Redis、NATS、pgAdmin 默认由 Docker 启动。

| 项目 | 当前值 |
| --- | --- |
| Compose 服务名 | `postgres` |
| 容器名 | `campusos-postgres` |
| 数据库 | `campusos` |
| 用户 | `campusos` |
| 密码 | `campusos_dev` |
| 容器内端口 | `5432` |
| 宿主机端口 | 默认 `${POSTGRES_PORT:-5432}` |
| 数据卷 | Compose 卷 `postgres-data`，实际通常显示为 `campusos_postgres-data` |
| 迁移记录表 | `schema_migrations` |

如果宿主机已经有 PostgreSQL 占用 `5432`，推荐在项目根目录 `.env` 中使用：

```env
POSTGRES_PORT=5433
DATABASE_DSN=postgres://campusos:campusos_dev@localhost:5433/campusos?sslmode=disable
```

此时连接规则是：

| 场景 | Host | Port |
| --- | --- | --- |
| 后端 `DATABASE_DSN` | `localhost` | `5433` |
| 宿主机 `psql` | `127.0.0.1` | `5433` |
| pgAdmin 页面里注册 CampusOS 数据库 | `postgres` | `5432` |
| 进入 PostgreSQL 容器后连接 | 不需要 host | 容器内本地连接 |

pgAdmin 中不要填 `127.0.0.1` 连接 CampusOS Docker PostgreSQL。pgAdmin 自己也在容器内，`127.0.0.1` 指 pgAdmin 容器自身，不是 `campusos-postgres`。

当前开发机已清理宿主机 `127.0.0.1:5432` 上的本地 `campusos` 数据库。后续如果执行：

```bash
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5432 -U campusos -d campusos
```

看到 `数据库 "campusos" 不存在` 是预期结果。查询 CampusOS Docker 数据库应使用：

```bash
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5433 -U campusos -d campusos
```

或：

```bash
docker exec -it campusos-postgres psql -U campusos -d campusos
```

当前 `scripts/migrate.sh` 已支持自动选择 `psql` 执行方式：宿主机有 `psql` 时走宿主机连接；宿主机没有 `psql` 但 `campusos-postgres` 容器正在运行时，会自动改用容器内的 `psql`。因此卸载宿主机 PostgreSQL 后，`make migrate-up` 仍然可以作用到 Docker PostgreSQL。

## 2. 启动与连接

### 2.1 启动基础服务

```bash
make docker-up
```

该命令会调用 `scripts/docker-up.sh`，按端口占用情况启动：

```text
postgres
redis
nats
pgadmin
```

也可以手动启动指定服务：

```bash
docker compose up -d postgres pgadmin
```

如果修改了 `.env` 中的 `POSTGRES_PORT`，建议重建容器使端口映射生效：

```bash
docker compose up -d --force-recreate postgres pgadmin
```

该命令不会删除 volume，不会清空数据库数据。

### 2.2 宿主机命令行连接

默认端口：

```bash
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5432 -U campusos -d campusos
```

如果 `.env` 设置了 `POSTGRES_PORT=5433`：

```bash
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5433 -U campusos -d campusos
```

也可以通过环境变量写法：

```bash
PGPASSWORD=campusos_dev psql \
  -h 127.0.0.1 \
  -p "${POSTGRES_PORT:-5432}" \
  -U campusos \
  -d campusos
```

### 2.3 容器内连接

```bash
docker exec -it campusos-postgres psql -U campusos -d campusos
```

### 2.4 两种 `psql` 连接方式的区别

常见会看到两类命令：

```bash
docker exec -it campusos-postgres psql -U campusos -d campusos
```

```bash
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5432 -U campusos -d campusos
```

通俗理解：

| 命令 | 从哪里连接 | 走不走宿主机端口 | 什么时候用 |
| --- | --- | --- | --- |
| `docker exec -it campusos-postgres psql ...` | 先进入 PostgreSQL 容器内部，再在容器里运行 `psql` | 不走宿主机 `5432/5433` 映射 | 容器已经启动，但宿主机端口冲突或不想管端口时 |
| `PGPASSWORD=... psql -h 127.0.0.1 -p ...` | 在宿主机直接运行本机安装的 `psql` | 走 Docker 暴露到宿主机的端口 | 平时在项目目录、本机终端里直接查库 |

可以把它们理解成：

| 类比 | 说明 |
| --- | --- |
| `docker exec ...` | 进到数据库所在的“房间”里面操作数据库 |
| `psql -h 127.0.0.1 -p ...` | 站在宿主机“门外”，通过映射出来的端口访问数据库 |

参数含义：

| 参数 | 含义 | CampusOS 默认值 |
| --- | --- | --- |
| `-U` | 数据库用户名 | `campusos` |
| `-d` | 数据库名，不是密码 | `campusos` |
| `-h` | 数据库地址 | 宿主机连接时用 `127.0.0.1` |
| `-p` | 数据库端口 | 默认 `5432`，端口冲突时可能是 `5433` |
| `PGPASSWORD` | 数据库密码 | `campusos_dev` |

注意下面这个命令通常是不正确的：

```bash
docker exec -it campusos-postgres psql -U campusos -d campusos_dev
```

原因是 `-d campusos_dev` 表示连接名为 `campusos_dev` 的数据库；但本项目默认数据库名是 `campusos`，`campusos_dev` 是密码。正确写法是：

```bash
docker exec -it campusos-postgres psql -U campusos -d campusos
```

如果宿主机端口已经改成 `5433`，宿主机连接命令也要改端口：

```bash
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5433 -U campusos -d campusos
```

### 2.5 后端连接字符串

默认：

```env
DATABASE_DSN=postgres://campusos:campusos_dev@localhost:5432/campusos?sslmode=disable
```

宿主机端口改为 `5433` 时：

```env
DATABASE_DSN=postgres://campusos:campusos_dev@localhost:5433/campusos?sslmode=disable
```

后端不会自动把 `POSTGRES_PORT` 拼进 `DATABASE_DSN`。如果改了 `POSTGRES_PORT`，需要同步修改 `DATABASE_DSN`。

后端启动时会按“进程环境变量 -> 项目根目录 `.env` -> 代码默认值”的顺序读取配置。建议从项目根目录启动后端，确保 `.env` 中的 `DATABASE_DSN` 生效。

## 3. pgAdmin 使用

### 3.1 访问地址和登录账号

启动 pgAdmin：

```bash
make docker-up
# 或
docker compose up -d pgadmin
```

访问：

```text
http://localhost:5050
```

登录账号：

| 项目 | 值 |
| --- | --- |
| Email | `admin@campusos.dev` |
| Password | `pgadmin123` |

### 3.2 注册 CampusOS 数据库连接

登录 pgAdmin 后：

1. 右键 `Servers`
2. 选择 `Register` -> `Server`
3. `General` 页签填写 `Name = CampusOS`
4. `Connection` 页签填写：

| 项目 | 值 |
| --- | --- |
| Host name/address | `postgres` |
| Port | `5432` |
| Maintenance database | `campusos` |
| Username | `campusos` |
| Password | `campusos_dev` |
| Save password | 可勾选 |

关键点：

| 不要这样填 | 原因 |
| --- | --- |
| `127.0.0.1:5432` | 在 pgAdmin 容器里代表 pgAdmin 自己 |
| `localhost:5432` | 同上 |
| `campusos-postgres:5433` | `5433` 是宿主机映射端口，不是容器内端口 |

正确方式是：

```text
Host = postgres
Port = 5432
```

### 3.3 pgAdmin 连接排查

检查 pgAdmin 页面是否可访问：

```bash
curl -I --max-time 10 http://127.0.0.1:5050/
```

正常会返回 `302` 到 `/login`。

检查 pgAdmin 容器能否解析 PostgreSQL 服务名：

```bash
docker exec campusos-pgadmin getent hosts postgres
```

检查 pgAdmin 容器能否连到 PostgreSQL 容器端口：

```bash
docker exec campusos-pgadmin python3 -c 'import socket; s=socket.create_connection(("postgres",5432),5); print("postgres:5432 reachable"); s.close()'
```

## 4. 5432 端口冲突处理

如果启动容器时看到：

```text
failed to bind host port 0.0.0.0:5432/tcp: address already in use
```

说明宿主机 `5432` 已经被本机 PostgreSQL 或其他容器占用。先确认：

```bash
ss -ltnp | grep ':5432' || true
docker ps -a --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}" | grep 5432 || true
```

推荐处理方式：

1. 在 `.env` 中设置：

```env
POSTGRES_PORT=5433
DATABASE_DSN=postgres://campusos:campusos_dev@localhost:5433/campusos?sslmode=disable
```

2. 重建容器：

```bash
docker compose up -d --force-recreate postgres pgadmin
```

3. 验证：

```bash
docker compose ps postgres pgadmin
docker exec campusos-pgadmin getent hosts postgres
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5433 -U campusos -d campusos -c 'SELECT 1;'
```

不建议用 `docker network connect` 修复端口冲突。端口冲突是宿主机端口映射问题，不是单纯的 Docker 网络别名问题。

## 5. 迁移机制

迁移文件位于：

```text
migrations/
```

当前已存在：

| 版本 | 文件 | 作用 |
| --- | --- | --- |
| `000001` | `000001_init_schema` | 初始化身份、社区、系统基础表 |
| `000002` | `000002_add_roles` | 增加 RBAC：角色、用户角色、权限 |
| `000003` | `000003_add_plugins` | 增加插件和 API Key 表 |
| `000004` | `000004_seed_admin` | 初始化默认管理员和默认版块 |
| `000005` | `000005_plugin_schema_alignment` | 增加插件权限和插件日志表 |
| `000006` | `000006_add_ai_call_logs` | 增加 AI Gateway 调用日志 |
| `000007` | `000007_add_user_spaces` | 增加用户个人主页配置 |
| `000008` | `000008_add_user_space_contents` | 增加个人主页同步内容 |
| `000009` | `000009_add_user_space_styles` | 增加个人主页风格包状态 |
| `000010` | `000010_fix_admin_seed_password` | 修复旧 admin 种子密码哈希 |

常用命令：

```bash
make migrate-up
make migrate-down
make migrate-reset
make migrate-status
```

实现原理：

| 机制 | 说明 |
| --- | --- |
| 文件发现 | `up` 扫描 `migrations/*.up.sql`，按文件名排序执行 |
| 状态记录 | `schema_migrations(version, name, applied_at)` 记录已执行版本 |
| 幂等跳过 | 已记录的版本会 `skip`，不会重复执行 |
| 执行工具 | 优先使用宿主机 `psql`；如果宿主机没有 `psql`，自动使用 `docker exec campusos-postgres psql` |
| 配置来源 | 优先读取 `.env`，然后使用脚本默认值 |

`make migrate-up` 不需要在 Makefile 里手写每个 SQL 文件。新增 `000011_xxx.up.sql` 后，脚本会自动按顺序纳入。

如果需要强制使用某一种执行方式，可以在命令前设置：

```bash
PSQL_MODE=host make migrate-up
PSQL_MODE=docker make migrate-up
```

默认容器名是 `campusos-postgres`。如果以后调整容器名，可以通过 `POSTGRES_CONTAINER` 覆盖：

```bash
POSTGRES_CONTAINER=新的容器名 PSQL_MODE=docker make migrate-up
```

## 6. 当前表结构总览

当前业务/系统表共 21 张，不含 `schema_migrations`。

| 领域 | 表 |
| --- | --- |
| 身份域 | `users`、`accounts`、`sessions` |
| 社区域 | `categories`、`threads`、`posts`、`tags`、`likes` |
| 系统域 | `audit_logs`、`notifications`、`configurations` |
| RBAC | `roles`、`user_roles`、`permissions` |
| 插件域 | `plugins`、`api_keys`、`plugin_permissions`、`plugin_logs` |
| AI | `ai_call_logs` |
| 个人主页 | `user_spaces`、`user_space_contents` |

### 6.1 身份域

#### `users`

保存用户基础资料，不保存密码哈希。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 雪花 ID，BIGINT 主键 |
| `username` | 用户名，软删除条件下唯一 |
| `nickname` | 昵称 |
| `email` | 邮箱，软删除条件下唯一 |
| `avatar` / `bio` | 头像和个人简介 |
| `status` | `active` 等状态 |
| `created_at` / `updated_at` / `deleted_at` | 创建、更新、软删除时间 |

#### `accounts`

保存登录凭据。邮箱密码登录使用：

| 字段 | 说明 |
| --- | --- |
| `type` | 账号类型，例如 `email` |
| `identifier` | 邮箱或其他登录标识 |
| `credential` | bcrypt 密码哈希 |
| `verified` | 是否已验证 |
| `user_id` | 关联 `users.id` |

注意：当前密码哈希在 `accounts.credential`，不是 `users.password_hash`。

#### `sessions`

保存刷新 token 和设备信息，字段包括 `refresh_token`、`device_id`、`device_name`、`ip_address`、`user_agent`、`last_active_at`、`expires_at`。

### 6.2 社区域

#### `categories`

版块表。核心字段包括 `name`、`slug`、`description`、`parent_id`、`sort_order`、`thread_count`、`post_count`、`is_closed`。

#### `threads`

主题帖表。核心字段包括 `title`、`content`、`author_id`、`author_name`、`category_id`、`status`、`tags`、`metadata`、置顶/锁定/高亮和统计字段。

`tags` 是 `TEXT[]`，默认 `{}`。

#### `posts`

回复表。核心字段包括 `thread_id`、`author_id`、`parent_id`、`content`、`status`、`floor_number`、`metadata`。

#### `tags` / `likes`

`tags` 保存标签字典和帖子计数。

`likes` 使用 `target_type + target_id` 支持对 thread/post 点赞。

### 6.3 系统域

| 表 | 作用 |
| --- | --- |
| `audit_logs` | 审计日志，只追加写入，记录 trace、操作者、资源、前后数据和 metadata |
| `notifications` | 用户通知 |
| `configurations` | 系统配置，支持 `is_secret` 标识敏感配置 |

### 6.4 RBAC

| 表 | 作用 |
| --- | --- |
| `roles` | 系统角色，默认 `admin`、`moderator`、`member`、`guest` |
| `user_roles` | 用户角色映射，支持 `scope_type` 和 `scope_id` |
| `permissions` | 角色权限，字段为 `resource` 和 `action` |

默认管理员角色为 `admin`，拥有 user/thread/post/category/role 相关权限。

### 6.5 插件域

| 表 | 作用 |
| --- | --- |
| `plugins` | 插件记录，保存名称、版本、runtime、status、config、api_key 等 |
| `api_keys` | API Key，支持用户或插件绑定 |
| `plugin_permissions` | 插件权限声明 |
| `plugin_logs` | 插件运行日志和 Host API 调用日志 |

### 6.6 AI 与个人主页

| 表 | 作用 |
| --- | --- |
| `ai_call_logs` | AI Gateway 调用日志，记录 provider、model、source、status、耗时和 token 统计 |
| `user_spaces` | 用户个人主页配置，包括标题、简介、主题、布局、同步规则和风格包状态 |
| `user_space_contents` | 同步到个人主页的公开内容，按 `thread_id` 唯一 |

`user_space_contents.tags` 是 `TEXT[] NOT NULL DEFAULT '{}'`。后端写入时必须传空数组而不是 SQL `NULL`，否则会违反 not-null 约束。

## 8. 常用运维命令

当前开发机的 CampusOS Docker PostgreSQL 使用宿主机端口 `5433`。如果以后修改 `.env` 中的 `POSTGRES_PORT`，下面命令中的端口也要同步调整。

### 8.1 查看表

```bash
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5433 -U campusos -d campusos -c '\dt'
```

### 8.2 查看迁移状态

```bash
make migrate-status
```

或：

```bash
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5433 -U campusos -d campusos -c \
  "SELECT version, name, applied_at FROM schema_migrations ORDER BY version;"
```

### 8.3 备份数据库

```bash
mkdir -p backups
PGPASSWORD=campusos_dev pg_dump \
  -h 127.0.0.1 \
  -p 5433 \
  -U campusos \
  -d campusos \
  -F c \
  -f backups/campusos-$(date +%Y%m%d-%H%M%S).dump
```

### 8.4 恢复数据库

恢复前应确认目标数据库可以被覆盖。开发环境可以先备份，再恢复：

```bash
PGPASSWORD=campusos_dev pg_restore \
  -h 127.0.0.1 \
  -p 5433 \
  -U campusos \
  -d campusos \
  --clean \
  --if-exists \
  backups/campusos-YYYYMMDD-HHMMSS.dump
```

### 8.5 查看容器和日志

```bash
docker compose ps
docker logs campusos-postgres --tail 80
docker logs campusos-pgadmin --tail 80
```

## 9. 排查清单

### 9.1 后端无法连接数据库

检查：

```bash
cat .env
docker compose ps postgres
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5433 -U campusos -d campusos -c 'SELECT 1;'
```

如果 `.env` 设置了 `POSTGRES_PORT=5433`，但 `DATABASE_DSN` 仍是 `localhost:5432`，后端会连到错误数据库或连接失败。需要同步改为：

```env
DATABASE_DSN=postgres://campusos:campusos_dev@localhost:5433/campusos?sslmode=disable
```

### 9.2 宿主机 psql 和 docker exec 查询结果不一致

如果出现：

```bash
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5432 -U campusos -d campusos
```

能查到完整数据，但：

```bash
docker exec -it campusos-postgres psql -U campusos -d campusos
```

查到的数据不全，通常说明宿主机 `5432` 和 Docker 容器 PostgreSQL 不是同一个实例。详细说明见：

```text
docs/help/PostgreSQL双实例数据不一致说明.md
```

当前开发机已经按该说明清理了宿主机 `5432` 上的本地 `campusos` 数据库，并保留备份：

```text
/tmp/campusos-db-backups/campusos-host5432-20260702-180921.dump
```

后续以 Docker PostgreSQL 为准：

```bash
PGPASSWORD=campusos_dev psql -h 127.0.0.1 -p 5433 -U campusos -d campusos
```

### 9.3 pgAdmin 连接失败

优先检查：

```bash
docker compose ps postgres pgadmin
docker exec campusos-pgadmin getent hosts postgres
docker exec campusos-pgadmin python3 -c 'import socket; s=socket.create_connection(("postgres",5432),5); print("ok"); s.close()'
```

pgAdmin 注册服务器时必须使用：

```text
Host = postgres
Port = 5432
```

### 9.4 `make migrate-up` 没有执行新文件

检查文件命名：

```text
000011_xxx.up.sql
000011_xxx.down.sql
```

检查是否已经记录：

```bash
make migrate-status
```

如果 `schema_migrations` 已有该 version，`migrate-up` 会跳过。

### 9.5 `make migrate-up` 提示 `psql: 未找到命令`

旧版脚本直接调用宿主机 `psql`，宿主机卸载 PostgreSQL 或未安装 `postgresql-client` 时会报：

```text
psql: 未找到命令
```

当前脚本已经改为自动 fallback：

| 场景 | 行为 |
| --- | --- |
| 宿主机有 `psql` | 使用宿主机 `psql` 连接 `DB_HOST:DB_PORT` |
| 宿主机没有 `psql`，但 `campusos-postgres` 正在运行 | 使用 `docker exec -i campusos-postgres psql` 执行迁移 |
| 两者都不可用 | 停止并提示安装 `postgresql-client` 或启动 Docker PostgreSQL |

所以在当前 Docker 开发模式下，只要先确认容器运行：

```bash
docker ps --filter name=campusos-postgres
make migrate-up
```

迁移脚本会把本地 `migrations/*.up.sql` 内容通过标准输入送进容器内的 `psql` 执行。SQL 文件仍然来自项目目录，作用目标是 Docker 中的 `campusos` 数据库。

## 10. 生产环境提醒

当前配置面向本地开发。生产环境至少需要调整：

| 项目 | 要求 |
| --- | --- |
| 数据库密码 | 不使用 `campusos_dev` |
| pgAdmin 密码 | 不使用 `pgadmin123` |
| 管理员密码 | 首次部署后立即修改 `Admin@123456` |
| 端口暴露 | PostgreSQL 不应直接暴露到公网 |
| 备份 | PostgreSQL 数据、插件数据目录、上传文件和配置一起纳入备份 |
| 密钥 | JWT、AI Provider Key、API Key 不进入日志或公开仓库 |

本项目当前仍以 Ubuntu/WSL2 + Docker Compose 为主开发路径。
