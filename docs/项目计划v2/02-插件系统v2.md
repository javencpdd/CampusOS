# 插件系统 v2 — API 获取、权限管理与存储策略

> **版本**：v2.0（基于 v1 插件系统设计 + 实际开发经验优化）
> **核心改进**：明确插件如何获取和调用 CampusOS API、权限分层模型、数据存储策略

## 一、插件如何获取与使用 CampusOS API

### 1.1 两种 API 访问模式

```
┌─────────────────────────────────────────────────────────┐
│                   插件访问 CampusOS API                    │
│                                                          │
│  模式 1: gRPC 插件（进程外）          模式 2: Wasm 插件    │
│  ┌──────────────────────┐           ┌──────────────────┐ │
│  │  独立 Go 进程          │           │  Wasm 沙箱内      │ │
│  │                       │           │                  │ │
│  │  方式A: Host API      │           │  方式: Host API   │ │
│  │  (gRPC 反向调用)      │           │  (宿主函数调用)   │ │
│  │                       │           │                  │ │
│  │  方式B: REST API      │           │  ❌ 不能直接      │ │
│  │  (HTTP 直接调用)      │           │  调用 HTTP       │ │
│  │  需要 API Key        │           │                  │ │
│  └──────────────────────┘           └──────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

### 1.2 Host API（推荐方式）

Host API 是插件调用 CampusOS 核心能力的**统一抽象层**，无论 Wasm 还是 gRPC 插件都通过同一套接口访问。

#### gRPC 插件调用 Host API

```go
// 插件代码（独立进程）
package main

import (
    "github.com/campusos/sdk/grpc/hostapi"
)

func main() {
    // 1. 创建 Host API 客户端（自动连接 CampusOS 主进程）
    host := hostapi.NewClient()

    // 2. 调用身份服务
    user, err := host.Identity.GetUser(ctx, "user-123")

    // 3. 查询帖子数据
    threads, err := host.Data.Query(ctx, "thread", &hostapi.Filter{
        Where: "category_id = ? AND status = ?",
        Args:  []interface{}{"cat-001", "published"},
        Limit: 20,
    })

    // 4. 发布事件
    err := host.Event.Publish(ctx, &hostapi.Event{
        Type:    "moderation.result",
        Subject: "thread.thread-456",
        Data:    map[string]interface{}{"safe": true},
    })

    // 5. 发送通知
    err := host.Notification.Send(ctx, "user-123", "thread_replied", map[string]string{
        "thread_title": "Hello",
        "replier":      "Alice",
    })

    // 6. 调用 AI 网关
    result, err := host.AI.ChatCompletion(ctx, []hostapi.Message{
        {Role: "system", Content: "你是内容审核助手"},
        {Role: "user", Content: thread.Content},
    })

    // 7. 插件自己的数据存储
    err := host.Storage.SQLite.Exec(ctx, "INSERT INTO logs (action, ts) VALUES (?, ?)", "approved", time.Now())
    rows, err := host.Storage.SQLite.Query(ctx, "SELECT * FROM logs WHERE action = ?", "approved")
}
```

#### Wasm 插件调用 Host API

```go
// Wasm 插件代码（编译为 .wasm）
package main

import "github.com/campusos/sdk/wasm"

func main() {
    wasm.Handle("onThreadBeforeCreate", func(ctx *wasm.Context, event *wasm.Event) *wasm.Result {
        // 通过 ctx 访问 Host API
        user, err := ctx.Host.Identity.GetUser(event.Data.AuthorID)
        if err != nil {
            return wasm.Allow() // 获取失败时放行
        }

        // 查询用户历史帖子数
        count, _ := ctx.Host.Data.Count(ctx, "thread", &wasm.Filter{
            Where: "author_id = ?",
            Args:  []interface{}{user.ID},
        })

        // 新用户限制：帖子数 < 5 时需要审核
        if count < 5 {
            return wasm.Block("新用户帖子需人工审核")
        }

        return wasm.Allow()
    })
}
```

### 1.3 REST API 直接调用（仅 gRPC 插件可用）

gRPC 插件作为独立进程，可以直接通过 HTTP 调用 CampusOS REST API：

```go
// 插件直接调用 REST API
resp, err := http.Get("http://localhost:8080/api/v1/threads?keyword=campus")

// 带认证的调用
req, _ := http.NewRequest("GET", "http://localhost:8080/api/v1/auth/me", nil)
req.Header.Set("Authorization", "Bearer "+apiKey)
resp, err := http.DefaultClient.Do(req)
```

**API Key 认证方式（v0.2 规划）：**

```bash
# 插件注册时获得 API Key
curl -X POST http://localhost:8080/api/v1/plugins/register \
  -H "Content-Type: application/json" \
  -d '{"name":"my-plugin","permissions":["thread:read","user:read"]}'

# 返回
{
  "api_key": "pk_live_xxxxxxxxxxxx",
  "permissions": ["thread:read", "user:read"]
}

# 使用 API Key 调用
curl -H "Authorization: Bearer pk_live_xxxxxxxxxxxx" \
  http://localhost:8080/api/v1/threads
```

### 1.4 三种访问方式对比

| 方式 | 适用运行时 | 认证方式 | 优点 | 缺点 |
|:---|:---|:---|:---|:---|
| **Host API（gRPC）** | gRPC | 自动（进程间连接） | 权限自动检查、类型安全、低延迟 | 需要 SDK |
| **Host API（Wasm）** | Wasm | 自动（宿主函数） | 唯一方式、安全沙箱 | 仅限声明的 Host Functions |
| **REST API 直接调用** | gRPC | API Key | 灵活、可用标准 HTTP 客户端 | 需管理 Key、延迟略高 |

## 二、插件权限管理（v2 优化）

### 2.1 三层权限模型

```
┌─────────────────────────────────────────────┐
│              权限分层模型                      │
│                                              │
│  第一层：API 级权限                           │
│  ┌──────────────────────────────────────┐   │
│  │  插件可以调用哪些 API                  │   │
│  │  thread:read, thread:write, user:read│   │
│  └──────────────────────────────────────┘   │
│                     │                        │
│                     ▼                        │
│  第二层：资源级权限                           │
│  ┌──────────────────────────────────────┐   │
│  │  可以操作哪些数据                      │   │
│  │  - 只能读取特定版块的帖子              │   │
│  │  - 只能读取公开用户信息                │   │
│  └──────────────────────────────────────┘   │
│                     │                        │
│                     ▼                        │
│  第三层：字段级权限                           │
│  ┌──────────────────────────────────────┐   │
│  │  可以看到哪些字段                      │   │
│  │  - 能看 username，不能看 email        │   │
│  │  - 能看 content，不能看 metadata      │   │
│  └──────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
```

### 2.2 权限声明（Manifest）

```yaml
permissions:
  # ── API 级权限 ──
  api:
    - resource: "thread"
      actions: ["read"]              # 只读
    - resource: "post"
      actions: ["read"]
    - resource: "user"
      actions: ["read"]
    - resource: "notification"
      actions: ["send"]
    - resource: "event"
      actions: ["publish"]

  # ── 资源级权限（可选，默认全部）──
  dataScope:
    - resource: "thread"
      categories: ["tech", "help"]   # 只能访问这些版块
    - resource: "user"
      status: ["active"]             # 只能访问活跃用户

  # ── 字段级权限（可选，默认全部）──
  fieldScope:
    - resource: "user"
      allowed: ["id", "username", "nickname", "status"]
      denied: ["email", "avatar", "bio"]
    - resource: "thread"
      allowed: ["id", "title", "content", "status", "author_id", "tags"]
      denied: ["metadata"]
```

### 2.3 权限检查流程

```
插件调用 host.Data.Query("thread", filter)
         │
         ▼
Plugin Manager 检查权限:
  ┌─────────────────────────────────┐
  │ 1. API 级: 有 thread:read 权限？ │
  │    ├── 否 → 返回 PermissionDenied│
  │    └── 是 → 继续 ↓              │
  │                                  │
  │ 2. 资源级: 过滤 categories/status│
  │    添加 WHERE category_id IN (...)│
  │                                  │
  │ 3. 字段级: 过滤返回字段           │
  │    SELECT id, title, content ... │
  │    （不返回 denied 字段）         │
  └─────────────────────────────────┘
         │
         ▼
返回过滤后的数据
```

### 2.4 角色系统（v0.2 实现）

| 角色 | 说明 | 权限 |
|:---|:---|:---|
| **admin** | 系统管理员 | 全部权限 |
| **moderator** | 版主 | 管理帖子/用户（可按版块分配） |
| **member** | 普通会员 | 发帖/回帖/点赞 |
| **guest** | 未登录 | 只读浏览 |

**插件角色分配：**

```yaml
# plugin.yaml
roles:
  - name: "moderator"
    description: "内容审核管理员"
    permissions:
      api:
        - resource: "thread"
          actions: ["read", "update"]    # 可以修改帖子状态
        - resource: "post"
          actions: ["read", "delete"]    # 可以删除回复
        - resource: "user"
          actions: ["read", "suspend"]   # 可以封禁用户
```

### 2.5 RBAC + 插件权限的关系

```
用户操作 → CampusOS 核心权限检查（RBAC）
                │
                ├── 有权限 → 执行操作 → 发布事件
                │                           │
                │                           ▼
                │                    插件权限检查
                │                    （插件的 Manifest 权限）
                │                           │
                │                           ├── 有权限 → 插件处理
                │                           └── 无权限 → 跳过
                │
                └── 无权限 → 返回 403
```

**关键设计**：核心 RBAC 和插件权限是独立的两层，互不干扰：
- 核心 RBAC 控制"用户能做什么"
- 插件权限控制"插件能访问什么数据"
- 即使插件有 `thread:read` 权限，也不能读取被 RBAC 策略过滤的帖子

## 三、插件数据存储策略（v2 确定）

### 三种方案

| 方案 | 类型标识 | 适用场景 |
|:---|:---|:---|
| **独立 SQLite** | `sqlite` | 大多数插件（推荐） |
| **共享 PostgreSQL** | `postgresql` | 需要关联核心数据 |
| **无存储** | `none` | 无状态插件 |

### Manifest 声明

```yaml
storage:
  type: "sqlite"           # sqlite | postgresql | none
  sqlite:
    filename: "data.db"    # 自动创建在插件目录
  postgresql:
    tables:
      - name: "moderation_logs"
        schema: "CREATE TABLE plugin_moderation_logs (...)"
```

### 安全控制

- **SQLite**：文件级隔离，插件只能访问自己的 `data.db`
- **PostgreSQL**：表级隔离，表名带 `plugin_` 前缀，插件只能操作 Manifest 声明的表
- **大小限制**：SQLite 默认 100MB 上限，可配置

## 四、插件获取 CampusOS 数据的完整流程

### 4.1 事件驱动模式（被动获取）

```
用户发帖 → CampusOS 创建帖子 → 发布 thread.created 事件
                                      │
                                      ▼
                               Plugin Manager 分发事件
                                      │
                    ┌─────────────────┼─────────────────┐
                    ▼                 ▼                 ▼
              Plugin A           Plugin B           Plugin C
              (内容审核)         (统计分析)         (QQ 同步)
              
              检查帖子内容      更新统计数据       同步到 QQ 群
              调用 AI 判断      写入 SQLite        调用 NapCat API
```

**插件代码：**
```go
wasm.Handle("onThreadCreated", func(ctx *wasm.Context, event *wasm.Event) *wasm.Result {
    // 事件 payload 包含完整帖子数据
    var thread Thread
    event.UnmarshalPayload(&thread)
    
    // 不需要再调 API 获取帖子，事件中已有全部数据
    ctx.Host.Storage.SQLite.Exec(ctx, 
        "INSERT INTO threads (id, title) VALUES (?, ?)", 
        thread.ID, thread.Title)
    
    return wasm.Allow()
})
```

### 4.2 查询模式（主动获取）

```
插件需要数据 → 调用 Host API → Plugin Manager 检查权限 → 返回数据

host.Data.Query("thread", filter)
    │
    ▼
权限检查: api + dataScope + fieldScope
    │
    ▼
PostgreSQL 查询（带过滤条件）
    │
    ▼
返回过滤后的数据给插件
```

### 4.3 实时流模式（长连接）

```
gRPC 插件启动 → 建立 StreamEvents 连接 → 持续接收事件流

Plugin Manager ←─gRPC Stream──→ Plugin
     │                              │
     │  thread.created event        │
     │  ────────────────────────→   │
     │                              │
     │  post.created event          │
     │  ────────────────────────→   │
     │                              │
     │  ←────────────────────────   │
     │  host.event.publish(...)     │
     │  (插件发布自定义事件)         │
```

## 五、插件开发流程总结

### 开发一个插件的步骤

```
1. 创建插件目录
   plugins/my-plugin/
   ├── plugin.yaml          # Manifest
   ├── main.go              # 代码
   └── data.db              # SQLite（自动创建）

2. 编写 Manifest
   - 声明运行时（wasm / grpc）
   - 声明事件订阅
   - 声明权限
   - 声明存储

3. 实现事件处理函数
   - 接收事件 payload
   - 调用 Host API 获取/操作数据
   - 处理业务逻辑

4. 编译
   - Wasm: GOOS=wasip1 GOARCH=wasm go build -o main.wasm
   - gRPC: go build -o plugin-server

5. 注册到 CampusOS
   - Plugin Manager 加载 Manifest
   - 验证权限声明
   - 初始化运行时

6. 测试
   - 触发事件，验证插件响应
   - 检查权限边界
   - 检查数据隔离
```

## 六、与 v1 插件系统设计的差异

| 维度 | v1 设计 | v2 优化 |
|:---|:---|:---|
| **API 获取** | 模糊的 Host API 列表 | 明确三种方式（Host API gRPC / Host API Wasm / REST 直接调用） |
| **权限模型** | 单层权限清单 | 三层模型（API 级 + 资源级 + 字段级） |
| **角色系统** | 未涉及 | RBAC 角色（admin/moderator/member/guest） |
| **数据存储** | 未明确 | 三种方案（SQLite/PostgreSQL/none）+ Manifest 声明 |
| **插件认证** | 未涉及 | API Key + Host API 自动认证 |
| **数据获取** | 仅事件驱动 | 三种模式（事件驱动/查询/实时流） |
