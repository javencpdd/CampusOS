昂欠# v1 → v2 变更说明

> 本文档详细说明项目计划从 v1 升级到 v2 的原因、变更内容，以及哪些 v1 文档仍然有效。

## 一、为什么需要 v2

### 1.1 v1 制定时的背景

v1 项目计划在项目**启动前**制定，基于以下假设：

- 一次性规划所有功能，期望 v0.1 就完成大部分基础能力
- 假设开发者对技术栈（Go/PostgreSQL/NATS/Wasm/gRPC）都非常熟悉
- 没有考虑到实际开发中的节奏和优先级调整

### 1.2 实际开发中发现的问题

| 问题 | 说明 |
|:---|:---|
| **v0.1 范围过大** | 计划 32 个 API + gRPC 插件 + RBAC 权限，实际 9 个迭代只完成约 60% |
| **前端未规划** | v1 完全没有前端规划，但 v0.1.2 就需要创建 Vue 3 UI |
| **插件存储未明确** | v1 的插件系统文档没有说明插件如何存储数据，开发过程中才讨论确定 |
| **权限模型过于理想** | v1 计划 v0.1 就完成 RBAC + ABAC，实际上连 admin 角色都没实现 |
| **文档粒度不一致** | v1 有 12 个文档，有些非常详细（插件系统 900 行），有些只是框架 |
| **API 端点偏差** | v1 计划 32 个端点，实际实现 19 个，且新增了事件历史查询等 |

### 1.3 v2 的目标

- **基于实际实现**重新评估，而非基于理想规划
- **聚焦变化部分**，不重写仍然有效的 v1 文档
- **补充缺失内容**（插件存储策略、三层权限模型、API 获取方式）
- **修正时间线**，反映实际开发节奏

## 三、核心变更详解

### 3.1 插件系统（03-插件系统.md → 02-插件系统v2.md）

| 维度 | v1 | v2 |
|:---|:---|:---|
| **API 获取** | 列出 Host API 表格，但没有说明插件如何调用 | 明确三种方式 + 完整代码示例 |
| **API 访问模式** | 未区分 Wasm/gRPC | 明确 Wasm 只能用 Host API，gRPC 还可以 REST 直接调用 |
| **权限模型** | 单层权限清单（hostAPI + dataScope） | 三层模型（API 级 + 资源级 + 字段级） |
| **数据存储** | 未涉及 | 三种方案（SQLite/PostgreSQL/none）+ Manifest 声明 |
| **数据获取** | 仅事件驱动 | 三种模式（事件驱动/查询/实时流） |
| **插件认证** | 未涉及 | API Key + Host API 自动认证 |
| **与核心 RBAC 关系** | 未说明 | 明确两层独立（核心 RBAC + 插件权限） |

**新增内容：**
- 完整的 gRPC 插件调用 Host API 代码示例
- 完整的 Wasm 插件调用 Host API 代码示例
- REST API 直接调用方式和 API Key 认证
- 三层权限模型的 Manifest 声明方式
- 三种数据获取模式的架构图

### 3.2 权限系统（04-权限系统.md → 03-权限系统v2.md）

| 维度 | v1 | v2 |
|:---|:---|:---|
| **实现计划** | v0.1 完成 RBAC + ABAC | v0.2 完成 RBAC，v0.3 完成插件三层权限 |
| **数据库设计** | 概念描述 | 完整的 SQL（roles/user_roles/permissions 三表） |
| **角色定义** | admin/moderator/user | admin/moderator/member/guest |
| **版主范围** | 未涉及 | 支持按版块分配版主（scope_type=category） |
| **插件权限** | 与用户权限混合 | 独立的三层模型（API/资源/字段） |
| **中间件** | 未给出代码 | 完整的 RequirePermission 中间件代码 |
| **内置数据** | 未给出 | 完整的 INSERT 语句（4 个角色 + 27 条权限） |

### 3.3 开发路线图（10-开发路线图.md → 04-开发路线图v2.md）

### 3.4 架构设计（v1 无独立文档 → 01-架构设计v2.md）

v1 没有独立的架构设计文档，架构信息分散在 `01-总体架构.md` 和 `02-核心设计.md` 中。

v2 新增了独立的架构文档，包含：
- 基于实际实现的完整架构图（前端 → 网关 → Handler → Service → Repository → 基础设施）
- 模块依赖关系图
- 事件总线架构
- 认证流程
- v2 架构规划（插件运行时、Host API）

## 五、使用建议

### 新加入项目的开发者

1. 先读 `v2/00-项目概览与v2修订说明.md` 了解全貌
2. 再读 `v2/01-架构设计v2.md` 了解实际架构
3. 需要了解插件系统时读 `v2/02-插件系统v2.md`
4. 需要了解权限时读 `v2/03-权限系统v2.md`
5. 需要了解其他规范时读 v1 对应文档

### 项目规划参考

- **当前版本进度**：参考 `v2/00-项目概览与v2修订说明.md`
- **下一版本计划**：参考 `v2/04-开发路线图v2.md`
- **技术规范**：参考 v1 的 `06-API设计.md` / `07-数据库设计.md` / `08-开发规范.md`

### 两个目录的关系

```
docs/项目计划v1/     ← 基础规范（仍然有效，作为参考）
docs/项目计划v2/     ← 修订和补充（基于实际开发，覆盖 v1 中变更的部分）
docs/releases/      ← 每个版本的实际发布记录
```

**原则**：v2 不是废除 v1，而是对 v1 中不准确的部分进行修正和补充。v1 中未被 v2 替代的文档仍然是权威参考。
