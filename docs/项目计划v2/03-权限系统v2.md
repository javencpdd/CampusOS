## v0.2 目标：RBAC 角色系统

### 数据库表设计

```sql
-- 角色表
CREATE TABLE roles (
    id          BIGINT PRIMARY KEY,
    name        VARCHAR(32) NOT NULL,    -- admin / moderator / member
    description VARCHAR(255) DEFAULT '',
    is_system   BOOLEAN NOT NULL DEFAULT FALSE,  -- 系统内置角色不可删除
    created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    deleted_at  TIMESTAMP NULL
);

CREATE UNIQUE INDEX uk_roles_name ON roles(name) WHERE deleted_at IS NULL;

-- 用户-角色关联表
CREATE TABLE user_roles (
    id          BIGINT PRIMARY KEY,
    user_id     BIGINT NOT NULL,
    role_id     BIGINT NOT NULL,
    scope_type  VARCHAR(20) DEFAULT 'global',  -- global / category
    scope_id    BIGINT NULL,                    -- category_id (版块版主)
    created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    deleted_at  TIMESTAMP NULL
);

CREATE UNIQUE INDEX uk_user_roles ON user_roles(user_id, role_id, scope_type, scope_id) WHERE deleted_at IS NULL;

-- 权限表
CREATE TABLE permissions (
    id          BIGINT PRIMARY KEY,
    role_id     BIGINT NOT NULL,
    resource    VARCHAR(64) NOT NULL,    -- thread / post / user / category
    action      VARCHAR(32) NOT NULL,    -- read / write / delete / manage
    created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    deleted_at  TIMESTAMP NULL
);

CREATE UNIQUE INDEX uk_permissions ON permissions(role_id, resource, action) WHERE deleted_at IS NULL;
```

### 中间件实现

```go
// RequirePermission 权限检查中间件
func RequirePermission(resource, action string) gin.HandlerFunc {
    return func(c *gin.Context) {
        userID, _ := c.Get("user_id")
        
        // 查询用户角色和权限
        hasPermission, err := permissionService.Check(ctx, userID, resource, action)
        if err != nil || !hasPermission {
            c.JSON(403, gin.H{"code": 40003, "msg": "permission denied"})
            c.Abort()
            return
        }
        c.Next()
    }
}

// 路由注册
admin := v1.Group("")
admin.Use(middleware.JWTAuth(jwtMgr))
admin.Use(middleware.RequirePermission("user", "suspend"))
{
    admin.POST("/users/:id/suspend", userHandler.SuspendUser)
    admin.POST("/users/:id/activate", userHandler.ActivateUser)
}
```

## v0.3 目标：三层权限模型（用于插件）

### 第一层：API 级权限

插件在 Manifest 中声明可以调用哪些 API：

```yaml
permissions:
  api:
    - resource: "thread"
      actions: ["read"]
    - resource: "post"
      actions: ["read"]
```

### 第二层：资源级权限

限制插件可以访问哪些数据：

```yaml
permissions:
  dataScope:
    - resource: "thread"
      categories: ["tech", "help"]    # 只能访问这些版块
    - resource: "user"
      status: ["active"]              # 只能访问活跃用户
```

### 第三层：字段级权限

限制插件可以看到哪些字段：

```yaml
permissions:
  fieldScope:
    - resource: "user"
      allowed: ["id", "username", "nickname", "status"]
      denied: ["email", "avatar", "bio"]
```

### 权限检查流程

```
插件调用 host.Data.Query("thread", filter)
         │
         ▼
Plugin Manager 检查权限:
  1. API 级: 有 thread:read 权限？ → 否则返回 PermissionDenied
  2. 资源级: 过滤 categories/status
  3. 字段级: 过滤返回字段
         │
         ▼
返回过滤后的数据
```

## 与 v1 权限系统的差异

| 维度 | v1 设计 | v2 优化 |
|:---|:---|:---|
| 范围 | 用户权限 + 插件权限混合 | 用户 RBAC + 插件权限独立分层 |
| 插件权限 | 单层权限清单 | 三层模型（API/资源/字段） |
| 角色 | 未明确数据库设计 | roles + user_roles + permissions 三表 |
| 版主范围 | 未涉及 | 支持按版块分配版主（scope_type） |
| 实现顺序 | v0.1 全部完成 | v0.2 用户 RBAC，v0.3 插件权限 |
