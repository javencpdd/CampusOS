# CampusOS v4 实现状态与后续规划总结

> 日期：2026-07-02
> 范围：对照 `docs/项目计划v4/00-v4版本计划书.md` 检查 v4 计划完成情况，并整理后续版本安排
> 结论：按 v4 原计划全量验收，尚未全部实现；按当前项目路线，v0.4-dev 已完成阶段性收口，可进入 v0.5-dev 承接未完成任务。

## 1. 总体结论

严格对照 `00-v4版本计划书.md`，v4 原计划没有全部实现。原因是原计划同时包含 AI Gateway、个人主页、风格包、MCP Server、Webhook、IM/机器人、管理后台运营视图、观测备份和跨平台验证等多个方向，其中 MCP、Webhook、IM/机器人和完整运营后台仍未落地。

但从实际开发进度看，v0.4-dev 已经完成一组可作为新基线的核心能力：

| 领域 | 当前结论 |
| --- | --- |
| v3 技术债收尾 | 已完成主要收尾项，原生 Windows 实机验证按计划跳过 |
| 插件交付 | 已完成最小插件包打包、安装、导入、导出和校验闭环 |
| AI Gateway | 已完成 OpenAI-compatible Provider、配置、限流、调用日志和管理查询的最小闭环 |
| 个人主页 | 已完成公开主页、用户主页配置、内容同步和用户侧管理入口 |
| 风格包 | 已完成风格包格式、校验、预览、导入、导出、一键应用和示例风格包 |
| 数据库与 UI 稳定化 | 已处理 pgAdmin、Docker 数据库、迁移、管理员登录、密码模式和管理后台创建版块等近期问题 |

因此建议采用以下判断：

```text
v4 原计划：未全部完成。
v0.4-dev 阶段：已完成阶段性收口。
后续路线：不要继续把所有剩余外部集成都压在 v4 尾部，应拆入 v0.5-dev 和后续版本。
```

## 6. 后续计划安排

### 6.1 规划调整原则

v4 原计划未完成的任务不应简单全部顺延到 v5。更稳妥的方式是按风险和依赖关系重新分层：

| 层级 | 处理方式 | 原因 |
| --- | --- | --- |
| 现有能力稳定化 | v5 必做 | 个人主页、风格包、插件包、后台和数据库已经进入用户测试，应先保证可靠 |
| 管理后台运营化 | v5 必做 | 没有统一运营入口，后续 MCP、Webhook、IM 即使做出来也难以排错 |
| Webhook | v5 主线 | 事件外发确定性强，是外部自动化和后续 IM 同步的基础 |
| MCP Server | v5 受控实验 | 价值高，但权限和审计风险高，首轮只做只读工具和开关 |
| Message/IM | v5 做协议，真实适配器后置 | QQ/机器人生态依赖外部环境，首轮先固定 Message 协议和 mock/local adapter |
| 完整 Docker、原生 Windows、AI 审核 | 暂时搁置 | 依赖部署、兼容、安全和人工复核体系，等系统核心能力成熟后再推进 |

### 6.2 IM/机器人协议设计判断

IM/机器人集成不建议直接仿照 Discord 机器人协议作为 CampusOS 核心协议。Discord 的 Gateway、Interaction、权限、频道和用户体系都强绑定 Discord 平台；如果直接照搬，后续接入 QQ/NapCat、AstrBot、Telegram、飞书、企业微信或校园自有系统时会被单一平台模型限制。

更稳妥的设计是：

```text
CampusOS Core
  -> CampusOS Message Protocol
  -> Adapter/Plugin
  -> Discord / NapCat / AstrBot / Telegram / Feishu / Custom System
```

| 设计点 | 说明 |
| --- | --- |
| 核心协议 | CampusOS 自己定义平台无关的 Message 协议，描述消息、会话、发送者、附件、命令、权限和关联资源 |
| 可借鉴对象 | 可以借鉴 Discord 机器人的事件、命令、频道、权限和 Webhook 思路，但不把 Discord 协议作为内部标准 |
| 适配方式 | 通过 Adapter 或插件把外部平台协议转换为 CampusOS Message，再把 CampusOS 事件转换为外部平台消息 |
| 插件边界 | 真实 IM 平台适配器应优先作为插件或可替换模块存在，不写死到核心业务 |
| v5 范围 | v5 只做 Message 协议、Adapter 接口、账号绑定模型和 mock/local adapter |
| 后续范围 | Discord、NapCat、AstrBot 等真实适配器放到 v0.6-dev 或专项，根据测试环境和用户需求逐个接入 |

这样做的结果是：CampusOS 不是“仿 Discord 机器人系统”，而是拥有自己的标准消息接口；其他平台通过插件适配这个接口。后续要兼容 Discord，可以做 Discord Adapter；要兼容 QQ/NapCat，可以做 NapCat Adapter；要兼容校园系统，也可以做自定义 Adapter。

### 6.3 v0.5-dev 优化后主线

v0.5-dev 建议定位为“运营化与低风险开放集成阶段”，而不是“把所有外部生态一次接完”。

| 建议版本 | 主题 | 关键交付 | 验收重点 |
| --- | --- | --- | --- |
| v0.5.0 | 稳定化与基线确认 | 回归个人主页同步、风格应用、管理员登录、pgAdmin、迁移、前后端构建 | 现有能力不回归，文档不夸大 |
| v0.5.1 | 管理后台集成中心骨架 | 统一展示 AI、插件、个人主页、风格包、数据库状态和后续集成入口 | 管理员能判断系统当前状态 |
| v0.5.2 | 个人主页与风格包运营化 | 风格快照、回滚、恢复默认、同步状态、管理员禁用和审核入口 | 用户可恢复，管理员可处理问题 |
| v0.5.3 | 插件包治理增强 | checksum、安装预检、权限预览、版本策略和回滚说明 | 插件来源和风险可追踪 |
| v0.5.4 | Webhook 基础闭环 | endpoint、事件过滤、HMAC 签名、异步投递、失败记录和重试 | mock endpoint 可验证投递与失败处理 |
| v0.5.5 | MCP Server 只读实验闭环 | 只读工具、服务 token、RBAC、审计日志和后台开关 | 未授权拒绝，写操作默认关闭 |
| v0.5.6 | Message 协议与本地适配器 | Message 模型、Adapter 接口、账号绑定模型、mock/local adapter | 不绑定具体 IM 平台也能验证协议 |
| v0.5.7 | 观测、备份与运行文档收口 | 结构化日志、关键 metrics、备份恢复说明、Docker Compose 运行说明、Ubuntu/WSL2 验证 | 外部集成问题可定位，数据可备份；不承诺完整 Docker 产品化封装 |

### 6.4 v0.6-dev 或专项任务

以下内容建议在 v5 稳定后再进入 v0.6-dev 或独立专项：

| 方向 | 建议阶段 | 说明 |
| --- | --- | --- |
| Discord/NapCat/AstrBot 等真实适配器 | v0.6-dev 或 IM 专项 | v5 先完成 Message 协议和本地适配器，真实平台通过 Adapter/Plugin 逐个接入 |
| 插件市场 | v0.6-dev 或 v1.0 前专项 | 依赖插件签名、审核、版本、权限、回滚和运营流程 |
| AI 内容审核 | 暂时搁置，后续审核专项 | 需要稳定 AI Gateway、审核策略、样本评估和人工复核 |
| 更强的个人主页编辑器 | v0.6-dev | 可视化组件、布局编辑器和模板市场应建立在风格包安全模型之后 |

| 完整 Docker 产品化封装 | 暂时搁置，后续部署专项 | 等系统模块稳定后再做完整镜像、备份恢复演练、监控告警和升级回滚流程 |
| Windows 原生验证 | 暂时搁置，后续兼容性专项 | 继续优先 Ubuntu/WSL2，原生 Windows + Docker Desktop 等系统成熟后单独记录 |

## 7. 推荐执行策略

后续开发建议按以下顺序推进：

1. 先完成 v0.5.0 稳定化回归，确保用户侧和管理员侧已暴露的问题不再反复出现。
2. 再建设管理后台集成中心骨架，让管理员先有统一查看状态和进入操作的入口。
3. 再补个人主页与风格包运营化，优先做回滚、恢复默认、同步状态和管理员禁用。
4. 再增强插件包治理，解决来源、权限、版本和回滚的可追踪问题。
5. 再做 Webhook，因为它确定性强，并且是外部自动化和后续 IM 同步的基础。
6. 再做 MCP 只读实验闭环，所有写操作默认关闭，必须经过显式权限和审计。
7. 最后做 Message 协议和本地适配器，把 Discord、NapCat、AstrBot 等真实平台接入留给 v6 或专项。
8. 阶段末补齐观测、备份、Docker Compose 运行说明和 Ubuntu/WSL2 验证文档；完整 Docker 产品化封装暂不作为 v5 目标。

这个顺序的核心判断是：v5 应先把已有用户可见能力做成可运营、可恢复、可排错，再逐步开放外部集成。当前不建议把 MCP、Webhook、真实 IM 适配器、插件市场、完整 Docker 产品化封装、原生 Windows 和 AI 审核同时压入 v5 主线。

## 8. 总结

v4 的实际价值主要体现在三个方面：

| 价值 | 说明 |
| --- | --- |
| 平台能力前进 | AI Gateway、插件交付、个人主页和风格包让 CampusOS 从论坛内核进入可扩展平台形态 |
