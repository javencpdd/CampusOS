# CampusOS 项目计划 v4

> 文档状态：规划草案
> 编写日期：2026-06-30
> 适用范围：CampusOS 项目计划第四版、v0.4-dev 阶段开发安排
> 参考基线：项目计划 v1/v2/v3、`README.md`、`docs/进度/v0.3-dev`、当前代码结构

## 1. 文档定位

`docs/项目计划v4` 是 CampusOS 的第四版项目规划文档集合。这里的 v4 指“项目计划文档版本”，对应的实现阶段建议命名为 `v0.4-dev`，不是产品语义版本 `v4.0.0`。

v4 计划的核心任务不是继续重复 v3 的插件运行时建设，而是在 v3 已完成的插件安全运行、Host API、SDK/CLI 雏形和工程化基础上，进入下一阶段：

> 让 CampusOS 从“可扩展的社区内核 + 插件运行时”推进到“可接入 AI、可连接外部 IM/机器人生态、可被插件稳定交付和运营的平台”。

## 2. v1 到 v3 更新回顾

### 2.1 v1：确立方向和社区内核

v1 的主要价值是确定 CampusOS 不只是传统论坛，而是一个面向校园场景的 Community OS。

| 方向 | v1 沉淀 |
| --- | --- |
| 产品定位 | 论坛是基础 App，系统核心是身份、权限、内容、事件、插件和开放 API |
| 架构理念 | API First、Plugin First、Event Driven |
| 核心对象 | 用户、账号、版块、主题、回复、角色、权限、插件、事件 |
| 技术路线 | Go 后端、Vue 前端、PostgreSQL、Redis、NATS、Docker Compose |
| 生态目标 | 插件系统、Host API、第三方集成、AI 能力、开发者工具链 |

v1 阶段完成了基础系统方向，但早期规划范围偏大，很多能力仍停留在设计层。

### 2.2 v2：按真实实现重排路线

v2 基于 v0.1.x 的实际实现对路线进行收敛，把过大的 v1 目标拆成更可执行的阶段。

| 领域 | v2 完成或明确的内容 |
| --- | --- |
| 基础后端 | 用户注册、登录、JWT、用户列表、主题、回复、版块和事件历史 |
| 前端 | 用户前台 `web/` 建立，后续管理后台 `admin/` 独立拆分 |
| 事件总线 | NATS + 内存回退思路形成，业务行为开始事件化 |
| 权限 | RBAC 从早期计划进入实际数据结构和接口设计 |
| 插件 | gRPC 插件、Host API、插件元数据、API Key、插件仓储开始成型 |
| 工程化 | Docker Compose、Makefile、迁移脚本、进度文档逐步完善 |

v2 的重点是让项目从“能跑”变成“能管理、能扩展、能继续迭代”。

### 2.3 v3：插件运行时和工程稳定化

v3 主要对应 `v0.3-dev` 阶段。这个阶段先处理前置稳定化，再推进插件安全运行闭环。

| 领域 | v3 已完成 |
| --- | --- |
| 基础稳定化 | 修复注册 ID、发帖认证、默认版块、迁移版本管理、README 和帮助文档 |
| 数据库迁移 | `make migrate-up/down/reset/status` 通过脚本扫描 migration，并用 `schema_migrations` 记录版本 |
| 插件 Runtime | Wasm Runtime 接入 wazero，支持生命周期、事件分发、超时、trap 隔离和 payload ABI |
| Host API | 扩展 `GetUser`、`GetThread`、`GetReply`、`QueryThreads`、`PublishEvent`、`SendNotification`、`GetConfig`、`SetConfig`、`CheckPermission`、`Log`、Storage API |
| 权限与日志 | Host API 增加权限检查，插件运行和调用日志落入 `plugin_logs` |
| 示例插件 | `examples/plugins/hello-wasm` 可验证 Wasm 加载、事件处理和日志 |
| 管理后台 | 插件管理页增加插件日志查看入口 |
| SDK | `sdk/go` 提供事件、manifest 类型和 Host API HTTP Client |
| CLI | `cmd/campusosctl` 支持 `plugin init` 和 `plugin inspect` |
| 打包规范 | 固定插件目录、manifest、wasm/grpc 产物和发布包结构 |
| CI/CD | CI 覆盖后端测试、前端构建、Windows Go 测试；CD 可构建发布包并 SSH 部署 |
| 跨平台 | Ubuntu/WSL2 路线明确，原生 Windows 实机验证仍需后续完成 |

v3 的成果是插件平台进入可用雏形，但还有几个不应带入 v4 主线太久的技术债。

## 3. 当前基线判断

截至 v4 规划时，CampusOS 已具备以下基础：

| 模块 | 当前判断 |
| --- | --- |
| 社区核心 | 注册、登录、发帖、回复、版块、后台管理具备基础可用性 |
| 权限系统 | RBAC 基础存在，可继续用于后台、插件和 Host API 权限控制 |
| 插件系统 | gRPC/Wasm 双运行时基础具备，Wasm 已能安全接收事件 |
| Host API | 已有核心方法，但配置持久化、审计细节和存储持久化仍需补强 |
| SDK/CLI | 已有雏形，可继续扩展为插件开发闭环 |
| 前后端工程 | `web/`、`admin/`、Go 后端、Docker 服务和 GitHub Actions 已形成基本工作流 |
| 部署方式 | 本地依赖主要由 Docker 提供，Go 和前端工具链在本机运行 |
| Windows | 有 PowerShell 脚本和 Windows Go CI，但没有完成真实 Windows + Docker Desktop 实机验证 |

因此，v4 不建议一开始就直接做“插件市场”或“Kubernetes 高可用”。更稳妥的路线是先把插件交付闭环、AI 网关、MCP、Webhook 和 IM 适配器做成可验证能力。

## 4. v4 总目标

v4 对应 `v0.4-dev`，建议定位为：

> AI 与外部集成阶段：在稳定插件底座上接入 AI 能力、MCP Server、Webhook 和 IM/机器人适配器，形成可被插件、管理员和外部系统共同使用的开放平台。

### 4.1 核心目标

| 优先级 | 目标 | 说明 |
| --- | --- | --- |
| P0 | v3 技术债收尾 | 配置持久化、SQLite-backed KV、插件 pack/install、SDK 测试 harness |
| P0 | AI Gateway 最小闭环 | 统一模型调用接口、Provider 配置、密钥管理、调用日志、限流 |
| P0 | MCP Server 最小闭环 | 将安全的核心 API 暴露为 MCP 工具，受 RBAC 和审计控制 |
| P1 | 内容审核插件 | 基于 AI Gateway 的帖子/回复审核插件，验证事件、Host API 和 AI 能力 |
| P1 | Webhook 系统 | 将事件可靠转发给外部系统，支持签名、重试和失败记录 |
| P1 | IM/机器人集成 | 先完成统一 Message 协议，再接入 NapCat/AstrBot 中至少一个最小适配器 |
| P2 | 管理后台运营视图 | AI 配置、Webhook、集成状态、插件安装和调用日志的后台入口 |
| P2 | 发布与部署增强 | 更完整的 Docker 发布包、备份恢复、跨平台验证文档 |

### 4.2 设计原则

| 原则 | 要求 |
| --- | --- |
| 插件优先 | AI 审核、IM 同步、Webhook 等尽量以插件或可替换模块实现，不把核心写成不可扩展业务 |
| 安全优先 | AI、MCP、Webhook、IM 都必须经过权限、审计、限流和密钥保护 |
| 事件驱动 | 外部集成优先消费和发布标准事件，减少直接耦合业务表 |
| 可替换 Provider | AI Provider、IM Adapter、Webhook Sink 都应通过接口隔离具体实现 |
| 最小可运行 | 每个阶段必须可启动、可测试、可回滚，避免同时铺开过多抽象 |
| Ubuntu 优先，Windows 跟进 | Ubuntu/WSL2 作为主开发路径，原生 Windows 继续通过脚本和实机验证推进 |

## 5. v4 前置收尾清单

v4 正式进入 AI/集成开发前，建议先完成以下 v3 遗留项，避免新能力建在不稳定接口上。

| 任务 | 优先级 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| `SetConfig` 持久化 | P0 | 插件配置写入数据库或插件仓储 | 重启服务后配置仍存在，并有审计记录 |
| SQLite-backed 插件 KV | P0 | 将 Storage API 底层从内存实现升级为 SQLite-backed KV | 插件 `StorageSet/Get/Delete` 重启后仍可用，路径隔离测试通过 |
| `campusosctl plugin pack` | P0 | 根据打包规范生成 `.campusos-plugin.tar.gz` | `hello-wasm` 可被 CLI 打包并通过 inspect 校验 |
| `campusosctl plugin install` 最小版 | P1 | 从本地包安装到插件目录或仓储 | 安装后可启用、停用、查看 manifest |
| SDK 测试 harness | P1 | 本地模拟 Host API 和事件的测试工具 | 示例插件不依赖完整服务也能跑核心逻辑测试 |
| Windows 实机验证 | 本轮跳过 | 后续独立验证记录 | 不作为 v4 前置收尾门槛；不得写成已实机通过 |
| README/帮助文档同步 | P1 | 更新 v4 相关帮助文档 | 文档与实际命令一致 |

## 6. v4 任务包

### 6.1 AI Gateway

目标：提供统一大模型调用入口，让审核、摘要、问答、机器人和 MCP 能力不绑定单一模型厂商。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| Provider 接口 | P0 | 定义 `ChatCompletion`、`Embedding` 可选接口和错误模型 |
| OpenAI-compatible Provider | P0 | 先支持兼容 OpenAI API 形态的 provider，方便接入多种服务 |
| Provider 配置 | P0 | 支持 base URL、model、timeout、并发、限流等配置 |
| 密钥管理 | P0 | API Key 不进入日志，不直接暴露给前端，后续可接入加密存储 |
| 调用日志 | P0 | 记录 provider、model、耗时、状态、token 用量估算、调用来源 |
| 限流与熔断 | P1 | 防止插件或外部请求无限制消耗模型额度 |
| 管理后台配置页 | P1 | 管理员可配置 provider，查看调用状态和错误 |
| AI Gateway SDK 封装 | P2 | 插件通过 Host API 或 SDK 调用 AI 能力 |

建议先只做文本对话能力，不在 v4 第一轮引入复杂 RAG、向量库和多模态。

### 6.2 AI 内容审核插件

目标：用一个真实插件验证 v4 的 AI Gateway、事件订阅、Host API、权限、日志和后台可观测性。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| 订阅事件 | P0 | 订阅 `thread.created`、`post.created` |
| 读取内容 | P0 | 使用 Host API 读取主题或回复详情 |
| 调用 AI Gateway | P0 | 根据审核 prompt 输出审核结论和理由 |
| 发布审核事件 | P0 | 发布 `moderation.flagged` 或 `moderation.passed` |
| 写入日志 | P0 | 记录审核耗时、模型、结果和错误 |
| 后台审核入口 | P1 | 管理员可查看被标记内容并手动处理 |
| 策略配置 | P1 | 管理员可配置敏感类别、阈值、是否自动隐藏 |

验收时必须明确：AI 审核只能辅助决策，默认不应直接永久删除内容。

### 6.3 MCP Server

目标：让 AI Agent 可以通过受控工具访问 CampusOS，而不是直接绕过业务和权限。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| MCP Server 骨架 | P0 | 新增可独立启停的 MCP 服务模块 |
| 工具白名单 | P0 | 首批只暴露安全工具，如健康检查、查询帖子、查询版块、创建草稿 |
| 鉴权模型 | P0 | MCP 调用必须绑定用户或服务账号，继承 RBAC 权限 |
| 审计日志 | P0 | 记录工具名、调用者、参数摘要、结果状态和 trace id |
| 写操作保护 | P1 | 发帖、审核、封禁等写操作必须有明确权限和可追踪审计 |
| 管理后台开关 | P1 | 管理员可启停 MCP Server、查看工具列表和调用日志 |
| 开发文档 | P1 | 提供 AI Agent 连接示例和安全说明 |

v4 不建议让 MCP 一开始拥有管理员级全量操作能力。先从只读和低风险写操作开始。

### 6.4 Webhook 系统

目标：让外部系统可以订阅 CampusOS 事件，形成标准的事件外发能力。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| Webhook Endpoint 模型 | P0 | 保存 URL、订阅事件、secret、状态、失败次数 |
| 事件过滤 | P0 | 按 event type、source、category 等条件过滤 |
| 签名机制 | P0 | 使用 HMAC 签名请求体，外部系统可验证来源 |
| 重试策略 | P0 | 支持指数退避、最大重试次数、失败记录 |
| Dead Letter | P1 | 多次失败后进入失败队列，管理员可重放 |
| 管理后台 | P1 | 创建、测试、启停、查看最近投递 |
| 文档与示例 | P1 | 提供 curl/mock server 示例 |

Webhook 是 IM、自动化、低代码平台和外部校园系统集成的基础，不应和某个具体平台绑定。

### 6.5 IM/机器人集成

目标：先建立统一 Message 协议，再接入一个最小可用适配器，避免直接把 QQ/机器人协议写入核心业务。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| Message 协议 | P0 | 定义外部消息、发送者、会话、附件、来源平台、关联资源 |
| Adapter 接口 | P0 | 统一 inbound/outbound 消息处理接口 |
| NapCat 最小适配器 | P1 | 验证 QQ 群消息与 CampusOS 事件互通 |
| AstrBot 最小适配器 | P2 | 如 NapCat 已跑通，再接入 AstrBot |
| 命令权限 | P1 | 群命令必须映射到用户或服务账号权限 |
| 消息同步插件 | P1 | 示例：新帖同步到群、群命令查询帖子 |
| 管理后台 | P2 | 显示 Adapter 状态、最近消息、错误日志 |

v4 第一阶段只要求一个适配器最小跑通，不追求覆盖所有 IM 特性。

### 6.6 插件交付闭环

目标：让插件从“源码目录”走向“可打包、可安装、可检查、可回滚”的交付模式。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| `plugin pack` | P0 | 生成标准插件包，排除 `data/`、`.git/`、`node_modules/` |
| `plugin install` | P1 | 从本地插件包安装到指定插件目录 |
| 包校验 | P0 | 校验 manifest、runtime、入口文件、路径穿越和权限声明 |
| 插件版本 | P1 | 同名插件版本升级、降级、回滚策略 |
| 插件配置 schema | P1 | manifest 声明配置项，后台按 schema 渲染表单 |
| 插件签名预研 | P2 | 后续插件市场前置，不在 v4 强制完成 |

### 6.7 管理后台增强

目标：让管理员能实际运营 AI、Webhook、插件、MCP 和外部集成。

| 页面/能力 | 优先级 | 说明 |
| --- | --- | --- |
| AI Provider 配置 | P1 | 配置 provider、model、超时、限流和启停 |
| AI 调用日志 | P1 | 查看调用来源、耗时、错误、token 用量估算 |
| MCP 工具管理 | P1 | 启停工具、查看调用日志 |
| Webhook 管理 | P1 | endpoint 创建、测试、启停、重放失败事件 |
| 插件安装向导 | P1 | 上传或选择本地包，展示权限，确认安装 |
| 集成状态面板 | P2 | 显示 IM Adapter、Webhook、MCP、AI Gateway 健康状态 |

后台页面应优先服务真实操作，不做纯展示页面。

### 6.8 观测、部署与备份

目标：v4 引入外部调用和 AI 成本后，必须补齐基础观测和可恢复能力。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| 结构化日志字段统一 | P0 | trace id、user id、plugin name、event type、request id |
| Metrics 最小集 | P1 | API 延迟、插件调用、AI 调用、Webhook 投递、错误计数 |
| 插件数据备份说明 | P1 | PostgreSQL + 插件 SQLite/data 目录一起备份 |
| 发布包完善 | P1 | 后端二进制、前端 dist、migrations、scripts、示例配置 |
| Docker 封装路线 | P1 | 在 Ubuntu 跑通后，继续推进更完整 Docker 镜像 |
| Windows/WSL2 文档 | P1 | 记录真实验证命令、限制和推荐路径 |

## 7. v4 版本拆分建议

| 版本 | 主题 | 关键交付 |
| --- | --- | --- |
| v0.4.0 | v3 收尾与接口固化 | 配置持久化、SQLite-backed KV、Host API 审计补强、SDK 测试 harness |
| v0.4.1 | 插件交付闭环 | `plugin pack/install`、包校验、插件配置 schema 草案 |
| v0.4.2 | AI Gateway 最小闭环 | Provider 接口、OpenAI-compatible provider、密钥配置、调用日志、限流基础 |
| v0.4.3 | AI 审核插件 | 内容审核插件、审核事件、后台审核入口 |
| v0.4.4 | MCP Server 最小闭环 | MCP 服务、工具白名单、鉴权、审计、只读工具示例 |
| v0.4.5 | Webhook 系统 | endpoint 模型、签名、事件过滤、重试、管理后台 |
| v0.4.6 | IM/机器人最小集成 | Message 协议、Adapter 接口、NapCat 或 AstrBot 最小适配器 |
| v0.4.7 | 管理后台与观测增强 | AI/Webhook/MCP/集成状态页面、metrics 最小集 |
| v0.4.8 | 发布与跨平台稳定化 | Docker 发布包完善、备份恢复文档、Ubuntu/WSL2/Windows 验证总结 |

实际执行时可以合并小版本，但不建议跳过 `v0.4.0` 和 `v0.4.1`。AI 和外部集成会放大插件配置、存储和交付链路的问题，必须先收口。

## 8. 验收标准

v4 完成时，建议按以下标准验收：

| 类别 | 标准 |
| --- | --- |
| 基础功能 | 注册、登录、发帖、回帖、后台登录、插件启停不回归 |
| 插件交付 | `hello-wasm` 可通过 CLI 打包、安装、检查、启用、停用 |
| 配置持久化 | 插件配置通过 Host API 或后台更新后，服务重启仍存在 |
| 插件存储 | 插件 KV 可持久化到隔离 SQLite 存储，路径穿越测试通过 |
| AI Gateway | 至少一个 provider 可配置并完成一次受控调用，调用日志可查 |
| AI 审核 | 新帖或回复触发审核插件，能产出审核事件和插件日志 |
| MCP Server | AI Agent 可调用至少 3 个只读工具，调用受鉴权和审计控制 |
| Webhook | 事件可投递到外部 mock endpoint，签名验证和失败重试可用 |
| IM 集成 | 至少一个适配器完成最小消息同步或命令查询闭环 |
| 后台运营 | 管理员可查看 AI、Webhook、MCP、插件和集成状态 |
| 跨平台 | Ubuntu/WSL2 路径文档完整，原生 Windows 实机验证结果明确记录 |

## 10. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| AI 调用成本失控 | 插件或用户误触发大量调用 | provider 级限流、调用来源记录、管理员开关 |
| MCP 权限过大 | AI Agent 可能执行高风险操作 | 默认只读工具、写操作白名单、RBAC 和审计 |
| Webhook 外部服务不稳定 | 事件投递失败或阻塞 | 异步投递、重试、dead letter、超时控制 |
| IM 协议复杂 | 适配器拖慢主线 | 先统一 Message 协议，只做一个最小适配器 |
| 插件包安全问题 | 路径穿越、伪造 manifest、恶意资源 | 安装前校验、权限确认、后续签名机制 |
| Windows 差异 | 脚本、Docker、psql 和路径问题 | WSL2 推荐路径、原生 Windows 单独验证记录 |
| 过早做插件市场 | 范围膨胀，核心能力不稳 | v4 只做本地包安装和规范，市场放到 v1.0 前后 |

## 11. v4 明确不做

| 不做项 | 原因 |
| --- | --- |
| 完整插件市场 | 依赖插件包、签名、权限、审核、版本和安装链路稳定 |
| Kubernetes/Helm 生产高可用 | 当前更需要先固化 Docker 发布和备份恢复 |
| 多语言 SDK 全量建设 | Go SDK 先作为主路径，TypeScript/Rust SDK 后续再推进 |
| 原生 Windows 作为主开发环境 | 当前仍推荐 Ubuntu/WSL2，原生 Windows 先验证脚本和 Go 测试 |
| AI 自动执行高风险管理操作 | 必须先建立 MCP 鉴权、审计和管理员确认机制 |
| 全量 RAG/向量数据库 | v4 先完成 AI Gateway 和内容审核，RAG 作为后续增强 |

## 12. 建议执行顺序

1. 完成 v3 技术债收尾：配置持久化、SQLite-backed KV、插件打包安装。
2. 固定 AI Gateway 接口和 provider 配置模型。
3. 做 AI 内容审核插件，验证事件、Host API、AI Gateway、日志和后台。
4. 做 MCP Server 最小只读工具，建立鉴权和审计边界。
5. 做 Webhook 系统，形成标准事件外发。
6. 做统一 Message 协议，再接入一个最小 IM/机器人适配器。
7. 补管理后台运营页面、metrics、部署和跨平台文档。
8. 做 v4 总体验收，决定是否进入插件市场或生产部署专项。

## 13. 文档维护要求

v4 阶段建议延续 v3 的做法：

| 要求 | 说明 |
| --- | --- |
| 每个任务有进度文档 | 建议放入 `docs/进度/v0.4-dev`，命名为 `v0.4.X-dev.md` |
| 每个新能力有帮助文档 | AI Gateway、MCP、Webhook、插件打包安装都应进入 `docs/help` |
| README 持续同步 | README 应始终反映当前可运行命令和能力边界 |
| 验证结果写入文档 | 通过、未运行和无法验证的内容都要明确写出 |
| 不夸大 Windows 状态 | 只有真实 Windows + Docker Desktop 跑通过，才写“实机验证通过” |

## 14. 总结

v1 解决“CampusOS 是什么”，v2 解决“基础论坛和后台如何跑起来”，v3 解决“插件如何安全运行并被开发者使用”。v4 应解决“CampusOS 如何连接 AI 和外部生态，并让这些能力可运营、可审计、可交付”。

只要 v4 严格控制范围，先补齐插件交付闭环，再接 AI Gateway、MCP、Webhook 和一个最小 IM 适配器，CampusOS 就能从可扩展的社区内核进入真正的平台化阶段。
