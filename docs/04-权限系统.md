## 设计目标

| 目标 | 描述 |
|:---|:---|
| **细粒度** | 支持资源级、字段级、行级权限控制 |
| **灵活可配** | 权限规则可动态配置，无需修改代码 |
| **高性能** | 权限检查延迟 < 1ms（缓存命中时） |
| **可审计** | 所有权限变更都有审计记录 |
| **可扩展** | 支持自定义策略和权限模型扩展 |

## 权限模型概述

CampusOS 采用 **RBAC + ABAC 混合权限模型**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        权限模型总览                                       │
│                                                                          │
│  ┌───────────────────────┐     ┌───────────────────────────────────────┐ │
│  │      RBAC 层           │     │            ABAC 层                    │ │
│  │  (基于角色的访问控制)    │     │         (基于属性的访问控制)            │ │
│  │                        │     │                                       │ │
│  │  用户 ──→ 角色 ──→ 权限 │     │  权限 = f(主体属性, 资源属性,           │ │
│  │                        │     │          环境属性, 操作属性)            │ │
│  │  粗粒度控制:            │     │                                       │ │
│  │  "谁有什么角色"         │     │  细粒度控制:                            │ │
│  │                        │     │  "在什么条件下可以做什么"                │ │
│  └───────────────────────┘     └───────────────────────────────────────┘ │
│                                                                          │
│  ┌─────────────────────────────────────────────────────────────────────┐ │
│  │                        合并策略                                      │ │
│  │                                                                      │ │
│  │  1. RBAC 提供基础权限集                                               │ │
│  │  2. ABAC 在 RBAC 基础上叠加条件约束                                    │ │
│  │  3. 显式 DENY 优先于 ALLOW                                           │ │
│  │  4. 无匹配规则时默认拒绝                                              │ │
│  └─────────────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────────┘
```

## RBAC 模型设计

### 核心实体关系

```
┌──────────┐       ┌──────────────┐       ┌──────────┐
│   User   │──N:N──│  UserRole    │──N:1──│   Role   │
│          │       │              │       │          │
│ id       │       │ user_id      │       │ id       │
│ username │       │ role_id      │       │ name     │
│ ...      │       │ scope_type   │       │ is_system│
│          │       │ scope_id     │       │ priority │
└──────────┘       │ granted_by   │       └─────┬────┘
                   │ granted_at   │             │
                   │ expires_at   │        N:N  │
                   └──────────────┘             │
                                          ┌─────▼──────┐
                                          │ RolePerm   │
                                          │            │
                                          │ role_id    │
                                          └─────┬──────┘
                                                │
                                           N:1  │
                                          ┌─────▼──────┐
                                          │ Permission │
                                          │            │
                                          │ id         │
                                          │ resource   │
                                          │ action     │
                                          │ scope      │
                                          └────────────┘
```

### 角色分配带作用域

角色分配不是简单的"用户-角色"映射，而是带有**作用域 (Scope)** 的：

```go
// UserRole 用户角色关联（带作用域）
type UserRole struct {
    UserID    snowflake.ID
    RoleID    snowflake.ID
    ScopeType string        // global / category / group
    ScopeID   *snowflake.ID // 具体的版块/群组 ID，global 时为 nil
    GrantedBy snowflake.ID  // 授权人
    GrantedAt time.Time
    ExpiresAt *time.Time    // 可选过期时间
}

// 示例:
// 用户 A 在 "技术分享" 版块是版主
UserRole{UserID: A, RoleID: moderator, ScopeType: "category", ScopeID: &techShareID}

// 用户 B 是全局管理员
UserRole{UserID: B, RoleID: admin, ScopeType: "global", ScopeID: nil}

// 用户 C 临时获得 VIP 权限（30天）
UserRole{UserID: C, RoleID: vip, ScopeType: "global", ExpiresAt: &thirtyDaysLater}
```

## ABAC 扩展设计

### 策略引擎

```go
// Policy 策略实体
type Policy struct {
    ID         snowflake.ID
    Name       string           // 策略名称
    Effect     PolicyEffect     // allow / deny
    Priority   int              // 优先级（数字越大越优先）
    
    // 主体条件（谁）
    Subjects   []SubjectMatcher
    
    // 资源条件（什么资源）
    Resources  []ResourceMatcher
    
    // 操作条件（什么操作）
    Actions    []ActionMatcher
    
    // 环境条件（什么条件下）
    Conditions []Condition
    
    CreatedAt  time.Time
    UpdatedAt  time.Time
    Enabled    bool
}

// SubjectMatcher 主体匹配器
type SubjectMatcher struct {
    Type       string            // user / role / plugin
    Attributes map[string]string // 属性条件
}

// ResourceMatcher 资源匹配器
type ResourceMatcher struct {
    Type       string            // thread / post / user / category
    Attributes map[string]string // 属性条件
}

// Condition ABAC 条件
type Condition struct {
    // 属性路径（支持嵌套引用）
    // "subject.id"           → 主体的 ID
    // "resource.author_id"   → 资源的作者 ID
    // "resource.category_id" → 资源的版块 ID
    // "env.time"             → 环境时间
    // "env.ip"               → 请求 IP
    Attribute string
    
    // 操作符
    // eq / neq / in / not_in / contains / gt / lt / gte / lte
    Operator  string
    
    // 比较值（支持引用其他属性）
    // 固定值: "admin"
    // 属性引用: "subject.id" (以 subject. 开头)
    Value     interface{}
}
```

### 策略评估流程

```
请求到达
    │
    ▼
提取上下文信息:
  subject = { id: "user_123", roles: ["user", "moderator"], ... }
  resource = { type: "thread", id: "thread_456", author_id: "user_789", category_id: "cat_1" }
  action = "delete"
  env = { time: "2024-01-01T12:00:00Z", ip: "10.0.0.1" }
    │
    ▼
┌─────────────────────────────────────────────────────┐
│                 策略评估引擎                           │
│                                                      │
│  1. 从 RBAC 获取基础权限集:                           │
│     → ["thread:read:moderated", "thread:edit:own",   │
│        "thread:delete:moderated", ...]                │
│                                                      │
│  2. 检查请求的权限是否在基础集中:                       │
│     → "thread:delete" 匹配 "thread:delete:moderated" │
│                                                      │
│  3. 评估 ABAC 策略:                                   │
│                                                      │
│     Policy: "版主可删除管辖版块的帖子"                  │
│     Effect: ALLOW                                    │
│     Condition: resource.category_id IN               │
│                subject.moderated_categories           │
│                                                      │
│     → resource.category_id = "cat_1"                 │
│     → subject.moderated_categories = ["cat_1","cat_2"]│
│     → "cat_1" IN ["cat_1", "cat_2"] = true           │
│     → 条件满足，允许操作                               │
│                                                      │
│  4. 合并结果: ALLOW                                   │
└─────────────────────────────────────────────────────┘
    │
    ▼
返回: 允许访问
```

## 权限表达式语法

### 三段式格式

```
{resource}:{action}:{scope}
```

| 部分 | 说明 | 示例 |
|:---|:---|:---|
| `resource` | 资源类型 | `thread`, `post`, `user`, `category`, `plugin` |
| `action` | 操作类型 | `read`, `write`, `delete`, `manage`, `ban` |
| `scope` | 作用范围 | `own`, `moderated`, `global`, `assigned` |

### 预定义资源类型

```
thread          帖子
  ├── thread.title       帖子标题
  ├── thread.content     帖子内容
  ├── thread.status      帖子状态
  └── thread.category    帖子版块

post            回复
  ├── post.content       回复内容
  └── post.status        回复状态

user            用户
  ├── user.profile       用户资料
  ├── user.account       用户账号
  └── user.role          用户角色

category        版块
  ├── category.settings  版块设置
  └── category.moderator 版主管理

plugin          插件
  ├── plugin.install     安装
  ├── plugin.config      配置
  └── plugin.data        数据访问

system          系统
  ├── system.config      系统配置
  ├── system.audit       审计日志
  └── system.backup      备份管理
```

### 预定义操作类型

```
read            读取/查看
create          创建
update          编辑/修改
delete          删除
manage          管理（包含 CRUD）
ban             封禁
pin             置顶/精华
approve         审批
export          导出
import          导入
```

### 通配符规则

```
*:*:*                  匹配所有资源、所有操作、所有范围
thread:*:own           匹配帖子的所有操作，但仅限自己的
*:delete:moderated     匹配管辖范围内所有资源的删除操作
plugin:install:*       匹配所有插件安装操作
```

## 角色体系

### 内置角色

| 角色 | 标识 | 优先级 | 说明 | 是否可删除 |
|:---|:---|:---|:---|:---|
| **超级管理员** | `super_admin` | 1000 | 拥有系统全部权限，不可被限制 | ❌ |
| **管理员** | `admin` | 800 | 管理用户、版块、系统设置 | ❌ |
| **版主** | `moderator` | 600 | 管理特定版块的内容和用户 | ✅ |
| **VIP 用户** | `vip` | 400 | 拥有额外功能权限 | ✅ |
| **普通用户** | `user` | 200 | 基础读写权限 | ❌ |
| **受限用户** | `restricted` | 100 | 被限制的用户，功能受限 | ✅ |
| **匿名用户** | `anonymous` | 0 | 未登录用户的默认角色 | ❌ |

### 各角色默认权限


## 数据权限

### 行级权限控制

数据权限控制用户可以看到哪些数据行：

```go
// DataScope 数据权限范围
type DataScope struct {
    UserID     snowflake.ID
    Resource   string         // 资源类型
    ScopeType  DataScopeType  // 权限范围类型
    ScopeValue interface{}    // 范围值
}

type DataScopeType string

const (
    DataScopeAll       DataScopeType = "all"       // 全部数据
    DataScopeDepartment DataScopeType = "department" // 本部门数据
    DataScopeSelf      DataScopeType = "self"       // 仅自己的数据
    DataScopeCustom    DataScopeType = "custom"     // 自定义数据范围
)
```

### 查询时注入数据权限

```go
// 在查询时自动注入数据权限过滤条件
func (s *ThreadService) ListThreads(ctx context.Context, filter ThreadFilter) ([]*Thread, error) {
    // 获取当前用户
    user := auth.UserFromContext(ctx)
    
    // 构建基础查询
    query := s.db.Model(&Thread{}).Where("deleted_at IS NULL")
    
    // 根据数据权限注入过滤条件
    switch user.DataScope("thread") {
    case DataScopeAll:
        // 不添加额外过滤条件
    case DataScopeSelf:
        query = query.Where("author_id = ?", user.ID)
    case DataScopeDepartment:
        deptIDs := s.getDepartmentIDs(user.ID)
        query = query.Where("department_id IN ?", deptIDs)
    case DataScopeCustom:
        allowedIDs := s.getCustomScopeIDs(user.ID, "thread")
        query = query.Where("id IN ?", allowedIDs)
    }
    
    // 应用用户提供的过滤条件
    query = s.applyFilter(query, filter)
    
    var threads []*Thread
    err := query.Find(&threads).Error
    return threads, err
}
```

## 字段级权限

### 字段权限定义

```go
// FieldPermission 字段级权限
type FieldPermission struct {
    ID         snowflake.ID
    RoleID     snowflake.ID
    Resource   string   // 资源类型
    Field      string   // 字段名
    Access     FieldAccess // visible / hidden / masked
}

type FieldAccess string

const (
    FieldAccessVisible FieldAccess = "visible"  // 完全可见
    FieldAccessHidden  FieldAccess = "hidden"   // 完全隐藏
    FieldAccessMasked  FieldAccess = "masked"   // 脱敏显示
)
```

### 字段脱敏规则

| 字段 | 普通用户 | 版主 | 管理员 |
|:---|:---|:---|:---|
| `user.email` | masked (`z***@example.com`) | visible | visible |
| `user.phone` | masked (`138****1234`) | masked | visible |
| `user.ip_address` | hidden | masked (`10.0.*.*`) | visible |
| `user.status` | hidden | visible | visible |
| `user.created_at` | visible | visible | visible |
| `user.last_login_at` | hidden | visible | visible |

### 字段过滤实现

```go
// 响应过滤器 - 根据权限过滤响应字段
func (r *ResponseFilter) Filter(ctx context.Context, resource string, data interface{}) interface{} {
    user := auth.UserFromContext(ctx)
    fields := r.getFieldPermissions(user, resource)
    
    result := make(map[string]interface{})
    v := reflect.ValueOf(data)
    
    for i := 0; i < v.NumField(); i++ {
        field := v.Type().Field(i)
        fieldName := strings.ToLower(field.Name)
        
        access := fields[fieldName]
        switch access {
        case FieldAccessVisible:
            result[fieldName] = v.Field(i).Interface()
        case FieldAccessMasked:
            result[fieldName] = r.maskValue(fieldName, v.Field(i).Interface())
        case FieldAccessHidden:
            // 不包含在响应中
        }
    }
    
    return result
}
```

## API 权限控制

### 中间件实现

```go
// RequirePermission 权限检查中间件
func RequirePermission(resource, action string, opts ...PermissionOption) gin.HandlerFunc {
    return func(c *gin.Context) {
        // 获取当前用户
        user, err := auth.UserFromContext(c.Request.Context())
        if err != nil {
            c.JSON(http.StatusUnauthorized, response.Error(ErrCodeUnauthorized, "unauthorized"))
            c.Abort()
            return
        }
        
        // 解析选项
        opt := &permissionOptions{}
        for _, o := range opts {
            o(opt)
        }
        
        // 构建权限检查请求
        checkReq := &permission.CheckRequest{
            UserID:    user.ID,
            Resource:  resource,
            Action:    action,
        }
        
        // 如果需要检查资源作用域
        if opt.scopeExtractor != nil {
            scope, err := opt.scopeExtractor(c)
            if err != nil {
                c.JSON(http.StatusBadRequest, response.Error(ErrCodeBadRequest, "invalid scope"))
                c.Abort()
                return
            }
            checkReq.Scope = scope
        }
        
        // 执行权限检查
        result, err := permissionService.Check(c.Request.Context(), checkReq)
        if err != nil {
            c.JSON(http.StatusInternalServerError, response.Error(ErrCodeInternal, "permission check failed"))
            c.Abort()
            return
        }
        
        if !result.Allowed {
            c.JSON(http.StatusForbidden, response.Error(ErrCodeForbidden, result.Reason))
            c.Abort()
            return
        }
        
        // 将权限上下文注入到请求上下文中
        c.Set("permission_context", result.Context)
        c.Next()
    }
}

// 使用示例
router.DELETE("/threads/:id",
    middleware.RequirePermission("thread", "delete",
        middleware.WithScopeExtractor(func(c *gin.Context) (*permission.Scope, error) {
            threadID := c.Param("id")
            // 获取帖子信息以确定作用域
            thread, _ := threadService.GetByID(c, threadID)
            return &permission.Scope{
                Type: "category",
                ID:   thread.CategoryID,
            }, nil
        }),
    ),
    threadHandler.DeleteThread,
)
```

## 插件权限管理

### 插件权限声明

插件在 Manifest 中声明所需权限，安装时由管理员审批：

```yaml
# plugin.yaml
permissions:
  hostAPI:
    - resource: "thread"
      actions: ["read"]
    - resource: "user"
      actions: ["read"]
    - resource: "notification"
      actions: ["send"]
```

### 插件权限审批流程

```
插件安装请求
    │
    ▼
解析 Manifest 中的权限声明
    │
    ▼
展示权限需求列表给管理员
    │
    ▼
管理员审批:
    ├── 全部批准 → 插件获得声明的权限
    ├── 部分批准 → 插件获得批准的权限（可能降级运行）
    └── 拒绝 → 安装失败
    │
    ▼
权限生效，记录到审计日志
```

### 插件权限范围

```go
// PluginPermission 插件权限实体

    GrantedAt  time.Time
```

## 权限缓存策略

### 多级缓存

```
┌──────────────────────────────────────────────────────┐
│                    权限缓存架构                         │
│                                                       │
│  ┌─────────────────────────────────────────────────┐  │
│  │  L1: 进程内缓存 (sync.Map / LRU)                 │  │
│  │  TTL: 5 分钟                                      │  │
│  │  存储: 用户角色、权限列表                           │  │
│  └──────────────────────┬──────────────────────────┘  │
│                         │                             │
│  ┌──────────────────────▼──────────────────────────┐  │
│  │  L2: Redis 缓存                                  │  │
│  │  TTL: 15 分钟                                     │  │
│  │  Key: perm:{user_id}:{resource}:{action}          │  │
│  │  存储: 权限检查结果                                │  │
│  └──────────────────────┬──────────────────────────┘  │
│                         │                             │
│  ┌──────────────────────▼──────────────────────────┐  │
│  │  L3: PostgreSQL                                   │  │
│  │  权限数据的持久化存储                               │  │
│  └─────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

### 缓存失效策略

```
权限变更事件（角色分配、撤销、策略更新）
    │
    ▼
发布 permission.changed 事件
    │
    ▼
┌──────────────────────────────────────────┐
│  所有实例收到事件后:                        │
│                                           │
│  1. 清除本地 L1 缓存中该用户的权限数据      │
│  2. 清除 Redis L2 缓存中该用户的权限数据    │
│  3. 下次请求时重新从数据库加载并缓存        │
└──────────────────────────────────────────┘
```

## 权限审计

### 记录的权限事件

| 事件 | 记录内容 | 触发条件 |
|:---|:---|:---|
| `permission.role_created` | 角色名称、权限列表 | 创建新角色 |
| `permission.role_assigned` | 用户ID、角色ID、作用域、授权人 | 分配角色 |
| `permission.role_revoked` | 用户ID、角色ID、撤销人 | 撤销角色 |
| `permission.policy_created` | 策略详情 | 创建 ABAC 策略 |
| `permission.policy_updated` | 变更前后快照 | 更新策略 |
| `permission.check_denied` | 用户ID、资源、操作、原因 | 权限检查失败 |
| `permission.grant_expired` | 用户ID、角色ID、过期时间 | 临时权限过期 |
