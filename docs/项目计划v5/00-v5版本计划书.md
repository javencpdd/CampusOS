# CampusOS 项目计划 v5

> 文档状态：规划优化版
> 编写日期：2026-07-02
> 适用范围：CampusOS 项目计划第五版、`v0.5-dev` 阶段开发安排
> 参考基线：项目计划 v1/v2/v3/v4、`docs/进度/v0.4-dev`、`docs/项目计划v4/02-v4实现状态与后续规划总结.md`、当前代码结构和近期 UI 测试反馈

## 1. 文档定位

`docs/项目计划v5` 用于记录 CampusOS 第五版项目规划。这里的 v5 指“项目计划文档版本”，建议对应实现阶段命名为 `v0.5-dev`，不是产品语义版本 `v5.0.0`。

v5 不应简单理解为“把 v4 未完成的所有外部集成全部搬过来”。v4 已经完成 AI Gateway、插件包最小交付、个人主页和风格包 P0 闭环，但也暴露出数据库、迁移、后台、登录、风格应用和文档边界等稳定性问题。因此 v5 的首要目标是：

> 把 v4 已经做出来的能力变成可运营、可恢复、可排错的系统基线，再以低风险顺序推进 Webhook、MCP 只读工具和 Message 协议。

v5 建议定位为：

> 运营化与低风险开放集成阶段：先稳定现有用户侧和管理员侧能力，再建设管理后台集成中心、个人主页/风格包运营能力、插件包治理、Webhook 基础闭环、MCP 只读实验闭环和 Message 协议基础。

## 2. 规划优化说明

原 v4 计划里 MCP、Webhook、IM/机器人、插件市场前置能力、观测备份、跨平台验证都在同一阶段出现。按当前代码和测试反馈看，如果 v5 继续同时推进这些方向，会带来三个问题：

| 问题 | 影响 |
| --- | --- |
| 外部集成过多 | MCP、Webhook、IM 都涉及权限、审计、外部依赖和后台运营，容易互相阻塞 |
| 用户侧能力未运营化 | 个人主页和风格包已经可用，但缺少回滚、恢复默认、管理员禁用和同步状态查看 |
| 管理后台入口分散 | 插件、AI、个人主页、风格包和未来集成能力缺少统一状态页，排障成本高 |

因此 v5 做如下优化：

| 调整项 | 优化后安排 |
| --- | --- |
| Webhook | 保留为 v5 主线，因为它确定性强，是外部自动化和后续 IM 同步基础 |
| MCP Server | 保留为 v5 受控实验，只做只读工具、鉴权、审计和后台开关 |
| IM/机器人 | v5 先做 Message 协议和 mock/local adapter，真实 NapCat/AstrBot 适配器后置到 v6 或专项 |
| 个人主页和风格包 | 从“功能可用”推进到“可回滚、可审核、可恢复、可运营” |
| 插件交付 | 从“能导入导出”推进到“来源可追踪、安装可预检、风险可展示” |
| 完整 Docker 产品化封装 | 暂时搁置；v5 只维护开发期 Docker Compose、数据库容器、pgAdmin 和运行文档 |
| 原生 Windows 实机兼容 | 暂时搁置；继续推荐 Ubuntu/WSL2，等系统成熟后再做兼容专项 |
| AI 内容审核 | 暂时搁置；不纳入 v5 主线，等待 AI Gateway、审核策略、人工复核和样本体系成熟 |

| 阶段 | 已形成的核心价值 | 对 v5 的影响 |
| --- | --- | --- |
| v1 | 明确 CampusOS 是面向校园场景的 Community OS，不只是论坛 | v5 继续围绕身份、内容、事件、插件和开放接口建设平台能力 |
| v2 | 注册、登录、发帖、回复、版块、后台、事件历史和 Docker 基础运行路径成型 | v5 的外部集成必须继续复用既有社区核心和权限模型 |

| 模块 | 当前状态 | v5 处理方式 |

| 主页风格导入导出 | 已完成 P0 闭环 | 增加应用前快照、恢复默认、站内分享和管理员处理入口 |
| 管理后台运营视图 | 部分完成 | 建立集成中心骨架，先统一状态和入口 |
| MCP Server | 未完成 | v5 做只读实验闭环，不开放高风险写操作 |
| Webhook 系统 | 未完成 | v5 做基础闭环 |
| IM/机器人集成 | 未完成 | v5 做 Message 协议和本地适配器，真实平台后置 |
| 观测与备份 | 部分完成 | v5 阶段末补齐最小观测、备份和开发期运行说明 |

## 5. v5 设计原则

| 原则 | 要求 |
| --- | --- |
| 先运营化后扩展 | 先让已有能力可恢复、可禁用、可排错，再扩大外部集成 |
| 后台先行 | 每个新集成都必须有管理员可见的状态、开关、日志或故障入口 |
| 事件驱动 | Webhook、Message、个人主页同步优先消费标准事件，不直接散落读取业务表 |
| 权限可追踪 | MCP、Webhook、Message 命令和插件安装必须记录操作者、权限、参数摘要和结果 |
| 最小闭环 | 每个集成先做本地可验证闭环，再扩展真实外部平台 |
| 默认保守 | MCP 写操作、任意 JS 主页、AI 自动审核和真实 IM 适配器默认不作为首轮能力 |
| Ubuntu/WSL2 优先 | 继续推荐 Ubuntu 或 WSL2 作为主开发路径，原生 Windows 不作为 v5 主门槛 |

### 5.1 IM/机器人协议设计判断

v5 的 IM/机器人方向不应直接复制 Discord 机器人协议。Discord 的协议设计服务于 Discord 自己的 Gateway、Interaction、频道、权限和用户体系，直接作为 CampusOS 内部标准会让后续兼容 QQ/NapCat、AstrBot、Telegram、飞书、企业微信或校园自有系统变得困难。

CampusOS 更适合采用“内部标准消息协议 + 外部平台适配器”的结构：

```text
CampusOS 事件和业务 API
  -> CampusOS Message Protocol
  -> Adapter/Plugin
  -> Discord / NapCat / AstrBot / Telegram / Feishu / Custom System
```

| 层级 | 职责 |
| --- | --- |
| CampusOS Message Protocol | 定义平台无关的消息、会话、发送者、附件、命令、权限、关联资源和时间戳 |
| Adapter 接口 | 统一 inbound、outbound、状态检查、错误处理和账号绑定能力 |
| 平台插件 | 把 Discord、NapCat、AstrBot 等平台协议转换为 CampusOS Message，或把 CampusOS 事件转换为外部平台消息 |
| 管理后台 | 查看适配器状态、最近消息、错误日志、账号绑定和权限配置 |

可以借鉴 Discord 机器人的事件、命令、频道、权限和 Webhook 思路，但 CampusOS 不把 Discord 协议写死为核心协议。后续需要兼容 Discord 时，新增 Discord Adapter；需要兼容 QQ/NapCat 时，新增 NapCat Adapter；需要兼容校园内部系统时，新增自定义 Adapter。

v5 的验收边界是 Message 协议、Adapter 接口、账号绑定模型和 mock/local adapter。真实平台适配器放到 v0.6-dev 或专项处理。

## 6. v5 阶段门槛

v5 每个阶段应先满足对应门槛，再进入下一个阶段。

| 门槛 | 目标 | 通过标准 |
| --- | --- | --- |
| Gate 0 | 基线稳定 | Go 测试、web/admin 构建、迁移状态、登录、发帖、个人主页和风格应用回归通过 |
| Gate 1 | 后台可运营 | 管理员能看到插件、AI、个人主页、风格包和集成状态入口 |
| Gate 2 | 用户能力可恢复 | 风格应用可回滚，个人主页可禁用或恢复默认 |
| Gate 3 | 外部事件可投递 | Webhook 可向 mock endpoint 投递、签名、重试和记录失败 |
| Gate 4 | AI Agent 受控访问 | MCP 只读工具可调用，未授权拒绝，写操作默认关闭 |
| Gate 5 | 消息协议可验证 | Message 协议和 mock/local adapter 可通过本地测试验证 |
| Gate 6 | 运行与排障可执行 | 日志、metrics、备份、Docker Compose 运行说明和帮助文档齐全；完整 Docker 产品化封装不作为 v5 门槛 |

### 7.5 `v0.5.4-dev`：Webhook 系统基础闭环

## 8. v5 版本拆分建议

## 9. 验收标准

v5 完成时建议按以下标准验收：

| 类别 | 标准 |
| --- | --- |
| 基础功能 | 注册、登录、发帖、回帖、后台登录、个人主页和风格切换不回归 |
| 后台集成中心 | 管理员可查看 AI、插件、个人主页、风格包、数据库、Webhook、MCP 和 Message 状态 |
| 个人主页 | 用户发帖可同步到个人主页，管理员可禁用违规主页 |
| 风格包 | 用户可应用、导出、导入、回滚、恢复默认和分享风格包 |
| 插件包 | 导入前可预览权限和版本，安装包有 checksum 或等价追踪信息 |
| Webhook | 事件可投递到 mock endpoint，HMAC 签名可验证，失败重试和失败记录可查 |
| MCP | AI Agent 可调用至少 3 个只读工具，未授权调用被拒绝，审计日志可查 |
| Message | Message 协议和 mock/local adapter 可完成本地收发或命令查询闭环 |
| 观测 | 关键外部调用有日志，Webhook/MCP/AI/插件错误可定位 |
| 部署 | Ubuntu/WSL2 推荐路径和 Docker Compose 依赖服务文档完整，备份恢复文档可执行；完整 Docker 产品化封装不纳入 v5 |
| CI | 后端测试、web 构建、admin 构建、迁移检查继续通过 |

## 12. v5 明确不做

| 不做项 | 原因 |
| --- | --- |
| 完整插件市场 | 仍依赖插件签名、审核、版本、权限、回滚和运营流程稳定 |
| 完整 Docker 产品化封装 | 暂时搁置；v5 只维护开发期 Docker Compose 和运行文档，不做完整应用镜像、一键发布包和升级回滚 |
| Kubernetes/Helm 高可用 | 当前优先级低于核心功能稳定、备份恢复和外部集成闭环 |
| 原生 Windows 实机兼容 | 暂时搁置；继续推荐 Ubuntu/WSL2，等系统成熟后再做 Windows + Docker Desktop 兼容专项 |
| 任意 JavaScript 个人主页代码 | 安全风险高，v5 优先配置化组件、CSS 变量和受控布局 |
| AI 内容审核插件 | 暂时搁置；模型效果和 prompt 稳定性不足，需等待 AI Gateway、人工复核和样本评估成熟 |
| AI 自动高风险管理操作 | MCP 和 AI 只能在明确权限、审计和管理员确认后扩展写操作 |
| 真实 IM 平台强制接入 | v5 只要求 Message 协议和本地适配器，NapCat/AstrBot 后置到 v6 或专项 |
| 完整 RAG/向量数据库 | AI Gateway 已有最小闭环，v5 主线是运营化和低风险集成 |

## 13. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| v4 bugfix 未回归 | v5 新功能建立在不稳定基线上 | `v0.5.0-dev` 先做稳定化、文档同步和 commit 边界确认 |
| 管理后台入口分散 | 管理员无法判断系统真实状态 | `v0.5.1-dev` 先建集成中心骨架 |
| 风格包缺少回滚 | 用户应用错误风格后难以恢复 | 应用前快照、恢复默认和管理员恢复入口作为 P0 |
| 插件包来源不可追踪 | 后续排障和审核困难 | 安装时记录 checksum、版本、权限和操作者 |
| Webhook 外部服务不稳定 | 事件投递失败或拖慢业务 | 异步投递、超时、重试、失败记录和 Dead Letter |
| MCP 权限过大 | AI Agent 可能执行高风险操作 | 默认只读工具、写操作关闭、RBAC、审计和后台开关 |
| IM 协议复杂 | 真实适配器拖慢主线 | v5 先抽象 CampusOS Message 协议和本地适配器；Discord、NapCat、AstrBot 等通过后续 Adapter/Plugin 接入 |
| 完整 Docker、原生 Windows、AI 审核过早启动 | 分散主线并引入大量部署、兼容和模型评估问题 | 暂时搁置，等 v5 核心能力稳定后再立专项 |
| 文档夸大完成度 | 后续开发误判项目状态 | 计划、README、help 和进度文档都区分“已完成、部分完成、未完成” |

## 14. 建议执行顺序

1. 完成 `v0.5.0-dev` 稳定化：回归个人主页同步、风格应用、管理员登录、pgAdmin、迁移和构建。
2. 建立 `v0.5.1-dev` 管理后台集成中心骨架，先统一状态和入口。
3. 完成 `v0.5.2-dev` 个人主页和风格包运营化，优先回滚、恢复默认、禁用和同步状态。
4. 完成 `v0.5.3-dev` 插件包治理，解决 checksum、安装预检、权限预览和版本策略。
5. 完成 `v0.5.4-dev` Webhook 基础闭环，先跑通 mock endpoint 投递、签名和失败记录。
6. 完成 `v0.5.5-dev` MCP 只读实验闭环，严格限制工具范围和权限。
7. 完成 `v0.5.6-dev` Message 协议和本地适配器，不强依赖真实 IM 环境。
8. 完成 `v0.5.7-dev` 观测、备份、Docker Compose 运行说明和 Ubuntu/WSL2 文档。
9. 做 v5 总体验收，决定 v6 是否进入真实 IM 适配器、插件市场、安全沙箱增强、完整 Docker 产品化封装、原生 Windows 或 AI 审核专项。

## 15. 总结

v5 的重点不是继续扩大愿景，而是提高 CampusOS 的可运营性和可恢复性。当前更合理的路线是：

```text
稳定现有能力 -> 建立后台集成中心 -> 运营化个人主页/风格包 -> 治理插件包 -> 做 Webhook -> 受控 MCP -> Message 协议 -> 观测发布收口
```

如果 v5 能完成这些内容，CampusOS 将具备更稳的平台化基础：社区内容是核心，插件和个人主页是扩展面，Webhook/MCP/Message 是对外连接层，后台集成中心是运营面。真实 IM 适配器通过 Adapter/Plugin 逐步接入；完整 Docker 产品化封装、原生 Windows 和 AI 内容审核暂时搁置，等系统成熟后再作为专项推进。
