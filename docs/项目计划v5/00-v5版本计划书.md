# CampusOS 项目计划 v5

> 文档状态：规划草案
> 编写日期：2026-07-02
> 适用范围：CampusOS 项目计划第五版、v0.5-dev 阶段开发安排
> 参考基线：项目计划 v1/v2/v3/v4、`docs/进度/v0.4-dev`、当前代码结构和近期 UI 测试反馈

## 1. 文档定位

`docs/项目计划v5` 用于记录 CampusOS 第五版项目规划。这里的 v5 指“项目计划文档版本”，建议对应实现阶段命名为 `v0.5-dev`，不是产品语义版本 `v5.0.0`。

v5 的核心任务不是重新扩大范围，而是承接 v4 阶段已经明确但尚未落地的开放集成能力，并把 v4 已完成的个人主页、风格包、AI Gateway 和插件交付能力推进到更稳定、可运营、可回滚、可审计的状态。

建议 v5 定位为：

> 开放集成与运营化阶段：补齐 MCP、Webhook、统一 Message/IM 适配器等外部集成闭环，并强化个人主页、风格包、插件包和后台运营能力，让 CampusOS 从“功能可用”进入“平台可运营”。

## 2. v1 到 v4 阶段回顾

| 阶段 | 已形成的核心价值 | 对 v5 的影响 |
| --- | --- | --- |
| v1 | 明确 CampusOS 是面向校园场景的 Community OS，不只是论坛 | v5 继续围绕身份、内容、事件、插件和开放接口建设平台能力 |
| v2 | 注册、登录、发帖、回复、版块、后台、事件历史和 Docker 基础运行路径成型 | v5 的外部集成必须继续复用既有社区核心和权限模型 |
| v3 | 插件 Runtime、Host API、SDK/CLI、CI/CD、迁移管理和工程稳定化完成 | v5 的 MCP、Webhook、IM 和风格生态应建立在插件、事件和 Host API 之上 |
| v4 | AI Gateway、插件包导入导出、个人主页、风格包 P0 闭环完成 | v5 优先补齐 v4 未完成的 MCP、Webhook、IM、回滚分享、运营后台和观测能力 |

截至 v5 规划时，v0.4-dev 更适合定义为阶段性收口，而不是 v4 原计划全量完成。根据 `docs/进度/v0.4-dev/v0.4.12-dev.md`，以下能力已经完成或基本完成：

| 模块 | 当前状态 |
| --- | --- |
| v3 前置收尾 | 已完成 |
| AI Gateway | 已完成最小闭环 |
| 插件交付闭环 | 已完成最小闭环 |
| 个人主页/博客园能力 | 已完成核心闭环 |
| 主页风格导入导出 | 已完成 P0 闭环 |
| 管理后台运营视图 | 部分完成 |
| 发布与跨平台 | 部分完成 |

以下能力仍应作为 v5 的主线继续推进：

| 模块 | 当前状态 | v5 处理方式 |
| --- | --- | --- |
| MCP Server | 未完成 | 作为 v0.5-dev P0 任务，优先只读工具、鉴权和审计 |
| Webhook 系统 | 未完成 | 作为 v0.5-dev P0/P1 任务，先完成签名、过滤、重试和失败记录 |
| IM/机器人集成 | 未完成 | 先做统一 Message 协议，再接入一个最小适配器 |
| 风格包回滚与分享 | 未完成 | 加入应用前快照、恢复默认、站内分享和审核能力 |
| 插件版本回滚与签名 | 未完成 | 先做 checksum、安装预检和回滚，再评估签名 |
| 运营后台 | 部分完成 | 建立集成中心，集中管理 MCP、Webhook、IM、个人主页和风格包 |
| 观测与备份 | 部分完成 | 补齐关键日志、metrics、备份恢复和发布验证文档 |

## 3. v5 总目标

v5 建议以 `v0.5-dev` 为实现阶段，围绕以下目标推进：

| 优先级 | 目标 | 说明 |
| --- | --- | --- |
| P0 | v4 已发现问题稳定化 | 回归个人主页同步、风格应用、pgAdmin、管理员登录、迁移和文档状态 |
| P0 | MCP Server 最小闭环 | 暴露安全工具给 AI Agent，调用必须受鉴权、RBAC 和审计约束 |
| P0 | Webhook 基础闭环 | CampusOS 事件可签名投递到外部系统，并支持失败记录和重试 |
| P0 | 统一 Message 协议 | 为 IM/机器人适配器建立统一消息模型和权限边界 |
| P1 | 一个 IM/机器人最小适配器 | 优先选择 NapCat 或其他当前最容易本地验证的适配器 |
| P1 | 管理后台集成中心 | 管理员可查看和操作 MCP、Webhook、IM、个人主页、风格包、插件交付状态 |
| P1 | 个人主页和风格运营化 | 增加风格回滚、恢复默认、站内分享、管理员禁用和审核入口 |
| P1 | 插件交付增强 | 插件包 checksum、版本策略、安装预检、权限预览和回滚说明 |
| P2 | 观测、备份和发布增强 | 补齐 metrics、备份恢复、Docker 发布包和 Ubuntu/WSL2 验证文档 |

## 4. v5 设计原则

| 原则 | 要求 |
| --- | --- |
| 先稳定后扩展 | v5 第一件事是确认 v4 UI 测试反馈和迁移状态，不在不稳定基础上继续堆功能 |
| 事件驱动 | Webhook、IM 同步、个人主页同步优先消费标准事件，不直接散落读取业务表 |
| 权限可追踪 | MCP、Webhook、IM 命令和插件安装必须能追踪调用者、权限、参数摘要和结果 |
| 最小闭环 | 每个集成先做一个能本地验证的闭环，再扩展高级功能 |
| 插件友好 | 新能力尽量沉淀为 Host API、SDK 或可替换模块，避免写死在单一业务流程 |
| Ubuntu/WSL2 优先 | 继续推荐 Ubuntu 或 WSL2 作为主开发路径，原生 Windows 不作为 v5 主门槛 |

v5 开始前建议先完成一次 v0.4 状态回归，作为 `v0.5.0-dev`。

| 文档状态同步 | P1 | README、help 和 v5 计划说明当前边界 | 不把未完成的 MCP/Webhook/IM 写成已完成 |

## 6. v5 任务包

### 6.1 `v0.5.0-dev`：稳定化与基线确认

目标：把 v4 阶段测试中发现的问题和当前迁移状态先收口，形成 v5 的可开发基线。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| v4 bugfix 回归 | P0 | 覆盖个人主页同步、风格应用、管理员登录和 pgAdmin |
| 数据库状态确认 | P0 | 保证迁移脚本、seed 数据和文档一致 |
| 前后端构建确认 | P0 | web、admin 和 Go 测试均通过 |
| v5 进度目录建立 | P1 | 新建 `docs/进度/v0.5-dev`，后续每个任务记录完成情况 |
| v5 帮助文档索引 | P1 | 在 `docs/help` 增补 v5 相关文档索引或导航说明 |

### 6.2 `v0.5.1-dev`：MCP Server 最小闭环

目标：让 AI Agent 可以通过受控工具访问 CampusOS，而不是绕过业务 API 和权限模型。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| MCP 服务模块 | P0 | 新增 `internal/mcp` 或等价模块，支持独立启停 |
| 工具注册表 | P0 | 首批只暴露低风险工具，如健康检查、版块列表、帖子列表、帖子详情 |
| 鉴权模型 | P0 | 支持服务 token 或绑定用户身份，继承 RBAC |
| 审计日志 | P0 | 记录工具名、调用者、参数摘要、结果状态、耗时和 trace id |
| 写操作边界 | P1 | 如加入创建草稿，必须默认关闭或要求显式权限 |
| 管理后台入口 | P1 | 管理员可查看工具列表、启停状态和调用日志 |
| 使用文档 | P1 | 提供本地连接、token 配置和安全说明 |

首轮验收标准：

| 类别 | 标准 |
| --- | --- |
| 工具数量 | 至少 3 个只读工具可调用 |
| 安全 | 未授权调用被拒绝，写操作默认不可用 |
| 审计 | 每次调用有日志可查 |
| 测试 | 工具注册、鉴权失败、成功调用和审计写入有测试 |

### 6.3 `v0.5.2-dev`：Webhook 系统基础闭环

目标：让 CampusOS 事件可以可靠投递到外部系统。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| Endpoint 数据模型 | P0 | 保存 URL、订阅事件、secret、状态、失败次数和最近错误 |
| 事件过滤 | P0 | 支持按 event type、source、category 等条件过滤 |
| HMAC 签名 | P0 | 使用 secret 对请求体签名，外部系统可验证来源 |
| 异步投递 | P0 | 事件进入投递流程，不阻塞主业务请求 |
| 重试策略 | P0 | 支持超时、最大重试次数、退避和失败记录 |
| Dead Letter | P1 | 多次失败进入失败队列，管理员可查看或重放 |
| mock receiver | P1 | 提供本地测试接收器或 curl 示例 |

首轮验收标准：

| 类别 | 标准 |
| --- | --- |
| 投递 | 新帖或测试事件可投递到 mock endpoint |
| 签名 | mock endpoint 能验证 HMAC |
| 失败处理 | 无效 URL 会记录失败并重试 |
| 管理 | 管理员可启停 endpoint 或查看最近投递记录 |

### 6.4 `v0.5.3-dev`：统一 Message 协议与 IM 适配器

目标：先建立平台无关的消息模型，再接入一个最小 IM/机器人适配器。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |
| Message 协议 | P0 | 定义平台、会话、发送者、内容、附件、关联资源和时间戳 |
| Adapter 接口 | P0 | 统一 inbound、outbound、状态检查和错误处理 |
| 用户绑定模型 | P0 | 支持外部账号和 CampusOS 用户或服务账号绑定 |
| 命令权限 | P0 | 群命令或机器人命令必须映射 RBAC 权限 |
| NapCat 最小适配器 | P1 | 优先验证 QQ 群消息到 CampusOS 查询或通知的闭环 |
| 示例同步插件 | P1 | 示例：新帖同步到群、群命令查询帖子 |
| 管理后台状态 | P2 | 展示适配器状态、最近消息和错误日志 |

首轮验收标准：

| 类别 | 标准 |
| --- | --- |
| 协议 | Message 类型可单元测试并覆盖常见文本消息 |
| 适配器 | 至少一个适配器可以本地完成最小收发或查询闭环 |
| 权限 | 未绑定或无权限用户不能执行敏感命令 |
| 文档 | 明确本地验证前置条件和限制 |

目标：给管理员一个统一入口，查看已经下载、启用、接入和报错的扩展能力。

| 页面/能力 | 优先级 | 说明 |

| 插件仓库增强 | P1 | 查看已安装插件、启停状态、版本、权限和导入导出入口 |
| Webhook 管理 | P1 | 创建、测试、启停、查看投递和重放失败事件 |
| MCP 管理 | P1 | 查看工具列表、启停、调用日志和风险标识 |
| IM 管理 | P1 | 查看适配器状态、账号绑定、最近消息和错误 |

### 6.6 `v0.5.5-dev`：个人主页与风格包运营化

目标：让个人主页和风格包从 P0 可用变成可恢复、可分享、可管理。

| 任务 | 优先级 | 说明 |
| --- | --- | --- |

| 内容同步可视化 | P2 | 用户和管理员可查看最近同步状态和失败原因 |

| 风险 | 要求 |

## 8. 验收标准

v5 完成时建议按以下标准验收：

| 类别 | 标准 |
| --- | --- |
| 基础功能 | 注册、登录、发帖、回帖、后台登录、个人主页和风格切换不回归 |
| MCP | AI Agent 可调用至少 3 个只读工具，未授权调用被拒绝，审计日志可查 |
| Webhook | 事件可投递到 mock endpoint，HMAC 签名可验证，失败重试和失败记录可查 |
| IM | 至少一个适配器完成最小消息同步或命令查询闭环 |
| 后台 | 管理员可查看和处理 MCP、Webhook、IM、插件、个人主页和风格包状态 |
| 个人主页 | 用户发帖可同步到个人主页，管理员可禁用违规主页 |
| 风格包 | 用户可应用、导出、导入、回滚和分享风格包 |
| 插件包 | 导入前可预览权限和版本，安装包有 checksum 或等价追踪信息 |
| 观测 | 关键外部调用有日志，MCP/Webhook/AI/IM 错误可定位 |
| 部署 | Ubuntu/WSL2 推荐路径文档完整，备份恢复文档可执行 |
| CI | 后端测试、web 构建、admin 构建、迁移检查继续通过 |

## 10. 文档维护要求

| 要求 | 说明 |
| --- | --- |
| 进度文档 | v5 任务进度放入 `docs/进度/v0.5-dev`，建议命名为 `v0.5.X-dev.md` |
| 帮助文档 | MCP、Webhook、IM、集成中心、风格回滚、插件校验进入 `docs/help` |
| README 同步 | README 只写已实现能力和真实可运行命令 |
| 验证结果 | 每次任务文档明确写出已运行、未运行和无法验证的命令 |
| 边界说明 | MCP/Webhook/IM 未完成前不得在 README 或计划总结中写成已完成 |
| 提交纪律 | 每个可验收任务单独提交，避免把无关修复和新功能混在一个 commit |

## 11. v5 明确不做

| 不做项 | 原因 |
| --- | --- |
| 完整插件市场 | 仍依赖插件签名、审核、版本、权限、回滚和运营流程稳定 |
| Kubernetes/Helm 高可用 | 当前优先级低于 Docker 发布、备份恢复和外部集成闭环 |
| 原生 Windows 主开发路径 | 继续推荐 Ubuntu/WSL2；原生 Windows 可做验证记录，但不作为 v5 主门槛 |
| 任意 JavaScript 个人主页代码 | 安全风险高，v5 优先配置化组件、CSS 变量和受控布局 |
| AI 自动高风险管理操作 | MCP 和 AI 只能在明确权限、审计和管理员确认后扩展写操作 |
| 全量多平台 IM 覆盖 | v5 只要求一个最小适配器，避免被各平台协议细节拖慢主线 |
| 完整 RAG/向量数据库 | AI Gateway 已有最小闭环，v5 主线是开放集成和运营化 |

## 12. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| v4 bugfix 未提交或未回归 | v5 新功能建立在不稳定基线上 | `v0.5.0-dev` 先做稳定化、文档同步和 commit 边界确认 |
| MCP 权限过大 | AI Agent 可能执行高风险操作 | 默认只读工具、写操作白名单、RBAC、审计和后台开关 |
| Webhook 外部服务不稳定 | 事件投递失败或拖慢业务 | 异步投递、超时、重试、失败记录和 Dead Letter |
| IM 协议复杂 | 适配器实现拖慢核心主线 | 先抽象 Message 协议，只接一个最小可验证适配器 |
| 风格包安全问题 | XSS、恶意外链、布局破坏 | 禁止危险字段、限制外部资源、应用前校验、支持回滚 |
| 插件包来源不可追踪 | 后续排障和审核困难 | 安装时记录 checksum、版本、权限和操作者 |
| 后台入口分散 | 管理员无法判断系统真实状态 | 建立集成中心，统一显示健康状态、日志和操作入口 |
| 文档夸大完成度 | 后续开发误判项目状态 | 计划、README、help 和进度文档都区分“已完成、部分完成、未完成” |

## 13. 建议执行顺序

1. 完成 `v0.5.0-dev` 稳定化：回归个人主页同步、风格应用、管理员登录、pgAdmin、迁移和构建。
2. 建立 `docs/进度/v0.5-dev`，明确每个任务的验收命令和提交边界。
3. 完成 MCP Server 最小只读工具，先把鉴权和审计边界建立起来。
4. 完成 Webhook endpoint、签名、过滤、投递和失败记录。
5. 完成统一 Message 协议，再接入一个最小 IM/机器人适配器。
6. 建立管理后台集成中心，避免外部集成能力只有 API 没有运营入口。
7. 增强个人主页和风格包的回滚、分享、审核和禁用能力。
8. 增强插件包 checksum、安装预检、版本策略和回滚说明。
9. 补齐 metrics、备份恢复、Docker 发布包和 Ubuntu/WSL2 验证文档。
10. 做 v5 总体验收，决定 v6 是否进入插件市场、安全沙箱增强或生产部署专项。

## 14. 总结

v5 不建议继续把所有方向都压进 v4 尾部，而应把 v4 已完成的能力作为基线，把未完成的 MCP、Webhook、IM 和运营后台拆成清晰的 v0.5-dev 任务。

如果 v5 能完成 MCP、Webhook、统一 Message/IM、集成中心、风格回滚分享和插件交付增强，CampusOS 将具备更清晰的平台化形态：社区内容是核心，插件和个人主页是扩展面，MCP/Webhook/IM 是对外连接层，后台集成中心是运营面。
