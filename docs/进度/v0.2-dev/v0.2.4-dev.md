## 核心功能详解

### 1. 角色管理 API（后端）

新增 4 个 RESTful API 端点，均需要 `role:manage` 权限：

| 端点 | 方法 | 说明 |
|:---|:---|:---|
| `/api/v1/roles` | GET | 列出所有角色（admin/moderator/member/guest） |
| `/api/v1/users/:id/roles` | GET | 获取指定用户的角色列表 |
| `/api/v1/users/:id/roles` | POST | 给用户分配角色（body: `{ "role_id": 1 }`） |
| `/api/v1/users/:id/roles` | DELETE | 撤销用户角色（body: `{ "role_id": 1 }`） |

**内置角色（4 个）：**

| 角色 | ID | 说明 |
|:---|:---|:---|
| admin | 1 | 系统管理员，拥有全部权限 |
| moderator | 2 | 版主，可管理帖子和用户 |
| member | 3 | 普通会员，基础读写权限 |
| guest | 4 | 访客，仅只读权限 |

### 2. 管理后台用户管理增强（前端）

**用户列表新增角色列：**
- 每个用户的角色以彩色标签显示（admin 红色、moderator 橙色、member 默认、guest 灰色）
- 未分配角色的用户默认显示 "member" 标签

**角色管理对话框：**
- 点击用户行的"角色"按钮打开对话框
- 显示当前用户名和昵称
- 当前角色以可关闭标签显示，点击 × 即可撤销
- 下拉选择器列出所有可分配角色，已分配的角色自动禁用

### 3. 路由守卫增强

- 读取 `admin_user` 中的 `role` 字段进行前端角色验证
- 预留了 admin/moderator/super_admin 角色检查逻辑
- 当前后端角色信息未写入 JWT Token，前端仅做预留
- 真正的权限校验由后端 `RequirePermission` 中间件执行
