# CampusOS v0.3-dev 开发指南

> 日期：2026-06-28
> 目标：为 v0.3-dev 插件生态开发提供工程入口、边界和验收口径

## 1. 开发目标

v0.3-dev 的目标不是继续堆社区功能，而是把插件系统从“可以接入”推进到“可以安全运行、可以被开发者开发、可以被管理员管理”。

核心交付顺序：

1. 统一 Runtime 生命周期接口。
2. 实现 Wasm Runtime 最小闭环。
3. 固定第一版 Host API 契约。
4. 引入插件权限校验和调用日志。
5. 提供 hello-wasm 示例插件。
6. 再推进 SDK、CLI、后台管理和跨平台验证。

## 2. 推荐开发顺序

### 2.1 Runtime 抽象收敛

当前接口位于 `internal/plugin/runtime.go`：

```go
type Runtime interface {
    Start(ctx context.Context, plugin *Plugin) error
    Stop(ctx context.Context, pluginName string) error
    SendEvent(ctx context.Context, pluginName string, event *EventMessage) (*PluginResponse, error)
    HealthCheck(ctx context.Context, pluginName string) error
    IsRunning(pluginName string) bool
    Type() string
}
```

第一轮不要大改 Manager 行为。优先保持 gRPC Runtime 可用，再新增 Wasm Runtime 实现同一接口。

验收标准：

| 项 | 标准 |
| --- | --- |
| gRPC Runtime | 现有 hello-plugin 不回归 |
| Wasm Runtime | 可以注册 `runtime: wasm` |
| Manager | `RegisterRuntime("wasm", runtime)` 后能走现有启停流程 |
| 测试 | Runtime 关键路径有单元测试或最小集成测试 |

### 2.2 Wasm Runtime 最小闭环

建议优先评估 `wazero`。第一轮只实现最小可运行闭环：

| 能力 | 第一轮要求 |
| --- | --- |
| 加载 | 从插件目录读取 wasm 文件 |
| 启动 | 实例化 wasm module |
| 事件 | `SendEvent` 能把事件传给插件入口 |
| 超时 | 每次调用必须有 context deadline |
| 错误隔离 | wasm trap 或错误不能拖垮主服务 |
| 日志 | 调用耗时和错误写入日志 |

暂不要求：

- 完整 WASI 文件系统权限。
- 完整 SDK。
- 多语言插件模板。
- 插件市场。

### 2.3 Host API 契约固定

当前 Host API 位于 `internal/plugin/hostapi`。第一版建议先固定以下能力：

| Host API | 优先级 | 说明 |
| --- | --- | --- |
| `GetUser` | P0 | 查询用户基础资料 |
| `GetThread` / `QueryThreads` | P0 | 查询主题或主题列表 |
| `PublishEvent` | P0 | 插件发布标准事件 |
| `Log` | P0 | 插件写入结构化日志 |
| `GetConfig` | P0 | 读取插件配置 |
| `CheckPermission` | P0 | 运行时权限判断 |
| `SendNotification` | P1 | 站内通知 |
| `SetConfig` | P1 | 写入插件配置 |

要求：

- API 输入输出结构要稳定。
- ID 类型对外使用 string。
- 所有敏感 Host API 必须经过插件权限校验。
- 错误码和错误消息要可被 SDK 处理。

### 2.4 插件权限模型

第一轮权限模型以 Manifest 声明为准。已有结构：

```yaml
permissions:
  api:
    - resource: "thread"
      actions: ["read"]
    - resource: "user"
      actions: ["read"]
```

建议扩展为三类检查：

| 层级 | 示例 | 第一轮是否做 |
| --- | --- | --- |
| API 级 | 插件是否能调用 `GetUser` | 是 |
| 资源级 | 插件是否能读取 `thread` | 是 |
| 字段级 | 是否能读取用户 email | 后续 |

权限失败时要：

1. 拒绝调用。
2. 返回明确错误。
3. 写入插件日志。

### 2.5 插件日志

`000005_plugin_schema_alignment` 已创建 `plugin_logs`。v0.3-dev 需要把运行时调用写入该表。

建议日志字段：

| 字段 | 用途 |
| --- | --- |
| `plugin_name` | 插件名 |
| `level` | info/warn/error |
| `message` | 日志正文 |
| `event_type` | 事件类型 |
| `trace_id` | 请求或事件 trace |
| `metadata` | 结构化信息 |

第一轮验收：

- 插件启动失败有日志。
- Host API 权限拒绝有日志。
- 事件处理失败有日志。

## 3. 目录建议

建议新增目录时保持现有边界：

```text
internal/plugin/
├── wasm/                 # Wasm Runtime 实现
├── hostapi/              # Host API 契约与权限检查
├── manager.go            # 生命周期编排
├── manifest.go           # Manifest 解析与校验
└── runtime.go            # Runtime 接口

examples/plugins/
└── hello-wasm/           # 最小 Wasm 示例
```

不要把 Wasm Runtime 逻辑塞进 `manager.go`。Manager 应只做生命周期编排和 Runtime 分发。

## 4. 测试要求

每次开发任务至少运行：

```bash
go test ./...
cd web && pnpm build
cd admin && pnpm build
```

涉及数据库迁移时再运行：

```bash
make migrate-up
make migrate-status
```

涉及 GitHub Actions 时再检查：

```bash
python3 - <<'PY'
from pathlib import Path
import yaml
for p in Path('.github/workflows').glob('*.yml'):
    yaml.safe_load(p.read_text())
    print(p, 'ok')
PY
```

## 5. 不建议第一轮处理的内容

| 内容 | 原因 |
| --- | --- |
| 插件市场 | 依赖权限、日志、Runtime 稳定后再做 |
| 完整 CLI | 先明确插件打包和 Runtime 输入输出 |
| 原生 Windows 生产部署 | 当前建议通过 Docker Desktop/WSL2 验证 |
| 管理后台大改 | 先保证后端能力真实存在 |
| UUID 主键迁移 | 当前 ID 策略已收敛到应用侧 BIGINT |

## 6. 完成定义

v0.3-dev 第一轮完成时，至少应满足：

1. gRPC hello-plugin 仍可加载。
2. hello-wasm 能加载、启动、接收事件并写日志。
3. 未授权 Host API 调用会被拒绝。
4. 插件错误不会导致主服务崩溃。
5. `go test ./...`、两个前端构建、迁移状态检查通过。
6. 文档同步更新到 `docs/进度/v0.3-dev`。
