# CampusOS v0.3.11-dev 进度说明

> 日期：2026-06-30
> 状态：已完成

## 1. 本次目标

完成 v0.3-dev 剩余任务中的插件 SQLite 存储设计，明确插件独立存储的路径、隔离、备份和后续最小实现边界。

## 2. 设计结论

| 项目 | 约定 |
| --- | --- |
| 适用范围 | `manifest.storage.type: sqlite` 的插件 |
| 默认目录 | `<plugin_dir>/data/` |
| 默认文件名 | `plugin.db` |
| 显式文件名 | 使用 `manifest.storage.sqlite.filename`，例如 `data.db` |
| 最终路径 | `<plugin_dir>/data/<filename>`；若 filename 已包含相对目录，则必须仍留在 `<plugin_dir>/data` 内 |
| 禁止路径 | 绝对路径、`..` 穿越路径、空文件名、跨插件目录路径 |
| 所有权 | 每个插件只能访问自己的 SQLite 文件 |
| 权限声明 | 读写 SQLite 需要声明 `storage/read`、`storage/write` 或通配权限 |
| 备份单位 | 单插件备份以 `<plugin_dir>/data/` 为单位 |
| 删除策略 | 卸载插件默认保留 `data/`，提供后续 `purge` 显式清理 |

## 3. Manifest 示例

```yaml
storage:
  type: sqlite
  sqlite:
    filename: "plugin.db"

permissions:
  api:
    - resource: "storage"
      actions: ["read", "write", "delete"]
```

如果插件不需要持久化，应显式声明：

```yaml
storage:
  type: none
```

## 4. 隔离策略

| 风险 | 策略 |
| --- | --- |
| 插件访问其他插件数据 | Host API 按调用插件 manifest 解析身份，存储请求必须匹配自身 plugin name |
| 路径穿越 | 后续实现 `ResolveSQLitePath(plugin)` 时必须使用 `filepath.Clean` 并校验前缀 |
| 外部暴露 | SQLite 文件不应由 HTTP 静态目录直接暴露 |
| 并发写入 | v0.3-dev 先采用单进程 Host API 串行化或连接池限制；后续再扩展 WAL 策略 |
| 损坏恢复 | 备份前先停止插件或进入只读窗口，避免热备份产生不一致 |

## 5. 后续最小实现接口

建议后续在 `internal/plugin/storage` 中增加最小接口：

```go
type StoreResolver interface {
    ResolveSQLitePath(pluginName string) (string, error)
}

type KVStore interface {
    Get(ctx context.Context, pluginName, key string) (value string, found bool, err error)
    Set(ctx context.Context, pluginName, key, value string) error
    Delete(ctx context.Context, pluginName, key string) error
}
```

v0.3-dev 当前已有 `StorageGet`、`StorageSet`、`StorageDelete` 的 Host API 内存实现；下一步可以在保持 Host API 契约不变的前提下，将底层从 `MemoryKVStore` 替换为 SQLite-backed KV。
