## 设计理念

### 为什么需要插件系统？

CampusOS 的核心定位是"社区操作系统"而非"论坛应用"。这意味着：

1. **核心系统只提供原子能力**：身份、权限、存储、事件总线等
2. **所有业务功能通过插件实现**：论坛、AI 助手、机器人等都是插件
3. **第三方开发者可以扩展系统**：无需修改核心代码即可添加新功能

### 设计目标

| 目标 | 描述 |
|:---|:---|
| **灵活扩展** | 支持从简单的事件监听到复杂的长驻服务 |
| **安全隔离** | 插件故障不影响核心系统和其他插件 |
| **统一接口** | 一套 API 适配多种运行时 |
| **易于开发** | 提供完善的 SDK、文档和调试工具 |
| **可分发** | 支持插件市场，一键安装卸载 |

## Manifest 规范

所有插件必须提供 `plugin.yaml` 配置清单，声明插件的元信息、运行环境、事件订阅与权限需求。

### 完整 Manifest 示例

```yaml
# plugin.yaml - 插件配置清单
apiVersion: campusos/v1alpha1
kind: Plugin

# ─── 元信息 ───
metadata:
  name: "content-moderator"            # 插件唯一标识（小写 + 连字符）
  version: "1.2.0"                     # 语义化版本号
  displayName: "智能内容审核"            # 显示名称
  description: "基于 AI 的帖子内容自动审核插件"  # 描述
  author: "CampusOS Team"              # 作者
  license: "MIT"                       # 许可证
  homepage: "https://github.com/campusos/content-moderator"
  repository: "https://github.com/campusos/content-moderator"
  tags: ["ai", "moderation", "content"]
  icon: "https://cdn.campusos.io/plugins/content-moderator/icon.png"
  
  # 兼容性声明
  compatibility:
    campusos: ">=0.3.0"                # 最低 CampusOS 版本
    go: ">=1.22"                       # Go 版本要求（gRPC 插件）

# ─── 运行环境 ───
spec:
  runtime: "wasm"                      # wasm | grpc
  
  # Wasm 运行时配置（runtime: wasm 时生效）
  wasm:
    entrypoint: "main.wasm"            # Wasm 入口文件
    memoryLimit: "128MB"               # 内存限制
    timeout: "30s"                     # 单次执行超时
    
  # gRPC 运行时配置（runtime: grpc 时生效）
  grpc:
    entrypoint: "plugin-server"        # 可执行文件路径
    port: 0                            # 端口（0 = 自动分配）
    healthCheck:                       # 健康检查配置
      path: "/health"
      interval: "10s"
    resources:
      memoryLimit: "512MB"
      cpuLimit: "0.5"

# ─── 事件订阅 ───
events:
  subscriptions:
    - topic: "thread.created.before"   # 订阅的事件主题
      handler: "onThreadBeforeCreate"  # 处理函数名
      priority: 100                    # 优先级（数字越小越先执行）
      timeout: "5s"                    # 该事件处理超时
      
    - topic: "thread.created.after"
      handler: "onThreadAfterCreate"
      priority: 200
      
    - topic: "post.created.before"
      handler: "onPostBeforeCreate"
      priority: 100

  # 事件发布声明（插件可以发布自定义事件）
  publications:
    - topic: "moderation.result"
      description: "审核结果事件"

# ─── 权限申请 ───
permissions:
  # 需要调用的 Host API 权限
  hostAPI:
    - resource: "thread"
      actions: ["read", "update"]      # 读取和更新帖子
    - resource: "user"
      actions: ["read"]                # 读取用户信息
    - resource: "notification"
      actions: ["send"]                # 发送通知
      
  # 需要访问的数据范围
  dataScope:
    - resource: "thread"
      fields: ["id", "title", "content", "status", "category_id", "author_id"]
    - resource: "user"
      fields: ["id", "username", "nickname", "status"]

# ─── 配置项 ───
configuration:
  schema:
    type: object
    properties:
      ai_model:
        type: string
        default: "deepseek-chat"
        description: "使用的 AI 模型"
      sensitivity_threshold:
        type: number
        default: 0.8
        minimum: 0
        maximum: 1
        description: "敏感内容阈值"
      auto_reject:
        type: boolean
        default: false
        description: "是否自动拒绝高风险内容"
    required: ["ai_model"]

# ─── 依赖声明 ───
dependencies:
  plugins: []                          # 依赖的其他插件
  services:                            # 依赖的核心服务
    - "ai-gateway"
    - "event-bus"
```

### Manifest 字段说明

| 字段 | 必填 | 说明 |
|:---|:---|:---|
| `apiVersion` | ✅ | API 版本，当前为 `campusos/v1alpha1` |
| `kind` | ✅ | 资源类型，固定为 `Plugin` |
| `metadata.name` | ✅ | 插件唯一标识，全局唯一 |
| `metadata.version` | ✅ | 语义化版本号 |
| `spec.runtime` | ✅ | 运行时类型：`wasm` 或 `grpc` |
| `events.subscriptions` | 可选 | 事件订阅列表 |
| `permissions.hostAPI` | 可选 | Host API 权限申请 |
| `configuration.schema` | 可选 | 插件配置的 JSON Schema |

## 统一 Plugin Manager

Plugin Manager 是插件系统的核心控制器，负责插件的注册、调度、生命周期管理。

### 核心接口

```go
// PluginManager 插件管理器接口
type PluginManager interface {
    // Install 安装插件
    Install(ctx context.Context, manifest *Manifest, artifact []byte) error
    
    // Uninstall 卸载插件
    Uninstall(ctx context.Context, pluginID string) error
    
    // Enable 启用插件
    Enable(ctx context.Context, pluginID string) error
    
    // Disable 禁用插件
    Disable(ctx context.Context, pluginID string) error
    
    // GetPlugin 获取插件信息
    GetPlugin(ctx context.Context, pluginID string) (*PluginInfo, error)
    
    // ListPlugins 列出所有插件
    ListPlugins(ctx context.Context, filter PluginFilter) ([]*PluginInfo, error)
    
    // UpdateConfig 更新插件配置
    UpdateConfig(ctx context.Context, pluginID string, config map[string]interface{}) error
    
    // HandleEvent 分发事件到插件
    HandleEvent(ctx context.Context, event *CloudEvent) error
}

// PluginRuntime 插件运行时统一接口
type PluginRuntime interface {
    // Init 初始化运行时
    Init(ctx context.Context, manifest *Manifest) error
    
    // Invoke 调用插件处理函数
    Invoke(ctx context.Context, function string, payload []byte) ([]byte, error)
    
    // Shutdown 关闭运行时
    Shutdown(ctx context.Context) error
    
    // HealthCheck 健康检查
    HealthCheck(ctx context.Context) error
}
```

### 调度策略

Plugin Manager 根据插件的 Manifest 自动选择运行时：

```
Manifest 到达 Plugin Manager
         │
         ▼
    解析 manifest.spec.runtime
         │
         ├── "wasm" → Wasm Runtime
         │     ├── 加载 .wasm 模块
         │     ├── 创建沙箱实例
         │     └── 注册事件处理器
         │
         └── "grpc" → gRPC Runtime
               ├── 启动独立进程
               ├── 建立 gRPC 连接
               ├── 执行健康检查
               └── 注册事件处理器
```

## Wasm Runtime

### 适用场景

- **事件监听器**：轻量级的事件处理（如内容审核、数据统计）
- **内容过滤**：Markdown 扩展、XSS 过滤
- **AI 后处理**：调用 AI 网关进行内容分析
- **数据转换**：格式转换、字段映射

### 安全隔离

Wasm 运行时提供天然的安全隔离：

| 隔离维度 | 实现方式 |
|:---|:---|
| **内存隔离** | Wasm 模块运行在独立的线性内存空间，无法访问宿主内存 |
| **文件系统** | 无直接文件系统访问，必须通过 Host API |
| **网络访问** | 无直接网络访问，必须通过 Host API |
| **系统调用** | 不支持直接系统调用 |
| **资源限制** | 内存上限、执行超时、CPU 配额 |

### 生命周期

```
事件到达
    │
    ▼
从对象池获取 Wasm 实例（或创建新实例）
    │
    ▼
加载事件 payload 到模块内存
    │
    ▼
调用事件处理函数
    │
    ▼
获取处理结果
    │
    ▼
清理模块内存，归还实例到对象池
```

### Proto 定义

```protobuf
// plugin_service.proto
syntax = "proto3";
package campusos.plugin;

service PluginService {
  // 获取插件清单
  rpc GetManifest(GetManifestRequest) returns (GetManifestResponse);
  
  // 处理事件
  rpc HandleEvent(HandleEventRequest) returns (HandleEventResponse);
  
  // 健康检查
  rpc HealthCheck(HealthCheckRequest) returns (HealthCheckResponse);
  
  // 优雅关闭
  rpc Shutdown(ShutdownRequest) returns (ShutdownResponse);
  
  // 双向流 - 用于实时事件推送
  rpc StreamEvents(stream StreamEventRequest) returns (stream StreamEventResponse);
}

message HandleEventRequest {
  string event_id = 1;
  string topic = 2;
  string source = 3;
  bytes payload = 4;
  map<string, string> attributes = 5;
}

message HandleEventResponse {
  enum Action {
    ALLOW = 0;      // 允许操作继续
    BLOCK = 1;      // 阻断操作
    MODIFY = 2;     // 修改 payload
  }
  
  Action action = 1;
  bytes modified_payload = 2;  // action=MODIFY 时使用
  string reason = 3;           // 操作原因
}
```

### 健康检查与自动重启

```go
// ProcessManager 管理插件进程的生命周期
type ProcessManager struct {
    plugins map[string]*ManagedProcess
}

type ManagedProcess struct {
    PluginID  string
    Cmd       *exec.Cmd
    Client    PluginServiceClient
    Status    ProcessStatus
    Restarts  int
    MaxRestarts int  // 最大重启次数，默认 3
}

// 健康检查循环
func (pm *ProcessManager) healthCheckLoop(ctx context.Context, p *ManagedProcess) {
    ticker := time.NewTicker(10 * time.Second)
    defer ticker.Stop()
    
    consecutiveFailures := 0
    
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            _, err := p.Client.HealthCheck(ctx, &HealthCheckRequest{})
            if err != nil {
                consecutiveFailures++
                if consecutiveFailures >= 3 {
                    // 连续 3 次失败，触发重启
                    pm.restartPlugin(ctx, p)
                    consecutiveFailures = 0
                }
            } else {
                consecutiveFailures = 0
            }
        }
    }
}
```

## Host API

Host API 是插件与核心系统交互的唯一通道，无论 Wasm 还是 gRPC 插件都通过统一接口访问。

### API 列表

| 类别 | API | 说明 |
|:---|:---|:---|
| **身份** | `host.identity.getUser(id)` | 获取用户信息 |
| **身份** | `host.identity.searchUsers(query)` | 搜索用户 |
| **权限** | `host.permission.check(user, resource, action)` | 权限检查 |
| **存储** | `host.storage.upload(file)` | 上传文件 |
| **存储** | `host.storage.getURL(key)` | 获取文件 URL |
| **数据** | `host.data.query(resource, filter)` | 查询数据 |
| **数据** | `host.data.get(resource, id)` | 获取单条数据 |
| **数据** | `host.data.update(resource, id, data)` | 更新数据 |
| **事件** | `host.event.publish(topic, payload)` | 发布事件 |
| **通知** | `host.notification.send(user, template, data)` | 发送通知 |
| **配置** | `host.config.get(key)` | 读取配置 |
| **日志** | `host.log.info/warn/error(msg)` | 写入日志 |
| **AI** | `host.ai.chatCompletion(messages)` | 调用大模型 |

### Host API 权限控制

每个插件在调用 Host API 时，Plugin Manager 会检查其 Manifest 中声明的权限：

```
插件调用 host.storage.upload(file)
         │
         ▼
Plugin Manager 检查权限:
  manifest.permissions.hostAPI 是否包含
  { resource: "storage", action: "upload" }
         │
         ├── 有权限 → 执行操作
         └── 无权限 → 返回 PermissionDenied 错误
```

### Wasm Host API 实现

```go
// 注册 Host Functions 到 Wasm 模块
func (r *WasmRuntime) registerHostFunctions(module wazero.HostModuleBuilder) {
    module.NewFunctionBuilder().
        WithFunc(r.hostAPICall).
        WithParameterNames("resource_ptr", "resource_len", "action_ptr", "action_len",
                           "params_ptr", "params_len", "result_ptr", "result_len").
        Export("host_api_call")
        
    module.NewFunctionBuilder().
        WithFunc(r.log).
        WithParameterNames("level", "message_ptr", "message_len").
        Export("log")
        
    module.NewFunctionBuilder().
        WithFunc(r.emitEvent).
        WithParameterNames("topic_ptr", "topic_len", "payload_ptr", "payload_len").
        Export("emit_event")
}

// hostAPICall 实现 - 带权限检查的 API 调用
func (r *WasmRuntime) hostAPICall(ctx context.Context, 
    resourcePtr, resourceLen, actionPtr, actionLen,
    paramsPtr, paramsLen, resultPtr, resultLen uint32) uint64 {
    
    // 1. 从 Wasm 内存读取参数
    resource := r.readStringFromMemory(ctx, resourcePtr, resourceLen)
    action := r.readStringFromMemory(ctx, actionPtr, actionLen)
    params := r.readBytesFromMemory(ctx, paramsPtr, paramsLen)
    
    // 2. 权限检查
    if !r.checkPermission(resource, action) {
        r.writeErrorToMemory(ctx, resultPtr, resultLen, "permission denied")
        return 1 // 错误码
    }
    
    // 3. 执行 API 调用
    result, err := r.hostAPI.Execute(ctx, resource, action, params)
    if err != nil {
        r.writeErrorToMemory(ctx, resultPtr, resultLen, err.Error())
        return 1
    }
    
    // 4. 将结果写回 Wasm 内存
    r.writeBytesToMemory(ctx, resultPtr, resultLen, result)
    return 0 // 成功
}
```

## 插件生命周期

```
┌─────────┐     ┌──────────┐     ┌──────────┐     ┌─────────┐
│  等待中  │────→│  初始化中 │────→│  运行中   │────→│  停止中  │
│ (Pending)│     │ (Init)   │     │ (Running)│     │(Stopping)│
└─────────┘     └──────────┘     └──────────┘     └─────────┘
     │               │                │                 │
     │               │                │                 │
     ▼               ▼                ▼                 ▼
  安装失败        初始化失败        运行异常           已停止
  (Failed)        (Failed)        (Error)          (Stopped)
                     │                │
                     │                │
                     ▼                ▼
                  自动重试          自动重启
                  (Retrying)       (Restarting)
```

### 状态说明

| 状态 | 说明 |
|:---|:---|
| **Pending** | 插件已安装，等待启用 |
| **Init** | 正在初始化运行时环境 |
| **Running** | 插件正常运行，可接收事件 |
| **Error** | 插件运行异常，等待恢复 |
| **Restarting** | 插件正在自动重启（仅 gRPC） |
| **Stopping** | 正在优雅停止插件 |
| **Stopped** | 插件已停止 |
| **Failed** | 插件启动或运行失败，已放弃重试 |

### 生命周期事件

| 事件 | 触发时机 |
|:---|:---|
| `plugin.installed` | 插件安装完成 |
| `plugin.enabled` | 插件被启用 |
| `plugin.disabled` | 插件被禁用 |
| `plugin.uninstalled` | 插件被卸载 |
| `plugin.error` | 插件运行出错 |
| `plugin.config.updated` | 插件配置更新 |

## 事件交互模型

### `.before` 事件（同步拦截）

`.before` 事件在核心操作执行前触发，允许插件**阻断或修改**操作：

```
核心服务即将执行操作（如创建帖子）
         │
         ▼
发布 thread.created.before 事件
         │
         ▼
Plugin Manager 按优先级依次调用订阅插件:
  │
  ├── Plugin A (priority: 100)
  │   └── 返回: { action: ALLOW }
  │
  ├── Plugin B (priority: 200) - 内容审核插件
  │   └── 返回: { action: MODIFY, payload: { content: "过滤后的内容" } }
  │
  └── Plugin C (priority: 300)
      └── 返回: { action: BLOCK, reason: "包含敏感词" }
         │
         ▼
汇总所有插件结果:
  ├── 任一返回 BLOCK → 阻断操作，返回错误
  ├── 有 MODIFY → 使用修改后的 payload
  └── 全部 ALLOW → 继续执行核心操作
```

### `.after` 事件（异步通知）

`.after` 事件在核心操作执行后触发，用于异步通知和副作用处理：

```
核心操作已完成（如帖子已创建）
         │
         ▼
发布 thread.created.after 事件
         │
         ▼
Plugin Manager 并行通知所有订阅插件（不等待结果）:
  │
  ├── Plugin A: 更新统计计数
  ├── Plugin B: 发送通知给关注者
  └── Plugin C: 触发 AI 内容分析
```

## 权限与安全

### 插件权限清单

每个插件必须在 Manifest 中显式声明所需权限，遵循最小权限原则：

```yaml
permissions:
  hostAPI:
    # 只申请需要的资源和操作
    - resource: "thread"
      actions: ["read"]              # 只读，不能写
    - resource: "user"
      actions: ["read"]
    - resource: "notification"
      actions: ["send"]
      
  # 未声明的权限 = 无权限
  # 例如：插件未声明 storage 权限，则无法调用文件上传 API
```

### 运行时安全策略

| 安全措施 | Wasm | gRPC |
|:---|:---|:---|
| **内存隔离** | ✅ 天然沙箱 | ✅ 进程隔离 |
| **文件系统** | ❌ 禁止直接访问 | ⚠️ 受限访问（通过 chroot/namespace） |
| **网络访问** | ❌ 仅通过 Host API | ⚠️ 受限访问（防火墙规则） |
| **资源限制** | ✅ 内存上限 + 超时 | ✅ cgroups 资源限制 |
| **权限检查** | ✅ 每次 Host API 调用 | ✅ 每次 Host API 调用 |
| **代码签名** | ✅ 支持 | ✅ 支持 |

## 插件间通信

插件之间不直接通信，而是通过 Event Bus 间接通信：

```
┌──────────┐                    ┌──────────┐
│ Plugin A │                    │ Plugin B │
│          │                    │          │
│ 发布事件: │    Event Bus       │ 订阅事件: │
│ custom.x │ ──── NATS ────→  │ custom.x │
│          │                    │          │
└──────────┘                    └──────────┘
```

**规范：**

- 插件发布的自定义事件 topic 必须以插件名为前缀：`{plugin_name}.{event_name}`
- 插件不得直接调用其他插件的函数
- 插件间数据共享应通过核心数据层（PostgreSQL）

## 错误处理与容错

### 超时处理

```
事件处理超时（默认 30s，.before 事件默认 5s）
         │
         ▼
Plugin Manager 强制终止插件执行
         │
         ▼
记录超时错误日志
         │
         ├── .before 事件: 该插件被视为 ALLOW（不阻断主流程）
         └── .after 事件: 丢弃，不影响主流程
```

### 重试策略（gRPC）

| 参数 | 默认值 | 说明 |
|:---|:---|:---|
| 最大重试次数 | 3 | 连续失败后放弃 |
| 重试间隔 | 指数退避 | 1s → 2s → 4s |
| 健康检查间隔 | 10s | 定期探测插件状态 |
| 连续失败阈值 | 3 次 | 触发重启 |

### 熔断机制

```
插件连续失败次数达到阈值
         │
         ▼
进入熔断状态（一段时间内不再调用该插件）
         │
         ▼
熔断超时后进入半开状态（允许一次试探调用）
         │
         ├── 成功 → 恢复正常
         └── 失败 → 重新进入熔断状态
```

## 插件示例

### 示例 1：内容审核插件 (Wasm)

```go
// plugins/content-moderator/main.go
package main

import (
    "github.com/campusos/sdk/wasm"
)

func main() {
    wasm.Handle("onThreadBeforeCreate", func(ctx *wasm.Context, event *wasm.Event) *wasm.Result {
        // 解析帖子内容
        var thread struct {
            Title   string `json:"title"`
            Content string `json:"content"`
        }
        event.UnmarshalPayload(&thread)
        
        // 调用 AI 网关进行内容审核
        result, err := ctx.Host.AI.ChatCompletion([]wasm.Message{
            {Role: "system", Content: "你是内容审核助手，请判断以下内容是否包含敏感信息。返回 JSON: {\"safe\": true/false, \"reason\": \"...\"}"},
            {Role: "user", Content: thread.Title + "\n" + thread.Content},
        })
        if err != nil {
            ctx.Log.Error("AI审核失败: " + err.Error())
            return wasm.Allow() // AI 失败时放行，降级处理
        }
        
        // 解析审核结果
        var review struct {
            Safe   bool   `json:"safe"`
            Reason string `json:"reason"`
        }
        wasm.Unmarshal(result, &review)
        
        if !review.Safe {
            return wasm.Block("内容审核未通过: " + review.Reason)
        }
        
        return wasm.Allow()
    })
}
```

### 示例 2：NapCat 适配器 (gRPC)

```go
// plugins/napcat-adapter/main.go
package main

import (
    "github.com/campusos/sdk/grpc"
    "github.com/campusos/sdk/grpc/hostapi"
)

type NapCatPlugin struct {
    grpc.UnimplementedPluginServiceServer
    hostAPI hostapi.Client
}

func (p *NapCatPlugin) HandleEvent(ctx context.Context, req *grpc.HandleEventRequest) (*grpc.HandleEventResponse, error) {
    switch req.Topic {
    case "thread.created.after":
        // 帖子创建后，同步到 QQ 群
        return p.syncToQQGroup(ctx, req.Payload)
    case "post.created.after":
        // 回帖后，通知 QQ 群
        return p.notifyQQGroup(ctx, req.Payload)
    }
    return &grpc.HandleEventResponse{Action: grpc.Action_ALLOW}, nil
}

func (p *NapCatPlugin) syncToQQGroup(ctx context.Context, payload []byte) (*grpc.HandleEventResponse, error) {
    // 1. 解析帖子数据
    // 2. 格式化为 QQ 消息格式
    // 3. 通过 NapCat API 发送到 QQ 群
    return &grpc.HandleEventResponse{Action: grpc.Action_ALLOW}, nil
}

func main() {
    plugin := &NapCatPlugin{
        hostAPI: hostapi.NewClient(),
    }
    
    // 连接到 NapCat WebSocket
    go plugin.connectToNapCat()
    
    // 启动 gRPC 服务
    grpc.Serve(plugin)
}
```
