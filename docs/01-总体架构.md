## 架构设计原则

CampusOS 的架构设计遵循以下核心原则：

### 1. 六边形架构 / 洋葱架构 (Hexagonal / Onion Architecture)

依赖方向严格由外向内，核心层不依赖任何上层：

- **核心层（Core）** 不知道外层的存在，也不依赖任何框架
- **外层依赖内层**，通过接口（Port）与内层通信
- **适配器（Adapter）** 实现具体的技术细节（如数据库、消息队列）

### 2. API First

- 所有服务的接口定义先于实现
- 使用 OpenAPI 3.0 定义 RESTful API
- 使用 Protobuf 定义 gRPC 服务接口
- 接口定义即文档，即契约

### 3. Plugin First

- 核心系统仅提供原子能力，所有业务功能优先通过插件实现
- 核心系统与业务应用严格分离
- 插件可以通过事件订阅与 Host API 双向交互

### 4. Event Driven

- 所有核心行为必须通过 Event Bus 广播
- 服务间通信优先使用异步事件，而非同步 RPC
- 事件是系统的"神经系统"，驱动一切业务流转

## 分层架构

系统采用严格的六边形架构思想，由外向内分为 6 层：

```
                    ┌─────────────────────────────────────────────────────────┐
                    │              Presentation Layer (表现层)                  │
                    │         RESTful API / GraphQL / WebSocket / Web UI       │
                    └───────────────────────────┬─────────────────────────────┘
                                                │
                    ┌───────────────────────────▼─────────────────────────────┐
                    │               Gateway Layer (网关层)                      │
                    │            统一鉴权 / 限流 / 路由分发                       │
                    └───────────────────────────┬─────────────────────────────┘
                                                │
        ┌───────────────────────────────────────┼───────────────────────────────┐
        │                                       │                               │
        ▼                                       ▼                               ▼
┌───────────────┐                    ┌───────────────────┐            ┌──────────────────┐
│  Core Layer   │                    │  Platform Layer   │            │  Plugin Runtime  │
│  (核心底座)    │                    │  (平台层)          │            │  (插件运行时)     │
│               │                    │                   │            │                  │
│ · Identity    │                    │ · Event Bus       │            │ · Wasm Runtime   │
│ · Permission  │                    │ · Task Queue      │            │ · gRPC Runtime   │
│ · Storage     │◄───────────────────│ · Cache           │            │ · Plugin Manager │
│ · Audit       │                    │                   │            │                  │
│ · Notification│                    └─────────┬─────────┘            └────────┬─────────┘
│ · Configuration│                              │                               │
└───────┬───────┘                              │                               │
        │                                      │                               │
        └──────────────────────────────────────┼───────────────────────────────┘
                                               │
                    ┌──────────────────────────▼──────────────────────────────┐
                    │           Infrastructure Layer (基础设施层)                │
                    │      PostgreSQL / Redis / MinIO / NATS / OpenTelemetry   │
                    └─────────────────────────────────────────────────────────┘
```

### 分层职责概览

| 层级 | 英文名称 | 职责 | 关键技术 |
|:---|:---|:---|:---|
| **L1 - 表现层** | Presentation | 对外暴露统一 API 接口与 Web UI | Gin / Echo, GraphQL, WebSocket |
| **L2 - 网关层** | Gateway | 统一鉴权、限流、路由、日志 | 中间件链 |
| **L3 - 核心层** | Core | 提供身份、权限、存储等原子能力 | 领域驱动设计 (DDD) |
| **L4 - 平台层** | Platform | 事件总线、任务队列、缓存调度 | NATS, Redis, Asynq |
| **L5 - 插件运行时** | Plugin Runtime | Wasm / gRPC 插件宿主环境 | WasmEdge/Wazero, gRPC |
| **L6 - 基础设施层** | Infrastructure | 底层存储、消息、可观测性 | PostgreSQL, Redis, MinIO |

## 各层详细说明

### L1 - Presentation Layer（表现层）

表现层是系统与外部交互的唯一入口，负责将内部能力以标准化接口对外暴露。

**职责：**

- 提供 RESTful API 接口
- 提供 GraphQL 查询能力（可选）
- 提供 WebSocket 实时通信通道
- 提供 Web UI 静态资源服务

**设计要点：**

- 表现层不包含任何业务逻辑，仅负责请求的接收、验证与响应的格式化
- 统一响应体封装：`{ "code": 0, "msg": "success", "data": {} }`
- 统一错误码体系，所有错误通过标准格式返回
- 请求参数验证在表现层完成，业务层不再重复验证

**示例：**

```go
// handler/user.go
func (h *UserHandler) GetUser(c *gin.Context) {
    id := c.Param("id")
    
    // 调用核心服务
    user, err := h.userService.GetByID(c.Request.Context(), id)
    if err != nil {
        c.JSON(http.StatusOK, response.Error(ErrCodeUserNotFound, "user not found"))
        return
    }
    
    c.JSON(http.StatusOK, response.Success(user))
}
```

### L2 - Gateway Layer（网关层）

网关层作为所有请求的拦截器，在请求到达业务逻辑之前完成通用处理。

**职责：**

- **统一鉴权**：验证 JWT Token / API Key，提取用户身份
- **请求限流**：基于令牌桶算法的细粒度 Rate Limiting
- **路由分发**：根据请求路径将请求路由到对应的服务
- **日志记录**：记录请求的 Trace ID、耗时、状态码
- **CORS 处理**：跨域资源共享配置

**中间件链设计：**

```
请求 → TraceID注入 → CORS → 限流 → 鉴权 → 路由分发 → 业务处理 → 响应
```

**示例中间件配置：**

```go
router := gin.Default()

// 通用中间件
router.Use(middleware.TraceID())
router.Use(middleware.CORS())
router.Use(middleware.RateLimit(100, time.Minute))  // 100 req/min

// 需要鉴权的路由组
auth := router.Group("/api/v1")
auth.Use(middleware.Auth())
{
    auth.GET("/users/:id", userHandler.GetUser)
    auth.POST("/threads", threadHandler.CreateThread)
}

// 公开路由组（无需鉴权）
public := router.Group("/api/v1")
{
    public.POST("/auth/login", authHandler.Login)
    public.GET("/threads", threadHandler.ListThreads)
}
```

### L3 - Core Layer（核心层 / 核心底座）

核心层是整个系统的基石，提供所有上层应用所需的通用原子能力。它是平台无关的，不依赖任何具体技术实现。

**核心服务：**

| 服务 | 英文 | 职责概述 |
|:---|:---|:---|
| 身份服务 | Identity | 用户注册、登录、多端统一认证 |
| 权限中心 | Permission | RBAC + ABAC 细粒度权限控制 |
| 存储引擎 | Storage | 统一文件抽象层（VFS） |
| 审计中心 | Audit | 防篡改的操作日志记录 |
| 通知服务 | Notification | 多渠道消息触达 |
| 配置中心 | Configuration | 热更新的动态配置管理 |

> 各核心服务的详细设计请参阅 [02-核心设计.md](./02-核心设计.md)

### L4 - Platform Layer（平台层）

平台层是系统的"中枢神经"，提供跨服务的通用基础设施能力。

**核心组件：**

#### Event Bus（事件总线）

- 基于 NATS 实现
- 采用 CNCF CloudEvents 1.0 规范
- 支持 `.before`（同步拦截）和 `.after`（异步通知）双模式
- At-least-once 投递保障

```
                    ┌─────────────┐
                    │  Event Bus  │
                    │   (NATS)    │
                    └──────┬──────┘
                           │
         ┌─────────────────┼─────────────────┐
         │                 │                 │
    ┌────▼────┐      ┌────▼────┐      ┌────▼────┐
    │ Plugin A │      │ Plugin B │      │ Plugin C │
    │ (Wasm)  │      │ (gRPC)  │      │ (Wasm)  │
    └─────────┘      └─────────┘      └─────────┘
```

#### Task Queue（任务队列）

- 基于 Redis 实现（Asynq）
- 支持延迟任务、定时任务、重试机制
- 适用于：邮件发送、数据统计、定期清理等异步操作

#### Cache（缓存调度）

- 基于 Redis 实现
- 统一缓存接口，支持 TTL、缓存穿透保护
- 多级缓存策略：本地缓存 + 分布式缓存

### L5 - Plugin Runtime Layer（插件运行时层）

插件运行时层是 CampusOS 最核心的扩展机制，采用双运行时设计。

**运行时对比：**

| 维度 | Wasm Runtime | gRPC Runtime |
|:---|:---|:---|
| **隔离级别** | 内存级沙箱隔离 | 进程级网络隔离 |
| **资源消耗** | 极低（MB 级） | 中等（独立进程） |
| **通信方式** | 函数调用（进程内） | gRPC（进程间） |
| **适用场景** | 事件监听、内容审核、Markdown 扩展 | IM 适配器、RAG 服务、长连接 |
| **开发语言** | Go / Rust / TypeScript | 任意支持 gRPC 的语言 |
| **生命周期** | 随事件触发，用完即销毁 | 长期驻留，独立管理 |

**统一 Plugin Manager：**

```
┌──────────────────────────────────────────────────┐
│                Plugin Manager                     │
│                                                   │
│  ┌─────────────┐  ┌─────────────┐  ┌──────────┐  │
│  │  Manifest   │  │  Scheduler  │  │ Lifecycle │  │
│  │  Parser     │  │             │  │ Manager   │  │
│  └──────┬──────┘  └──────┬──────┘  └─────┬────┘  │
│         │                │               │        │
│         ▼                ▼               ▼        │
│  ┌─────────────────────────────────────────────┐  │
│  │          Plugin Runtime Interface            │  │
│  └──────────┬──────────────────────┬───────────┘  │
│             │                      │              │
│    ┌────────▼────────┐    ┌────────▼────────┐     │
│    │  Wasm Runtime   │    │  gRPC Runtime   │     │
│    │  (WasmEdge/     │    │  (进程外托管)     │     │
│    │   Wazero)       │    │                  │     │
│    └─────────────────┘    └──────────────────┘     │
└──────────────────────────────────────────────────┘
```

> 插件系统的详细设计请参阅 [03-插件系统.md](./03-插件系统.md)

### L6 - Infrastructure Layer（基础设施层）

基础设施层提供系统运行所需的所有底层依赖。

| 组件 | 技术选型 | 用途 |
|:---|:---|:---|
| **关系型数据库** | PostgreSQL 16+ | 核心业务数据存储 |

| **对象存储** | MinIO (S3 兼容) | 文件、图片、附件存储 |

| **链路追踪** | Jaeger | 分布式链路追踪 |
| **容器编排** | Docker / Kubernetes | 应用部署与编排 |

## 架构决策记录 (ADR)

### ADR-001: 采用六边形架构

- **状态**：已采纳
- **背景**：需要确保核心业务逻辑与技术实现解耦
- **决策**：采用六边形架构，核心层通过接口（Port）定义能力，基础设施层通过适配器（Adapter）实现
- **后果**：核心层可以独立测试，基础设施可替换

### ADR-002: 采用 NATS 作为 Event Bus

- **状态**：已采纳
- **背景**：需要高性能、轻量级的消息系统
- **决策**：采用 NATS 而非 Kafka/RabbitMQ
- **理由**：NATS 单节点即可支撑百万级消息/秒，部署简单，原生支持 JetStream 持久化
- **后果**：需要自行实现 CloudEvents 封装与死信队列

### ADR-003: 双运行时插件架构

- **状态**：已采纳
- **背景**：需要同时满足轻量级扩展（内容审核）和重型集成（IM 适配器）需求
- **决策**：引入 Wasm（轻量沙箱）+ gRPC（进程外隔离）双运行时
- **后果**：增加了 Plugin Manager 的复杂度，但获得了最大的灵活性

### ADR-004: 全局采用分布式 ID

- **状态**：已采纳
- **背景**：需要支持未来分库分表与数据合并
- **决策**：所有核心业务表使用雪花算法生成的分布式 ID，禁止自增 ID
- **后果**：ID 不再可排序，但获得了全局唯一性和分布式友好性

### ADR-005: 强制软删除

- **状态**：已采纳
- **背景**：校园社区场景下数据误删的恢复成本极高
- **决策**：禁止物理删除，所有表必须包含 `deleted_at` 字段
- **后果**：查询需要额外过滤条件，但数据安全性得到保障
