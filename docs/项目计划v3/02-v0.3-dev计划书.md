# CampusOS v0.3-dev 计划书

> 文档状态：规划草案
> 编写日期：2026-06-26
> 最近审查：2026-06-27
> 适用范围：CampusOS v0.3-dev 开发、测试、部署与跨平台兼容规划
> 参考基线：项目计划 v1/v2/v3 文档、v0.2-dev 进度总结、当前代码结构、近期注册/发帖/迁移链路修复

## 1. 文档定位

v0.3-dev 不是重新做一个论坛系统，而是在 v0.1、v0.2 已经完成的身份、社区、权限、插件、事件、管理后台等基础之上，把 CampusOS 从“可运行的社区核心系统”推进到“可扩展、可隔离、可交付插件生态”的阶段。

本计划书主要回答四个问题：

| 问题 | 说明 |
| --- | --- |
| v1 完成了什么 | 回顾早期项目定位、领域拆分、数据库/API/开发规范与核心社区能力 |
| v2 完成了什么 | 回顾 v0.2-dev 阶段在权限、插件、后台、缓存、事件和前端上的落地 |
| v0.3-dev 要完成什么 | 明确 Wasm 插件运行时、Host API、权限隔离、插件 SDK/CLI、示例插件与验收标准 |
| 是否支持 Ubuntu 22.04 和 Windows | 可以尝试并建议纳入 v0.3-dev 目标，其中 Ubuntu 22.04 作为参考环境，Windows 以 WSL2/Docker Desktop 为推荐路径，原生 Windows 作为增强路径 |

### 2.1 近期修正后的基线变化

近期围绕注册、发帖和迁移链路做过一轮修正，v0.3-dev 计划应基于新的状态继续推进：

| 项目 | 当前变化 | 后续仍需处理 |
| --- | --- | --- |
| 注册链路 | 用户和账号创建已使用应用侧数字 ID，避免 `accounts.id` 为空 | 继续检查 sessions、audit、notifications 等边界写入路径 |
| 发帖链路 | 发帖/回帖从 JWT context 获取用户，不再依赖前端传 `X-User-ID`，并新增 JWT 发帖回归测试 | 后续扩展注册、登录、回帖端到端测试 |
| 社区域 ID | thread/post/category/plugin/api_key 创建改为数字 ID，匹配当前 BIGINT schema | 长期保持 API 以字符串暴露 ID，避免前端大整数精度问题 |
| 默认版块 | migration 和服务启动逻辑都能确保默认版块存在 | 后续可扩展为可配置的初始化数据 |
| 数据库迁移 | `make migrate-up/down/reset/status` 已由 `scripts/migrate.sh` 承载，并通过 `schema_migrations` 记录版本 | 长期可迁入 `campusosctl migrate` 或 `golang-migrate` |
| 前端发帖页 | 发帖页改为选择真实版块，且 `/threads/create` 需要登录 | 后续应补端到端测试，防止 401 后误清登录态的体验回归 |

## 5. v0.3-dev 总目标

v0.3-dev 的核心目标是：

> 让 CampusOS 的插件系统从“可以接入”升级为“可以安全运行、可以被开发者高效开发、可以被管理员可靠管理”。

建议将 v0.3-dev 的目标分成三个优先级：

| 优先级 | 目标 | 说明 |
| --- | --- | --- |
| P0 | 插件安全运行闭环 | Wasm 运行时、Host API、权限校验、超时/资源限制、插件崩溃隔离 |
| P1 | 插件开发闭环 | SDK、CLI、模板、示例插件、插件打包、插件安装与启停 |
| P2 | 插件运营闭环 | 管理后台配置、日志、审计、统计、市场化元数据、跨平台交付文档 |

## 6. v0.3-dev 计划任务包

### 6.1 插件运行时：Wasm Runtime

| 任务 | 说明 | 优先级 |
| --- | --- | --- |
| 设计统一 Runtime 接口 | 让 gRPC Runtime 与 Wasm Runtime 共享生命周期接口 | P0 |
| 引入 Wasm 执行引擎 | 建议优先评估 wazero，减少外部系统依赖 | P0 |
| 加载 `.wasm` 插件包 | 支持从插件目录或插件仓库加载 wasm 文件 | P0 |
| 插件生命周期 | 支持 install、enable、disable、reload、uninstall | P0 |
| 超时控制 | 每次插件调用必须有 context deadline | P0 |
| 内存限制 | 防止单个插件占用过多资源 | P0 |
| Panic/Trap 隔离 | 插件异常不能拖垮主服务 | P0 |
| 调用日志 | 记录插件调用耗时、错误、事件来源 | P1 |

### 6.2 Host API 与宿主能力

v0.3-dev 应把 Host API 从“已有雏形”推进为“稳定契约”。建议先固定一版最小可用 API：

| Host API | 能力说明 | 优先级 |
| --- | --- | --- |
| `GetUser` | 查询用户基础资料 | P0 |
| `GetThread` | 查询主题基础信息 | P0 |
| `GetReply` | 查询回复基础信息 | P1 |
| `PublishEvent` | 插件发布标准事件 | P0 |
| `SendNotification` | 发送站内通知或系统通知 | P1 |
| `GetConfig` | 读取插件配置 | P0 |
| `SetConfig` | 写入插件配置 | P1 |
| `CheckPermission` | 查询当前用户是否拥有某权限 | P0 |
| `Log` | 插件写入结构化日志 | P0 |

Host API 必须配套权限校验，不允许插件通过宿主接口绕过 RBAC、内容可见性或管理员配置。

### 6.3 插件权限与隔离模型

| 任务 | 说明 | 优先级 |
| --- | --- | --- |
| 插件 Manifest 权限声明 | 插件必须声明需要的 API、事件、配置、存储和网络能力 | P0 |
| 安装时权限审查 | 管理员安装插件时能看到权限清单 | P0 |
| 运行时权限校验 | Host API 每次调用都按插件授权校验 | P0 |
| 事件订阅权限 | 插件只能订阅被允许的事件类型 | P0 |
| 数据字段最小化 | 默认只返回插件需要的字段 | P1 |
| 审计记录 | 记录插件调用敏感 Host API 的行为 | P1 |

建议 Manifest 先采用稳定、易读的 JSON 或 YAML 结构，并在 v0.3-dev 内固定版本号，例如：

```yaml
manifest_version: 1
name: example-moderation
runtime: wasm
permissions:
  host_api:
    - GetUser
    - GetThread
    - PublishEvent
  events:
    subscribe:
      - thread.created
      - reply.created
    publish:
      - moderation.flagged
  storage:
    type: sqlite
```

### 6.4 插件存储模型

v0.3-dev 建议支持三种插件存储策略：

| 类型 | 说明 | v0.3-dev 建议 |
| --- | --- | --- |
| `none` | 插件无持久化存储 | 必须支持 |
| `sqlite` | 每个插件独立 SQLite 文件 | P0，推荐作为 Wasm 插件默认持久化方式 |
| `postgres` | 插件使用 PostgreSQL 命名空间或前缀表 | P1，适合高并发或强查询插件 |

SQLite 更适合作为 v0.3-dev 的默认目标，因为它跨平台、易分发、隔离性更好，也符合插件系统早期生态建设需求。

### 6.5 插件 SDK

| 任务 | 说明 | 优先级 |
| --- | --- | --- |
| Go SDK | 提供 Host API 调用封装、事件模型、Manifest 类型 | P0 |
| 示例工程模板 | 一键生成可编译插件 | P0 |
| 本地测试工具 | 不启动完整 CampusOS 也能测试插件逻辑 | P1 |
| 版本兼容声明 | SDK 与 CampusOS 主程序版本要有兼容矩阵 | P1 |

SDK 的目标不是把插件开发变复杂，而是让开发者只关注业务逻辑：

```go
func OnThreadCreated(ctx campusos.Context, event campusos.Event) error {
    thread, err := ctx.Host.GetThread(event.ResourceID)
    if err != nil {
        return err
    }
    return ctx.Host.Log("thread.created handled: " + thread.Title)
}
```

### 6.6 插件 CLI

建议新增 `campusos` 或 `campusosctl` 命令行工具，优先支持：

| 命令 | 说明 | 优先级 |
| --- | --- | --- |
| `plugin init` | 创建插件模板 | P0 |
| `plugin build` | 编译插件为 wasm 或 gRPC 二进制 | P0 |
| `plugin test` | 本地执行插件单元测试或模拟事件 | P1 |
| `plugin pack` | 打包插件、Manifest、静态资源 | P1 |
| `plugin inspect` | 查看插件权限、版本、入口点 | P1 |
| `migrate up` | 跨平台执行数据库迁移 | P1 |

如果希望更好兼容 Windows，数据库迁移、插件构建、初始化等任务应尽量放入 Go CLI，而不是只依赖 Bash/Makefile。

### 6.7 示例插件

v0.3-dev 至少应提供 2 到 3 个示例插件，用来验证运行时、权限、事件和存储能力：

| 插件 | 验证目标 | 优先级 |
| --- | --- | --- |

### 6.8 管理后台增强

## 8. v0.3-dev 验收标准

v0.3-dev 完成时，建议以以下标准验收：

| 类别 | 验收标准 |
| --- | --- |
| 基础功能 | 注册、登录、发帖、回帖、后台登录、权限管理仍然正常 |
| 插件运行 | 至少一个 Wasm 插件和一个 gRPC 插件可以安装、启用、停用 |
| 插件隔离 | 插件崩溃、超时或返回错误不会导致主服务崩溃 |
| Host API | 插件可以通过授权 Host API 查询内容、发布事件、写日志 |
| 权限控制 | 未授权插件调用敏感 Host API 会被拒绝并记录日志 |
| 插件存储 | 示例插件可以使用独立 SQLite 存储并正常读写 |
| SDK | 开发者可以通过模板在较短时间内写出一个最小插件 |
| CLI | 插件 init、build、pack 至少形成闭环 |
| 管理后台 | 管理员可以看到插件状态、权限、配置和基本日志 |
| 文档 | Ubuntu 22.04 与 Windows/WSL2 都有可执行的启动说明 |
| 测试 | 后端 `go test ./...`、前端构建、核心集成测试通过 |

### 8.1 进入 v0.3-dev 主线前的最小门槛

v0.3-dev 的重心是插件生态，但插件系统会放大宿主系统的问题。因此，在正式投入 Wasm Runtime、SDK、CLI 前，建议先把以下基础链路作为 v0.2.10 或 v0.3.0-alpha 的门槛：

| 门槛 | 验证方式 | 通过标准 |
| --- | --- | --- |
| 全新数据库初始化 | `make docker-up` 后执行 `make migrate-up` | 5 组 migration 全部成功，默认管理员和默认版块存在 |
| 核心用户链路 | 注册、登录、`GET /auth/me` | access token 可用，用户信息正确返回 |
| 核心社区链路 | 登录后创建帖子、查看帖子、创建回复 | 不出现 401 闪退，数据写入 PostgreSQL |
| 前端构建 | `npm run build` in `web` 和 `admin` | 类型检查和 Vite 构建通过 |
| 后端测试 | `go test ./...` | 所有 Go 包通过 |
| 插件基线 | 启动服务后加载 hello-plugin | 插件加载失败不能影响主服务启动 |
| 数据库重置 | 开发库执行 `make migrate-reset` | 先逆序回滚，再正序迁移，适用于开发环境 |

这组门槛比完整 v0.3 验收更小，但能保证后续插件扩展不是建立在摇晃的基础链路上。

## 9. Ubuntu 22.04 与 Windows 兼容计划

结论：可以尝试做到 Ubuntu 22.04 和 Windows 系统都能使用该项目，而且建议从 v0.3-dev 开始纳入正式目标。

推荐分成两级支持：

| 支持等级 | 环境 | 建议定位 |
| --- | --- | --- |
| 一级支持 | Ubuntu 22.04 | 参考开发和部署环境，文档、脚本、CI 优先保证 |
| 一级支持 | Windows 11 + WSL2 + Docker Desktop | Windows 用户推荐路径，尽量复用 Linux 开发体验 |
| 二级支持 | 原生 Windows + PowerShell + Docker Desktop | 可运行后端、前端和依赖服务，但脚本需要单独适配 |

### 9.1 为什么推荐 WSL2 作为 Windows 主路径

| 原因 | 说明 |
| --- | --- |
| 与生产环境更接近 | Go、PostgreSQL、Redis、NATS、Docker Compose 在 Linux 下更稳定 |
| 脚本复用度更高 | 现有 Makefile、shell 命令、路径习惯更接近 Ubuntu |
| 插件构建更简单 | Wasm 和 gRPC 插件的交叉编译链路更容易统一 |
| 降低维护成本 | 不需要每个 Bash 脚本都立即写一份 PowerShell 等价实现 |

因此 v0.3-dev 建议把 Windows 使用文档写成两条路径：

| 路径 | 说明 |
| --- | --- |
| 推荐路径 | Windows 安装 WSL2，在 Ubuntu 22.04 子系统中运行后端、前端和开发命令，依赖服务使用 Docker Desktop |
| 增强路径 | 原生 Windows PowerShell 运行 Go/Vite/pnpm，数据库和 Redis/NATS 仍使用 Docker Desktop |

### 9.5 推荐推进路径：先 Ubuntu 闭环，再 Docker 化分发

不建议在 v0.3-dev 初期直接以“Ubuntu 原生开发”和“Windows 原生开发”两套完整链路并行推进。更稳妥的方式是先把 Ubuntu 22.04 作为主要参考开发环境，把后端、前端、数据库迁移、插件运行、后台管理等核心流程跑通，然后再把可运行结果封装成 Docker 镜像和 Docker Compose 编排。

推荐路线如下：

| 阶段 | 目标 | 说明 |
| --- | --- | --- |
| 第一阶段 | Ubuntu 22.04 完整闭环 | 在 Ubuntu 上完成后端、前端、数据库、插件、后台、文档和测试 |
| 第二阶段 | 运行环境 Docker 化 | 将 PostgreSQL、Redis、NATS、后端服务、前端静态站点、管理后台纳入 Docker Compose |
| 第三阶段 | Windows Docker Desktop 验证 | 在 Windows 11 + WSL2 + Docker Desktop 上验证同一套 Compose 是否可启动 |
| 第四阶段 | 更高版本 Ubuntu 验证 | 在 Ubuntu 24.04 或后续版本上验证 Docker 化部署，不强依赖宿主机细节 |
| 第五阶段 | 原生 Windows 增强 | 视需求补 PowerShell 脚本、原生路径适配和 Windows CI |

这样做的好处是，项目真正交付给 Windows 或更高版本 Ubuntu 时，依赖的不是宿主机上手工安装的一堆服务，而是一套被固定下来的容器编排。Windows 用户只需要安装 Docker Desktop，Ubuntu 用户只需要安装 Docker 与 Docker Compose，就可以尽量使用同一套启动方式。

需要注意的是，“先 Ubuntu 后 Docker”并不代表现在可以忽略跨平台问题。v0.3-dev 开发时仍然应该提前遵守这些约束：

| 约束 | 原因 |
| --- | --- |
| 后端代码使用 `filepath` 处理路径 | 避免以后迁移到 Windows 或容器卷时出问题 |
| 配置全部走环境变量或配置文件 | 避免写死本机路径、账号和端口 |
| 上传目录、插件目录、日志目录设计成 volume | 方便 Docker Compose 在不同系统上挂载 |
| 数据库迁移提供容器内执行方式 | 避免要求宿主机必须安装 `psql` 或 migration 工具 |
| 前端构建产物可由 nginx 容器托管 | 避免生产部署依赖本机 Node 环境 |
| 插件优先推进 Wasm | Wasm 比原生二进制插件更适合跨系统分发 |

因此，v0.3-dev 的推荐策略可以概括为：开发基线先收敛到 Ubuntu 22.04，部署基线逐步收敛到 Docker Compose，Windows 支持优先通过 Docker Desktop/WSL2 获得，原生 Windows 兼容作为后续增强项。

## 13. 总结

v1 阶段解决了 CampusOS 的定位和社区核心雏形；v2 阶段把权限、后台、插件、事件、缓存等平台能力逐步落地；v0.3-dev 应把重点放在插件系统的安全隔离、开发体验和可交付能力上。

从当前项目形态看，本地真正需要直接安装和运行的主要是 Go 后端工具链，以及 Node/npm/pnpm 这一类前端工具链；PostgreSQL、Redis、NATS 等基础服务大多已经通过 Docker Compose 承载。因此，Windows 系统并不是无法部署或无法使用该项目。只要 Windows 上具备 Docker Desktop、Go、Node/npm/pnpm，并正确处理端口、环境变量和数据库迁移，项目具备在 Windows 环境运行的基础条件。

Windows 的主要难点不在数据库、缓存、消息队列这些服务本身，而在工程脚本和运行环境差异：

| 难点 | 具体表现 | v0.3-dev 建议 |
| --- | --- | --- |
| Shell/Makefile 依赖 | 原生 Windows 默认不支持 Bash 脚本，Makefile 也需要额外工具 | 提供 PowerShell 脚本，或把关键命令迁入 Go CLI |
| 路径差异 | Linux 使用 `/`，Windows 使用盘符和 `\`，Docker volume 映射也有差异 | 代码中使用 `filepath`，文档中分别给出 Windows/WSL2 路径示例 |
| 环境变量写法 | Linux shell 与 PowerShell 的变量设置方式不同 | `.env.example` 标准化，并提供 PowerShell 示例 |
| Docker Desktop 差异 | Windows 下 Docker 依赖 WSL2/Hyper-V，文件挂载和网络访问方式不同 | 推荐 Windows + WSL2 + Docker Desktop 作为主路径 |
| 数据库迁移 | 当前迁移命令如果依赖 shell/psql 本地命令，Windows 用户容易卡住 | 提供跨平台迁移命令，优先考虑 Go CLI 统一执行 |
| 前端依赖 | npm/pnpm 在 Windows 可用，但 node_modules、换行符、路径长度可能出问题 | 固定 Node/pnpm 版本，补充 `.gitattributes` 和安装说明 |
| 插件二进制 | gRPC 插件如果是本地二进制，需要按 GOOS/GOARCH 分别构建 | Wasm 插件作为优先方向，gRPC 插件提供跨平台构建说明 |
| 进程管理 | Linux 常用 systemd，Windows 原生服务管理方式不同 | v0.3-dev 暂不把原生 Windows 生产部署列为 P0 |

因此，Ubuntu 22.04 与 Windows 兼容是可行的，但建议采用务实分层策略：Ubuntu 22.04 作为参考环境，Windows 以 WSL2 + Docker Desktop 作为主支持路径，原生 Windows 通过 PowerShell 脚本和跨平台 Go CLI 逐步增强。这样既能照顾 Windows 用户，又不会让项目过早陷入多平台脚本和部署细节的维护成本。

换句话说，v0.3-dev 暂时不建议一开始就按“双系统原生兼容”投入开发资源。更推荐先在 Ubuntu 22.04 上把项目完整搭建完成，再通过 Docker 镜像和 Docker Compose 把运行环境固化下来，然后迁移验证到 Windows Docker Desktop、WSL2 或更高版本 Ubuntu。这样既能降低当前开发复杂度，也更符合项目中大部分基础服务已经 Docker 化的现实。
