# CampusOS — 项目介绍（v3 视角）

> **一句话定位**：CampusOS 是一个以**论坛帖子系统**为主体应用的**社区操作系统**，核心只提供身份、权限、事件、插件等原子能力，所有业务功能通过插件扩展实现。

---

## 一、项目本质：我们在做什么？

### 1.1 一个类比

CampusOS 的设计哲学可以用**操作系统**来理解：

```
┌─────────────────────────────────────────────────────┐
│                    应用层（插件）                      │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────┐ │
│   │ 论坛帖子  │ │ AI 助手  │ │ QQ 机器人 │ │ 校园应用│ │
│   │ (主体App) │ │ (插件)   │ │ (插件)   │ │ (插件) │ │
│   └────┬─────┘ └────┬─────┘ └────┬─────┘ └───┬───┘ │
│        │             │            │            │     │
│   ─────┴─────────────┴────────────┴────────────┴──  │
│                    系统调用层（Host API）              │
│   ────────────────────────────────────────────────  │
│                    内核层（Core + Platform）           │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌───────┐ │
│   │ 身份认证  │ │ 权限管理  │ │ 事件总线  │ │ 插件管理│ │
│   └──────────┘ └──────────┘ └──────────┘ └───────┘ │
│   ┌──────────┐ ┌──────────┐ ┌──────────┐           │

| **开发者工具** | SDK + CLI + 开发者中心 | 📋 v0.3 规划 |

### 1.2 核心理解

**CampusOS 不是在做一个论坛，而是在做一个社区操作系统。论坛帖子系统是这个操作系统的"第一个应用"——一个预装的基础 App。**

- **主体**：一个完整的帖子发布前后端系统（用户发帖、浏览、回复、版块分类、审核管理）
- **内核**：身份认证、权限管理、事件总线、插件系统等原子能力
- **扩展**：通过插件机制，任何新功能（AI 审核、QQ 同步、数据分析、校园通知）都以插件形式接入，不修改核心代码

## 二、当前实现的完整能力

### 2.1 内核层 — 原子能力

这是操作系统的核心，为所有应用（包括论坛本身和未来插件）提供基础服务。

#### 身份认证系统
- 用户注册、登录、Token 刷新
- JWT 认证（Access Token + Refresh Token）
- bcrypt 密码哈希
- API Key 认证（供插件/第三方调用）
- 多端登录支持（User 与 Account 解耦设计）

#### 权限管理系统（RBAC）
- 4 个内置角色：admin（管理员）/ moderator（版主）/ member（会员）/ guest（访客）
- 24 条权限规则的权限矩阵
- 权限中间件：JWT Claims → 角色查询 → 权限矩阵匹配
- 角色管理 API：分配/撤销/查询

#### 事件驱动系统
- 基于 NATS JetStream，支持内存回退
- CloudEvents 1.0 规范
- `.before`（同步拦截）与 `.after`（异步通知）双模式
- At-least-once 投递 + 死信队列 + 指数退避重试
- 事件历史查询 API

#### 插件系统
- gRPC 运行时（已实现）+ Wasm 运行时（v0.3 规划）
- Plugin Manager 统一调度 + 依赖注入
- Manifest 规范（YAML 声明运行环境、事件订阅、权限清单）
- Host API 8 个接口（插件调用核心能力）：
  - `GetUser` / `GetThread` — 数据查询
  - `PublishEvent` — 事件发布
  - `SendNotification` — 通知发送
  - `GetConfig` / `SetConfig` — 配置读写
  - `CheckPermission` — 权限检查
  - `Log` — 日志记录

#### 基础设施
- PostgreSQL 16+（16 张表，pgx 原生驱动）
- Redis 7+（缓存，支持 Memory/Noop 回退）
- NATS 2.x（消息队列，支持内存回退）
- 统一响应体 `{code, msg, data}`
- 完整中间件链：CORS → TraceID → Logger → Recovery → JWT → Permission → APIKey

## 三、架构设计：为什么这样设计？

### 3.1 核心原则：业务与内核分离

```
                    ┌─────────────────────┐
                    │    插件（未来扩展）    │
                    │  AI / QQ / 校园应用   │
                    └──────────┬──────────┘
                               │ Host API
                    ───────────┴───────────
                    ┌─────────────────────┐
                    │    论坛帖子系统       │  ← 第一个应用
                    │  (community 域)      │     遵循同样的分层
                    └──────────┬──────────┘
                               │ 使用内核能力
                    ───────────┴───────────
                    ┌─────────────────────┐
                    │    内核（原子能力）    │
                    │  身份/权限/事件/插件  │
                    └─────────────────────┘
```

**关键设计决策**：

1. **论坛帖子系统本身也遵循"应用"的分层结构**（domain → repository → service → handler），与未来插件的开发模式一致
2. **内核能力对所有应用平等开放**：论坛通过 service 层直接调用内核，插件通过 Host API 调用内核，接口一致
3. **事件驱动解耦**：论坛的帖子创建、回复等操作都会发布事件，插件可以监听这些事件实现扩展功能

### 3.2 分层架构（实际实现）

```
┌─────────────────────────────────────────────────────┐
│  前端层                                              │
│  用户前台 (web/ :3000)  +  管理后台 (admin/ :3001)    │
├─────────────────────────────────────────────────────┤
│  网关层 — Gin HTTP Server (:8080)                    │
│  中间件链: CORS → TraceID → Logger → Recovery        │
│           → JWT Auth → Permission → APIKey           │
├─────────────────────────────────────────────────────┤
│  处理层 — Handler                                    │
│  身份域(identity) + 社区域(community) + 插件域(plugin) │
├─────────────────────────────────────────────────────┤
│  服务层 — Service                                    │
│  业务逻辑 + 权限检查 + 事件发布                        │
├─────────────────────────────────────────────────────┤
│  仓储层 — Repository                                 │
│  PostgreSQL (pgx) + Memory (测试)                    │
├─────────────────────────────────────────────────────┤
│  基础设施层                                           │
│  PostgreSQL 16 + Redis 7 + NATS 2.x                 │
└─────────────────────────────────────────────────────┘
```

### 3.3 代码组织（DDD 领域驱动）

```

## 四、插件扩展机制：如何实现多样化功能？

### 4.1 插件的三种运行方式

| 方式 | 隔离级别 | 适用场景 | 状态 |
|:---|:---|:---|:---|
| **gRPC Runtime** | 进程级隔离 | IM 适配器、长连接、RAG 服务 | ✅ 已实现 |
| **Wasm Runtime** | 内存级沙箱 | 事件监听、内容审核、Markdown 扩展 | 📋 v0.3 规划 |
| **内置 Handler** | 无隔离 | 高性能核心扩展 | ✅ 已实现 |

### 4.2 插件与核心的交互方式

```
插件 ──(订阅事件)──→ Event Bus ←──(发布事件)── 论坛帖子系统
  │                                              │
  ├──(调用 Host API)──→ 内核能力 ←──(直接使用)────┤
  │                    (身份/权限/                  │
  │                     配置/缓存)                  │
  └──(返回结果)────────→ Handler ──→ Response ────┘
```

**事件驱动示例**：
```
用户发帖 → thread.created 事件发布到 Event Bus
  ├── 论坛系统：更新版块帖子计数
  ├── 内容审核插件（Wasm）：检查敏感词
  ├── QQ 机器人插件（gRPC）：同步到 QQ 群
  ├── 统计分析插件（Wasm）：更新访问统计
  └── AI 助手插件（gRPC）：生成帖子摘要
```

### 4.3 插件权限控制

三层权限模型（v0.3 完善）：
1. **运行时隔离**：Wasm 沙箱 / gRPC 进程隔离，插件崩溃不影响核心
2. **功能权限**：Manifest 声明可调用的 Host API 白名单
3. **数据权限**：控制插件可访问的数据行和字段

### 4.4 未来插件生态

```
v0.3  ──→  Wasm 运行时 + Go SDK + CLI 工具
v0.4  ──→  AI 网关 + MCP Server + QQ 机器人适配器
v1.0  ──→  插件市场 + 开发者中心 + 多语言 SDK
```

## 九、总结

**CampusOS 的核心思路**：

1. **内核提供原子能力**：身份、权限、事件、插件——这些是操作系统的基础
2. **论坛是第一个应用**：帖子/版块/回复/审核——这是当前的主体业务
3. **插件实现多样化**：AI、QQ 机器人、校园应用——这些通过插件扩展，不动核心
4. **事件驱动连接一切**：论坛发帖触发事件，插件监听事件做出响应，各组件松耦合

**当前状态**：内核已稳固，主体应用（论坛帖子系统）已完整可用，插件基础设施已就位（gRPC Runtime + Host API + 权限系统），等待 v0.3 引入 Wasm 运行时后，插件生态将正式启航。
